01

Foundation — What is a Decision Tree?

A decision tree is a flowchart of yes/no questions. Each internal node asks one question about one feature. Each branch is the answer. Each leaf node gives the final prediction.

Root: Credit Score > 700? ├── No (40 samples) ──▶ Income > 40k? │ ├── No ──▶ ❌ Deny (Leaf A) │ └── Yes ──▶ ✅ Approve (Leaf B) │ └── Yes (60 samples) ──▶ Age > 30? ├── No ──▶ ✅ Approve (Leaf C) └── Yes ──▶ ❌ Deny (Leaf D)

Internal Node

Asks one question on one feature. Stores: feature name, threshold, pointers to children. Never stores a prediction.

Leaf Node

Terminal node — no children. Stores: majority class (classification) or mean of targets (regression). This is the prediction.

A new sample travels from root to leaf by answering each question. It always reaches a leaf — never stops at an internal node.

The Fundamental Problem

At each node, you have a mixed group of samples. You want to ask a question that separates them as cleanly as possible. "Clean" means each child group is as pure as possible — ideally all one class.

Two formulas measure purity: Gini Impurity and Entropy.

02

Split Criteria — Gini, Entropy, Information Gain

Gini Impurity

Asks: "If I randomly picked two samples from this node, what is the probability they belong to different classes?"

Formula Gini = 1 − Σ(pᵢ²)
where pᵢ = proportion of class i in the node
Worked Example — Gini Impurity

A node has 4 cats and 6 dogs (10 samples total).

1

Calculate class proportions:

p(cat) = 4/10 = 0.4
p(dog) = 6/10 = 0.6
2

Apply Gini formula:

Gini = 1 − (0.4² + 0.6²)
     = 1 − (0.16 + 0.36)
     = 1 − 0.52
     = 0.48
3

Pure node (all 10 dogs):

Gini = 1 − (1.0²) = 0.0   ← perfectly pure
Gini = 0 means perfectly pure. Gini = 0.5 means maximally mixed (50/50). Lower is better.

Entropy

From information theory. Asks: "How much uncertainty or disorder is in this node?"

Formula Entropy = −Σ(pᵢ × log₂(pᵢ))
Worked Example — Entropy

Same node: 4 cats, 6 dogs.

Entropy = −(0.4 × log₂(0.4)) − (0.6 × log₂(0.6))
        = −(0.4 × −1.322) − (0.6 × −0.737)
        = 0.529 + 0.442
        = 0.971

Pure node (all dogs):
Entropy = −(1.0 × log₂(1.0)) = −(1.0 × 0) = 0.0

Information Gain — How Splits are Chosen

Information Gain measures how much a split reduced impurity. The tree picks the split with the highest gain.

Formula Gain = Impurity(parent) − weighted avg Impurity(children)

Weighted Impurity = (n_left / n_total) × Impurity(left) + (n_right / n_total) × Impurity(right)
The weight is simply the proportion of samples in each child. A large impure child hurts more than a small impure child — which is exactly what you want.
Worked Example — Information Gain

Parent: 5A + 5B = 10 samples. Gini = 0.5

After split: Left = 4A+1B (5 samples). Right = 1A+4B (5 samples).

1

Gini of each child:

Gini(left)  = 1 − ((4/5)² + (1/5)²)
            = 1 − (0.64 + 0.04) = 0.32

Gini(right) = 1 − ((1/5)² + (4/5)²)
            = 1 − (0.04 + 0.64) = 0.32
2

Weighted Gini of children:

Weighted = (5/10) × 0.32 + (5/10) × 0.32
         = 0.16 + 0.16
         = 0.32
3

Information Gain:

Gain = 0.50 − 0.32 = 0.18   ← 36% reduction in impurity

Gini vs Entropy — Which to Use?

GiniEntropy
ComputationFaster (no log)Slightly slower
BehaviourIsolates largest classMore balanced splits
sklearn defaultYes — criterion="gini"criterion="entropy"
Practical differenceUsually negligible — treat as a hyperparameter to tune
Gini and Entropy are different rulers for measuring impurity. Information Gain is the decision made using whichever ruler you chose. They are not alternatives — Gini/Entropy measure impurity, Gain decides which split to make.

How the Algorithm Searches for the Best Split

At every node, the algorithm runs an exhaustive search:

# Pseudocode
best_gain = 0
best_split = None

for feature in all_features:
    for threshold in unique_values(feature):
        left, right = split(data, feature, threshold)
        gain = impurity(parent) − weighted_impurity(left, right)
        if gain > best_gain:
            best_gain = gain
            best_split = (feature, threshold)

# Pick best_split for this node
Each split uses exactly ONE feature and ONE threshold. The tree never combines features in a single split (e.g. "Age > 30 AND Income > 50k" is two splits, not one). This keeps trees fast and interpretable but limits them to axis-aligned decision boundaries.

Can a feature appear multiple times? Yes — the same feature can appear on multiple nodes in a path, as long as thresholds are different. Example: Credit Score > 700 at depth 1 and Credit Score > 750 at depth 3 is valid — each split further refines the range. The gain would be zero for a redundant threshold, so the tree naturally avoids it.

03

How a Tree is Built — The Greedy Algorithm

The tree is built by one recursive function:

function build(node, data):
    if stopping_condition_met(data):
        make this a leaf → store majority class / mean
        return

    find best_split = (feature, threshold) with highest gain
    left_data, right_data = split(data, best_split)

    node.left  = build(left_data)
    node.right = build(right_data)

Stopping Conditions

The tree stops splitting at a node when any one of these is triggered:

ConditionWhat it checkssklearn param
Max depth reachedCurrent depth == max_depthmax_depth
Too few samplesNode has < N samplesmin_samples_split
Tiny leafSplit would create child with < N samplesmin_samples_leaf
Gain too smallBest gain < thresholdmin_impurity_decrease
Already pureGini = 0 or Entropy = 0Automatic

Why "Greedy"?

At each node, the algorithm picks the best split right now with no lookahead. It has no idea whether a slightly worse split today might enable a much better split two levels down.

Greedy picks this

Split A
Gain = 0.30 (higher ✓)

Children: Gini 0.4 / 0.4
→ still needs more splits
→ needs depth 4 to finish

Better global choice

Split B
Gain = 0.20 (lower ✗)

Children: Gini 0.0 / 0.2
→ one child already pure
→ finishes at depth 2
This greedy limitation is exactly why Random Forest and Boosting exist — they compensate by building many trees and combining them.

min_samples_split vs min_samples_leaf

# min_samples_split — checks the PARENT
# "Don't attempt a split if this node has fewer than N samples"
min_samples_split = 20
# → only gates whether to try splitting

# min_samples_leaf — checks BOTH CHILDREN after every candidate split
# "Reject this split if it would create a child with fewer than N samples"
min_samples_leaf = 5
# → guarantees EVERY node in the entire tree has at least 5 samples
# → more direct, more useful in practice
04

Overfitting — High Variance, Low Bias

A fully grown tree (no stopping conditions) splits until every leaf has exactly one sample or is pure. Training accuracy = 100%. But it has memorised the training data, not learned the pattern.

Tree depthTrain accTest accProblem
depth = 165%63%Underfit — misses pattern
depth = 488%85%Good generalisation
depth = 20100%61%Overfit — memorised noise

Bias vs Variance — Precise Definitions

High Bias

Too simple. Model is systematically wrong — misses the real pattern regardless of which training data you use. Underfitting.

Example: depth=1 stump trying to solve a complex problem.

High Variance

Too sensitive. Model's predictions change dramatically when trained on slightly different data. Overfitting.

Example: depth=20 tree — small data changes produce completely different trees.

The Tradeoff for Decision Trees Shallow tree → High Bias, Low Variance (underfits) Deep tree → Low Bias, High Variance (overfits) Tuned tree → Balanced Bias, Balanced Variance ✓
Random Forest fixes high variance by averaging many deep trees — errors cancel out. Boosting (GBM, XGBoost) fixes high bias by sequentially adding shallow trees — each one reduces bias a little.
05

Pruning — Pre and Post

Pre-Pruning

Stop the tree from growing too large during construction using stopping conditions.

Fast but blind — stops splits without knowing if they'd be useful deeper down.

Params: max_depth, min_samples_leaf, min_impurity_decrease

Post-Pruning

Grow the full tree first, then remove branches that don't add real value.

More principled — you can see the whole tree before deciding what to cut.

Params: ccp_alpha

Cost-Complexity Pruning — Full Worked Example

Post-pruning uses a penalty parameter alpha (α) — also called ccp_alpha in sklearn.

Cost Formula Cost(subtree) = weighted impurity of its leaves + α × number of leaves
Cost(as leaf) = impurity of the node itself + α × 1
Prune when: Cost(as leaf) ≤ Cost(subtree)

Let's use our loan approval tree. Full tree has 4 leaves:

Root (100 samples, Gini=0.50) ├── Income > 40k? (40 samples, Gini=0.42) │ ├── No → Leaf A: 20 Deny, 5 Approve — Gini=0.32 (25 samples) │ └── Yes → Leaf B: 14 Approve, 1 Deny — Gini=0.12 (15 samples) └── Age > 30? (60 samples, Gini=0.44) ├── No → Leaf C: 18 Approve, 2 Deny — Gini=0.16 (20 samples) └── Yes → Leaf D: 28 Deny, 12 Approve — Gini=0.41 (40 samples)
Should we prune the LEFT subtree (Income > 40k)?
1

Cost of keeping the subtree (2 leaves):

Weighted impurity = (25/40)×0.32 + (15/40)×0.12
                 = 0.200 + 0.045 = 0.245

Cost(subtree) = 0.245 + α × 2
2

Cost if we collapse into one leaf (40 samples: 34 Deny, 6 Approve):

Gini(collapsed) = 1 − ((34/40)² + (6/40)²)
               = 1 − (0.722 + 0.023) = 0.256

Cost(collapsed) = 0.256 + α × 1
3

Find the alpha threshold for pruning:

Prune when: Cost(collapsed) ≤ Cost(subtree)
0.256 + α  ≤  0.245 + 2α
0.256 − 0.245  ≤  2α − α
0.011  ≤  α

→ If α ≥ 0.011, prune the left subtree
4

Effective alpha (the exact pruning threshold):

Effective α = (impurity increase) / (leaves removed)
           = (0.256 − 0.245) / (2 − 1)
           = 0.011 / 1
           = 0.011
Should we prune the RIGHT subtree (Age > 30)?
Weighted impurity = (20/60)×0.16 + (40/60)×0.41
                 = 0.053 + 0.273 = 0.326

Collapsed leaf (46 Deny, 14 Approve, 60 samples):
Gini = 1 − ((46/60)² + (14/60)²) = 1 − (0.588 + 0.054) = 0.358

Effective α = (0.358 − 0.326) / (2 − 1) = 0.032
→ If α ≥ 0.032, prune the right subtree too

Full Pruning Story

α = 0.000 → no pruning — full tree (4 leaves) α = 0.011 → left subtree pruned (3 leaves) α = 0.032 → right subtree pruned (2 leaves) α = large → everything pruned — single leaf (predict majority class always)
Very large ccp_alpha collapses the tree to a single leaf that always predicts the majority class. Completely underfit — useless model.
XGBoost's γ (gamma) parameter does the exact same thing — it is the effective alpha threshold. Any leaf whose gain < γ gets pruned. This is why XGBoost's pruning is mathematically principled rather than just a crude depth limit.

Finding the Right Alpha in Practice

from sklearn.tree import DecisionTreeClassifier
from sklearn.model_selection import cross_val_score

clf = DecisionTreeClassifier()
path = clf.cost_complexity_pruning_path(X_train, y_train)
alphas = path.ccp_alphas  # All candidate alpha values

best_alpha, best_score = 0, 0
for alpha in alphas:
    clf = DecisionTreeClassifier(ccp_alpha=alpha)
    score = cross_val_score(clf, X_train, y_train, cv=5).mean()
    if score > best_score:
        best_score, best_alpha = score, alpha

final_tree = DecisionTreeClassifier(ccp_alpha=best_alpha)

Regression Trees

Classification trees predict a class. Regression trees predict a number. One change — three consequences.

Classification TreeRegression Tree
Leaf storesMajority classMean of target values
Split criterionGini / EntropyMSE (or MAE)
Error metricAccuracy, F1RMSE, MAE, R²

MSE Split Criterion

Mean Squared Error MSE = (1/n) × Σ(yᵢ − ȳ)² where ȳ = mean of target values in the node Gain = MSE(parent) − weighted MSE(children)
Worked Example — House Price Prediction

A node has 6 houses:

Size (sqft):  800   1000  1200  1500  1800  2000
Price (lakh):  40     50    55    80    85    90
1

Current node MSE:

Mean = (40+50+55+80+85+90)/6 = 66.67

MSE = ((40−66.67)² + (50−66.67)² + (55−66.67)²
     + (80−66.67)² + (85−66.67)² + (90−66.67)²) / 6
    = (711 + 278 + 136 + 178 + 336 + 544) / 6
    = 363.8
2

Try split: Size > 1200

Left  (≤1200): [40, 50, 55]  →  mean = 48.3
Right (>1200): [80, 85, 90]  →  mean = 85.0

MSE(left)  = ((40−48.3)² + (50−48.3)² + (55−48.3)²) / 3
           = (68.9 + 2.9 + 44.9) / 3 = 38.9

MSE(right) = ((80−85)² + (85−85)² + (90−85)²) / 3
           = (25 + 0 + 25) / 3 = 16.7
3

Weighted MSE and Gain:

Weighted MSE = (3/6)×38.9 + (3/6)×16.7
            = 19.45 + 8.35 = 27.8

Gain = 363.8 − 27.8 = 336.0
The tree tries every feature and threshold, picks Size > 1200 if this is the highest gain. Left leaf predicts 48.3, right leaf predicts 85.0.

Why XGBoost Always Uses Regression Trees

XGBoost uses regression trees internally even for classification problems. Why?

# In Gradient Boosting, each tree fits RESIDUALS
Residuals = actual − predicted_probability
         = [+0.40, −0.60, +0.40, ...]   ← continuous numbers!

# You can't fit continuous numbers with a classification tree
# So every internal tree in GBM/XGBoost is a REGRESSION tree
# Even when predicting Approve/Deny
XGBoost uses regression trees internally even for classification — because each tree fits residuals (continuous values), not class labels. This is a strong interview answer that most candidates miss.

Regression Trees Approximate Curves as Staircases

A regression tree always produces a staircase — constant predictions within each leaf region. More depth = more steps = closer to the true curve. Too many steps = overfitting noise.

True curve: smooth continuous function y = f(x)

Depth 1 (2 leaves):   ─────┐
                            └─────────       rough staircase

Depth 3 (8 leaves):   ──┐
                         └──┐
                             └──┐
                                 └──────     closer approximation

Depth ∞:              perfect fit of training data → overfits
07

Key Hyperparameters

Group 1 — Tree Size (most important)

max_depth
Maximum number of levels in the tree. Single biggest lever against overfitting. Try 3, 5, 7 with cross-validation. Every extra level doubles potential leaves.
default: None
min_samples_split
Minimum samples in a node before attempting a split. Checks the parent. Gates whether to try splitting at all.
default: 2
min_samples_leaf
Minimum samples that must land in each child after a split. More powerful than min_samples_split — guarantees every node in the entire tree has at least N samples. Tune this before min_samples_split.
default: 1

Group 2 — Split Quality

criterion
Which impurity measure to use. Classification: "gini" (default) or "entropy". Regression: "squared_error" (default) or "absolute_error" (robust to outliers). Practical difference is usually small.
default: "gini"
min_impurity_decrease
Only split if gain ≥ this value. Filters out technically positive but meaningless splits. Directly controls the noise threshold for splitting.
default: 0.0
max_leaf_nodes
Hard cap on total number of leaves. sklearn grows the best leaves first (highest gain) until limit hit. Alternative way to control complexity instead of max_depth.
default: None

Group 3 — Post-Pruning

ccp_alpha
Complexity penalty α for cost-complexity pruning. 0 = no pruning (full tree). Large = aggressive pruning (single leaf). Find optimal value using cross-validation over pruning path.
default: 0.0

Tuning Order for Interviews

1st  max_depth          → biggest single lever, start here, try [3, 5, 7, 10]
2nd  min_samples_leaf   → prevents fragile tiny leaves, try [5, 10, 20]
3rd  criterion          → gini vs entropy, quick test, rarely matters
4th  ccp_alpha          → best results, needs full cross-validation sweep

Feature Importance

Training a decision tree automatically tells you which features mattered most. sklearn calls this Mean Decrease in Impurity (MDI) — also known as Gini importance.

How it's Calculated

For each feature, sum up the weighted impurity decrease across every node where that feature was used as the split:

Formula Importance(feature f) = Σ (nₜ/N) × (impurity(t) − (n_left/nₜ)×impurity(left) − (n_right/nₜ)×impurity(right)) for every node t where feature f was used Then normalise so all importances sum to 1.0

Breaking it down: (nₜ/N) is the proportion of samples reaching node t. The rest is the impurity reduction from the split. More samples × bigger impurity reduction = higher importance.

Worked Example — Feature Importance Calculation

Our loan approval tree (100 total samples):

1

Root node — Credit Score > 700 (100 samples, Gini=0.50):

Left child:  40 samples, Gini = 0.42
Right child: 60 samples, Gini = 0.44

Weighted child impurity = (40/100)×0.42 + (60/100)×0.44
                       = 0.168 + 0.264 = 0.432

Impurity reduction = 0.50 − 0.432 = 0.068
Weighted by samples = (100/100) × 0.068 = 0.068

Credit Score gets += 0.068
2

Left node — Income > 40k (40 samples, Gini=0.42):

Left child:  25 samples, Gini = 0.32
Right child: 15 samples, Gini = 0.12

Weighted child impurity = (25/40)×0.32 + (15/40)×0.12
                       = 0.200 + 0.045 = 0.245

Impurity reduction = 0.42 − 0.245 = 0.175
Weighted by samples = (40/100) × 0.175 = 0.070

Income gets += 0.070
3

Right node — Age > 30 (60 samples, Gini=0.44):

Left child:  20 samples, Gini = 0.16
Right child: 40 samples, Gini = 0.41

Weighted child impurity = (20/60)×0.16 + (40/60)×0.41
                       = 0.053 + 0.273 = 0.326

Impurity reduction = 0.44 − 0.326 = 0.114
Weighted by samples = (60/100) × 0.114 = 0.068

Age gets += 0.068
4

Raw importances before normalisation:

Credit Score:  0.068
Income:        0.070

               ─────
Total:         0.206
5

Normalise to sum to 1.0:

Credit Score:  0.068 / 0.206 = 0.330  (33.0%)
Income:        0.070 / 0.206 = 0.340  (34.0%)

How to Get Feature Importances in sklearn

from sklearn.tree import DecisionTreeClassifier
import pandas as pd

clf = DecisionTreeClassifier()
clf.fit(X_train, y_train)

# Raw importances
importances = clf.feature_importances_  # array, sums to 1.0

# As a sorted DataFrame
fi = pd.DataFrame({
    'feature': X_train.columns,
    'importance': importances
}).sort_values('importance', ascending=False)

print(fi)

Important Caveats

MDI feature importance has a known bias toward high-cardinality features — features with many unique values (like IDs or zip codes) get artificially high importance because they have more thresholds to try. Use permutation importance for a more reliable estimate.
A single tree's feature importances are noisy and unstable — a slightly different training set can give very different importances. Always prefer Random Forest's feature importances (averaged over hundreds of trees) for reliable feature selection.

Permutation Importance — More Reliable Alternative

Randomly shuffle one feature's values and measure how much the model's accuracy drops. Big drop = important feature.

from sklearn.inspection import permutation_importance

result = permutation_importance(clf, X_test, y_test, n_repeats=30)
# Uses TEST data → measures real predictive value, not just training split gain
Feature importance is a free byproduct of training a decision tree. But for real feature selection, use Random Forest importances (averaged) or permutation importance (unbiased). Single tree importances are unreliable due to high variance.

Complete Summary

Advantages vs Disadvantages

Advantages

  • Interpretable — every prediction is a path of rules
  • Handles non-linear relationships
  • Works for classification and regression
  • No normalisation needed — splits are threshold-based
  • Handles mixed feature types (numerical + categorical)
  • Feature importance for free
  • Handles missing values better than most models

Disadvantages

  • Overfits easily without pruning
  • High variance — small data changes = different tree
  • Feature dominance — one strong feature overshadows others
  • Axis-aligned splits only — diagonal boundaries need many splits
  • Poor at simple linear relationships
  • Unstable feature importances from a single tree
  • Almost always beaten by ensembles in practice

Key Formulas at a Glance

Gini Impurity Gini = 1 − Σ(pᵢ²) range: [0, 0.5] Entropy Entropy = −Σ(pᵢ × log₂(pᵢ)) range: [0, 1] Information Gain Gain = Impurity(parent) − weighted Impurity(children) Weighted = (n_left/n) × Impurity(left) + (n_right/n) × Impurity(right) MSE (Regression) MSE = (1/n) × Σ(yᵢ − ȳ)² Feature Importance (MDI) Importance(f) = Σ (nₜ/N) × ImpurityReduction(t) [normalised to sum 1] Cost-Complexity Pruning Cost(subtree) = weighted leaf impurity + α × num_leaves Effective α = impurity increase / leaves removed

The One-Line Interview Answers

QuestionOne-line answer
How does a decision tree split?Exhaustive search over all features and thresholds, picks the split with highest Information Gain.
Gini vs Entropy?Different impurity rulers. Gini is faster (no log). Practical difference is negligible.
Why does a deep tree overfit?It memorises training data — each leaf has too few samples to represent a real pattern.
Pre vs post pruning?Pre-pruning stops early using hyperparameters. Post-pruning grows full tree then removes low-value branches using ccp_alpha.

Decision trees are rarely used alone in practice — their real value is as the building block for Random Forest and XGBoost, which fix most disadvantages while keeping the core strengths of interpretability and non-linearity.

Interview Questions — 30 Questions Across 3 Levels

Each question includes the must-have points an interviewer expects. Expand each answer only after you've attempted it yourself.

🟢 Easy — Expect These in Every Interview

Q1 What is a decision tree and how does it make predictions? ▼
Must-have points
  • Flowchart of yes/no questions — each internal node splits on one feature and one threshold
  • New sample is routed from root to leaf by answering each question
  • Leaf stores majority class (classification) or mean of targets (regression)
  • Internal nodes never store predictions — only routing logic
  • Always reaches exactly one leaf per prediction
Strong answer

"A decision tree is a flowchart of yes/no questions. Each internal node asks one question about one feature — e.g. 'Credit Score > 700?' — and routes the sample left or right. This continues until the sample reaches a leaf node, which stores the prediction: majority class for classification or mean of target values for regression. Internal nodes contain only the split condition — no prediction is made until a leaf is reached."

Q2 What is Gini impurity? How is it calculated? ▼
Must-have points
  • Formula: Gini = 1 − Σ(pᵢ²)
  • Measures probability that two randomly picked samples belong to different classes
  • Range: 0 (perfectly pure) to 0.5 (maximally mixed, 50/50)
  • Lower is better — purer node
  • Numerical example: 4 cats, 6 dogs → Gini = 1 − (0.4² + 0.6²) = 0.48
Strong answer
Node with 4 cats, 6 dogs (10 samples):
p(cat) = 0.4,  p(dog) = 0.6

Gini = 1 − (0.4² + 0.6²)
     = 1 − (0.16 + 0.36) = 0.48

Pure node (all dogs):
Gini = 1 − (1.0²) = 0.0   ← perfectly pure
Q3 What is Information Gain and how does the tree use it? ▼
Must-have points
  • Formula: Gain = Impurity(parent) − weighted Impurity(children)
  • Weighted by proportion of samples: (n_left/n) × Impurity(left) + (n_right/n) × Impurity(right)
  • Tree tries every feature and every threshold, picks the split with highest gain
  • Gini and Entropy are just different rulers for measuring impurity — both used to compute gain
  • Weights ensure a large impure child hurts more than a small impure child
Strong answer

"Information Gain measures how much a split reduced impurity. The tree computes it for every (feature, threshold) combination and picks the highest. The children's impurity is weighted by sample proportion — a large impure child hurts the gain more than a small impure child, which is the right behavior. Gini and Entropy are just different formulas for measuring the impurity — Information Gain is the decision on top of whichever you choose."

Q4 What is the difference between Gini impurity and Entropy? ▼
Must-have points
  • Both measure node impurity — they are different rulers, not alternatives to Information Gain
  • Gini is faster — no logarithm calculation
  • Entropy tends to give more balanced splits; Gini tends to isolate the largest class
  • Practical difference is usually negligible — treat as a hyperparameter
  • sklearn default is Gini; set criterion="entropy" for entropy
Strong answer

"Both measure how mixed a node is. Gini is computationally cheaper since it avoids a log calculation. Entropy comes from information theory and can produce slightly more balanced splits. In practice the difference is negligible — I'd treat criterion as a hyperparameter and do a quick cross-validation comparison. The default Gini is a fine starting point."

Q5 What is overfitting in decision trees and how do you fix it? ▼
Must-have points
  • A fully grown tree (max_depth=None) memorises training data — 100% train accuracy, poor test accuracy
  • Deep trees have low bias, high variance
  • Fix with pre-pruning: max_depth, min_samples_leaf, min_impurity_decrease
  • Fix with post-pruning: ccp_alpha (cost-complexity pruning)
  • Tuning order: max_depth first, then min_samples_leaf, then ccp_alpha
Strong answer

"A deep tree splits until every leaf is pure or has one sample — it memorises noise in the training data. I'd fix it in order: first reduce max_depth (biggest single lever), then increase min_samples_leaf to prevent fragile tiny leaves, then use ccp_alpha with cross-validation for cost-complexity pruning if I need the best possible generalization."

Q6 What is the difference between min_samples_split and min_samples_leaf? ▼
Must-have points
  • min_samples_split — checks the parent: "don't attempt a split if fewer than N samples in this node"
  • min_samples_leaf — checks both children: "reject this split if it would create a child with fewer than N samples"
  • min_samples_leaf is more powerful — guarantees every single node in the entire tree has at least N samples
  • min_samples_leaf is checked at every node at every depth, not just the final leaves
  • In practice: tune min_samples_leaf first — it's more direct and reliable
Strong answer

"min_samples_split gates whether to attempt a split on the parent node. min_samples_leaf rejects any split that would create a child node smaller than N — it's checked at every level, not just final leaves. min_samples_leaf is more useful in practice because it gives a global guarantee: no node anywhere in the tree will have fewer than N samples, making every prediction more statistically reliable."

Q7 What does max_depth=None mean? Why is it usually a bad idea? ▼
Must-have points
  • max_depth=None means no limit — tree grows until all leaves are pure or have fewer than min_samples_split samples
  • Can result in leaves with a single sample — memorising noise
  • Training accuracy = 100% but test accuracy collapses
  • High variance — the tree has fit to specific quirks of the training data
  • Good starting range: max_depth = 3 to 7 for most problems
Strong answer

"max_depth=None lets the tree grow until every leaf is pure or too small to split further. This almost always produces a tree that memorises the training data — 100% train accuracy and poor test accuracy. Each level doubles the potential number of leaves, so depth 20 can have over a million leaves on a small dataset. I'd always set a max_depth as a starting point and tune from there."

Q8 What is a decision stump and when would you use one? ▼
Must-have points
  • Decision stump = decision tree with max_depth=1 — one split, two leaves
  • Extremely weak learner — barely better than random guessing
  • Very high bias, very low variance
  • Used deliberately in AdaBoost — hundreds of stumps, each focusing on samples previous stumps got wrong
  • Each stump captures one small signal — hundreds together form a strong model
  • This is the formal definition of a "weak learner" in boosting theory
Strong answer

"A decision stump has max_depth=1 — one question, two answers. Alone it's nearly useless — high bias, can only draw one axis-aligned boundary. But it's the canonical weak learner in AdaBoost. AdaBoost chains hundreds of stumps, each one focusing on the mistakes of the previous ones. Individually weak, collectively strong. The low variance of each stump means the ensemble doesn't overfit even with hundreds of them."

Q9 Why is the decision tree algorithm called "greedy"? What is the consequence? ▼
Must-have points
  • At each node, picks the split with highest Information Gain right now — no lookahead
  • Has no idea whether a lower-gain split today enables better splits two levels down
  • Result: the tree is locally optimal at each node but may not be globally optimal overall
  • Consequence: different splits can lead to shallower, cleaner trees that the greedy algorithm misses
  • This is a core reason ensembles (Random Forest, Boosting) outperform single trees
Strong answer

"Greedy means the algorithm picks the best split available right now without considering future consequences. A split with gain 0.30 is chosen over one with gain 0.20 — but the 0.20 split might immediately create a pure child and finish in depth 2, while the 0.30 split needs depth 4 to get the same result. The tree can't know this without exhaustive lookahead, which is computationally infeasible. This is why Random Forest and Boosting exist — they compensate for this limitation."

Q10 Explain bias vs variance tradeoff in the context of decision trees. ▼
Must-have points
  • Bias = how systematically wrong the model is — misses the real pattern (underfitting)
  • Variance = how much predictions change when trained on different data (overfitting)
  • Shallow tree → high bias, low variance (too simple, misses pattern)
  • Deep tree → low bias, high variance (flexible but memorises noise)
  • Connection to ensembles: Random Forest reduces variance by averaging; Boosting reduces bias by adding weak learners
Strong answer

"A shallow tree has high bias — it's too rigid to capture the real pattern, making systematic errors regardless of which training data it sees. A deep tree has low bias but high variance — it's flexible enough to fit any pattern but so flexible it also fits the noise, so small changes in training data produce very different trees. This is why Random Forest averages many deep low-bias trees to reduce variance, while Boosting chains many shallow high-bias trees to reduce bias."

Q11 What is cost-complexity pruning? How does ccp_alpha work? ▼
Must-have points
  • Post-pruning — grow full tree first, then remove low-value branches
  • Formula: Cost = weighted leaf impurity + α × number of leaves
  • Prune a subtree when collapsing it to one leaf reduces or equals the cost
  • Effective alpha = impurity increase / leaves removed — the exact α at which a subtree gets pruned
  • α=0 → no pruning; large α → collapses to single leaf (predict majority always)
  • Find best α using cross-validation over the pruning path
  • Same concept as XGBoost's gamma (γ) parameter
Strong answer

"Cost-complexity pruning grows the full tree first, then iteratively removes the weakest subtree — the one where pruning causes the smallest impurity increase per leaf removed. The alpha parameter controls how aggressively to penalize extra leaves. Each subtree has an 'effective alpha' — the exact threshold at which pruning becomes worthwhile. You find the best alpha via cross-validation. This is also what XGBoost's gamma does — any leaf whose gain is below gamma gets pruned."

Q12 How does a regression tree differ from a classification tree? ▼
Must-have points
  • Leaf stores mean of target values (not majority class)
  • Split criterion is MSE not Gini/Entropy — minimises (1/n)×Σ(yᵢ−ȳ)²
  • Gain = MSE(parent) − weighted MSE(children)
  • Output is always a staircase — constant prediction per leaf region
  • More depth = more steps = closer approximation of true curve
  • Gini/Entropy can't be used — they require class probabilities, not continuous values
Strong answer

"Three differences: the leaf stores the mean of targets instead of majority class; splits minimize MSE reduction instead of Gini/Entropy (since there are no class probabilities for continuous targets); and the error metric is RMSE/MAE instead of accuracy. The prediction is always a staircase — constant within each leaf region. More depth means more steps and a closer approximation of the true function, but too many steps overfits noise."

Q13 What are the advantages and disadvantages of decision trees? ▼
Must-have points
  • Advantages: interpretable (each prediction is a path of rules), handles non-linear relationships, no normalisation needed, works for both classification and regression, handles mixed feature types, provides feature importance
  • Disadvantages: overfits easily, high variance (unstable — small data changes = different tree), feature dominance, axis-aligned splits only (staircase approximation of diagonal boundaries), poor at simple linear relationships
  • Key insight: decision trees are rarely used alone — their value is as a building block for ensembles
Strong answer

"Main advantages: interpretable (you can trace any prediction as a sequence of if-else rules), handles non-linearity without feature engineering, no need to normalise features, works for both classification and regression. Main disadvantages: overfits easily needing careful pruning, high variance (small data changes produce very different trees), can't draw diagonal boundaries without many splits, and tends to be dominated by the strongest feature. In practice, I'd almost always use Random Forest or XGBoost over a single tree."

Q14 How is feature importance calculated in a decision tree? ▼
Must-have points
  • Method is called Mean Decrease in Impurity (MDI) or Gini importance
  • Formula: for each feature, sum (nₜ/N) × ImpurityReduction(t) across all nodes where that feature was used
  • Weighted by proportion of samples reaching each node — nodes deeper in tree with fewer samples contribute less
  • Normalised to sum to 1.0
  • Caveat: biased toward high-cardinality features (many unique values)
  • Single tree importances are noisy — prefer Random Forest importances or permutation importance
Strong answer

"For each feature, sum the weighted impurity reduction across every node where that feature was used as the split. Weighted by the proportion of samples reaching that node — so a root-level split matters more than a deep split on few samples. Normalise to sum to 1. The caveat is MDI is biased toward high-cardinality features and is noisy from a single tree. For real feature selection I'd use Random Forest importances averaged over hundreds of trees, or permutation importance which measures actual predictive value on test data."

Q15 Can the same feature appear multiple times in one root-to-leaf path? Does it make sense? ▼
Must-have points
  • Yes — a feature can appear multiple times in the same path
  • Only makes sense if the threshold is different each time (further refinement)
  • Redundant threshold (e.g. Credit Score > 600 after Credit Score > 700) would have gain = 0, so the tree never actually does it
  • Consequence: a dominant feature can appear at every level, overshadowing others
  • Random Forest's max_features solves this by randomly excluding features at each node
Strong answer

"Yes — a feature can appear at depth 1 as 'Credit Score > 700' and at depth 3 as 'Credit Score > 750'. Each split further refines the range — meaningful. But a redundant threshold like 'Credit Score > 600' after 'Credit Score > 700' has zero gain so the tree never actually does it. The real problem is feature dominance — a single strong feature can appear at every level, drowning out other signals. Random Forest fixes this with max_features — randomly excluding features at each node, forcing others to contribute."

Q16 Why can't decision trees draw diagonal decision boundaries? ▼
Must-have points
  • Every split is on one feature with one threshold — always parallel to a feature axis
  • This means boundaries are always horizontal or vertical lines (axis-aligned)
  • A diagonal boundary requires both features simultaneously — not possible in one split
  • The tree approximates a diagonal with a staircase — needs many splits
  • More depth needed = higher risk of overfitting for diagonally-separable data
  • SVMs with linear kernel, logistic regression can draw diagonal boundaries in one step
Strong answer

"Each split uses exactly one feature — so every boundary is parallel to one axis (horizontal or vertical). A diagonal boundary like 'approve if Age + Income > threshold' requires both features together, which no single split can express. The tree approximates it with a staircase of many axis-aligned cuts, which needs more depth and therefore more risk of overfitting. This is a fundamental limitation compared to SVMs or logistic regression which can draw diagonal boundaries directly."

Q17 You have a dataset with 500 features. What problems do you expect training a decision tree and how do you fix them? ▼
Must-have points
  • Computation cost — 500 features × N thresholds per node → use max_features to subsample
  • Overfitting — default settings with 500 features will produce deeply overfit tree → tune max_depth, min_samples_leaf, ccp_alpha
  • Curse of dimensionality — data is sparse in 500D space, splits become unreliable → dimensionality reduction (PCA) before training
  • Spurious splits — with 500 features, some will look predictive by chance → feature selection
  • Unreliable feature importances — noisy from single tree → use Random Forest
  • Bonus: suggest Random Forest from the start for 500 features
Strong answer

"Five problems: first, computation — 500 features means millions of gain calculations per node, fix with max_features=sqrt. Second, severe overfitting — fix with max_depth and min_samples_leaf. Third, curse of dimensionality — 500D space is very sparse so splits become unreliable, fix with PCA or feature selection. Fourth, spurious correlations — some of 500 features will look predictive by chance. Fifth, unreliable feature importances from a single tree. Honestly for 500 features I'd use Random Forest from the start."

Q18 Why does XGBoost always use regression trees internally, even for classification? ▼
Must-have points
  • GBM/XGBoost fits residuals at each step — residuals are continuous numbers, not class labels
  • Residuals for classification: actual(1/0) − predicted_probability → continuous values like +0.4, −0.6
  • Classification trees predict discrete classes — can't predict continuous residuals
  • So regression trees are used at every step regardless of the final output type
  • Class labels are only used once — when computing residuals (actual − predicted probability)
  • Final classification: if final probability > 0.5 → class 1, else class 0
Strong answer

"In gradient boosting, every tree after the first is trained to predict the residuals — the difference between the actual value and the current ensemble's prediction. Even for classification, the residual is computed as actual(1/0) minus predicted probability — a continuous number like +0.4 or -0.6. A classification tree can only predict discrete classes, not continuous residuals. So every internal tree in GBM and XGBoost must be a regression tree. The class labels only appear once: when computing residuals. The final prediction converts the summed log-odds back to a probability via sigmoid."

Q19 What is the effective alpha in cost-complexity pruning? Walk through a numerical example. ▼
Must-have points
  • Effective alpha = the exact α value at which pruning a specific subtree becomes worthwhile
  • Formula: Effective α = (impurity increase from pruning) / (number of leaves removed)
  • Impurity increase = Gini(collapsed leaf) − weighted Gini(subtree leaves)
  • Algorithm finds the subtree with the lowest effective alpha and prunes it first
  • Repeats until tree is a single root — generates a sequence of alpha thresholds
  • You pick the best alpha from this sequence using cross-validation
Strong answer
Left subtree (Income > 40k), 40 samples:
  Weighted impurity of 2 leaves = 0.245
  Collapsed leaf impurity        = 0.256

Effective α = (0.256 − 0.245) / (2 − 1) = 0.011

→ If α ≥ 0.011, pruning this subtree is cheaper than keeping it.
  The complexity saving (1 fewer leaf) outweighs the impurity increase.

The algorithm prunes the subtree with the lowest effective α first,
then recalculates for remaining subtrees. Cross-validation picks
the best α from the resulting sequence.
Q20 How does GBM handle classification using residuals? Walk through the steps. ▼
Must-have points
  • Step 1: Initial prediction = log-odds of positive class in training data, converted to probability via sigmoid
  • Step 2: Residuals = actual (1 or 0) − predicted probability → continuous values
  • Step 3: Fit a regression tree to residuals — leaf stores mean residual
  • Step 4: Update: new log-odds = old log-odds + η × tree output
  • Step 5: Convert back to probability via sigmoid. Compute new residuals. Repeat.
  • Everything happens in probability/log-odds space — class labels only used once for residual calculation
  • This is why it's called "Gradient" Boosting — residuals are the gradient of the log loss
Strong answer
Step 1: Base prediction
  60/100 samples Approve → log-odds = log(0.6/0.4) = 0.405
  sigmoid(0.405) = 0.60 → initial P(Approve) = 0.60 for all samples

Step 2: Residuals
  Approved sample: 1 − 0.60 = +0.40 (underestimated)
  Denied sample:   0 − 0.60 = −0.60 (overestimated)

Step 3: Fit regression tree to [+0.40, −0.60, +0.40 ...]
  Each leaf stores mean residual of its samples

Step 4: Update log-odds += η × tree_prediction
  Then sigmoid(new log-odds) → new probability

Step 5: Recompute residuals using new probabilities. Repeat N times.

The name "gradient" boosting: residuals ARE the gradient of log loss.
Q21 What is the MDI feature importance bias and how do you fix it? ▼
Must-have points
  • MDI is biased toward high-cardinality features — features with many unique values have more thresholds to try, so they have more chances to "win" a split by luck
  • Example: a random ID column with 10,000 unique values will appear important even though it has no real predictive power
  • Fix 1: Permutation importance — shuffle one feature's values, measure accuracy drop on test data. Unbiased, uses actual predictive value
  • Fix 2: Use Random Forest importances averaged over many trees — more stable but still has cardinality bias
  • Fix 3: SHAP values — most principled, game-theory based, accounts for feature interactions
Strong answer

"MDI is biased toward high-cardinality features — a feature with 10,000 unique values has 10,000 thresholds to try at each node, so it's more likely to produce a high gain split by chance on training data. A random ID column would look important. The fix is permutation importance: shuffle one feature's values, measure how much test accuracy drops. Big drop = genuinely important feature. This is unbiased because it uses test data and measures actual predictive value. SHAP values are even more principled for production feature analysis."

Q22 What happens to a decision tree when you have highly imbalanced classes (e.g. 95% negative, 5% positive)? ▼
Must-have points
  • Gini impurity is biased toward the majority class — the tree achieves low Gini by predicting majority class everywhere
  • Result: tree learns to always predict "negative" — high accuracy (95%) but useless for detecting positives
  • Fix 1: class_weight="balanced" in sklearn — weights minority class samples more heavily in gain calculation
  • Fix 2: Oversample minority class (SMOTE) or undersample majority class before training
  • Fix 3: Use F1 score or AUC instead of accuracy to evaluate — accuracy is misleading on imbalanced data
  • Fix 4: Adjust decision threshold (predict positive if probability > 0.2 instead of 0.5)
Strong answer

"With 95% negatives, the tree can achieve Gini ≈ 0 by simply predicting 'negative' for everything — the splits look clean because the majority class dominates. The tree learns a biased shortcut. Accuracy is 95% but recall on the positive class is 0%. I'd fix it with class_weight='balanced' in sklearn, which reweights each sample's contribution to the gain calculation inversely proportional to class frequency. I'd also switch to F1/AUC as my evaluation metric and potentially adjust the classification threshold."

Q23 How does a decision tree handle missing values? How does XGBoost handle it differently? ▼
Must-have points
  • sklearn's DecisionTree — does NOT handle missing values natively, requires imputation beforehand
  • XGBoost — learns the best default direction for missing values at each split during training
  • XGBoost tries sending missing values both left and right, picks whichever gives better gain, stores that direction
  • At prediction time, if a feature is missing, the sample automatically goes in the learned direction
  • This means XGBoost can use missing-value patterns as predictive information
  • No preprocessing needed for XGBoost — missingness itself is informative
Strong answer

"sklearn's decision tree requires imputation before training — it breaks on NaN values. XGBoost is fundamentally different: during training, for each split, it tries routing missing values both left and right, whichever gives higher gain becomes the 'default direction' for that node. At prediction time, missing values are automatically routed the learned way. This means XGBoost treats missingness as potentially informative — 'this feature is missing' is itself a signal that can be exploited. No preprocessing needed."

Q24 What is the time complexity of training a decision tree? What makes it expensive? ▼
Must-have points
  • At each node: sort each feature O(N log N), evaluate N thresholds per feature → O(N × F) per node where F = features
  • Total nodes in a balanced tree of depth d: O(2^d)
  • Overall training complexity: roughly O(N × F × N × log N) = O(N² F log N) for full tree
  • Expensive because: feature sorting at every node, exhaustive threshold search, deep trees have exponentially many nodes
  • Fixes: max_features reduces F, max_depth reduces depth, XGBoost pre-sorts into column blocks once
  • Prediction complexity: O(depth) per sample — very fast
Strong answer

"Training is expensive because at every node the algorithm sorts each feature and evaluates N thresholds per feature — roughly O(N×F) gain calculations per node. With 2^d nodes in a full tree, total complexity is approximately O(N² F log N). The bottleneck is the exhaustive threshold search. XGBoost addresses this by pre-sorting features into compressed column blocks once at the start, then reusing the sorted order at every node — this is the main reason XGBoost trains much faster than naive gradient boosting. Prediction is fast — just O(depth) per sample."

Q25 Why is a single decision tree's feature importance unreliable? What's the better alternative? ▼
Must-have points
  • Single tree has high variance — different training sets produce different trees with different importances
  • MDI biased toward high-cardinality features
  • The greedy splits mean important features near the root dominate, obscuring features deeper in the tree
  • Random Forest importances: averaged over hundreds of trees → much more stable, still has cardinality bias
  • Permutation importance: shuffle one feature, measure test accuracy drop → unbiased, uses actual predictive power
  • SHAP values: game-theory based, accounts for feature interactions, most principled — gold standard
Strong answer

"A single tree's importances are unreliable for two reasons: high variance (small data changes produce different trees with different importances) and MDI's bias toward high-cardinality features. Random Forest importances are far more stable because they're averaged over hundreds of trees, though the cardinality bias remains. Permutation importance fixes the bias by measuring actual predictive value on held-out test data — shuffle a feature, measure accuracy drop. SHAP values are the gold standard for production work: they're game-theory based, handle feature interactions properly, and give per-sample explanations."

Q26 Two splits have the same Information Gain. How does the tree break the tie? Does it matter? ▼
Must-have points
  • sklearn breaks ties by returning the first one found in the search order (implementation-dependent)
  • The tie-break is essentially arbitrary — not mathematically principled
  • It can matter in small datasets where ties are more common — leads to instability
  • For large datasets, exact ties are rare for continuous features
  • One alternative: break ties by choosing the split that creates more balanced children (more robust)
  • This is another source of the high variance of decision trees — arbitrary tie-breaking propagates through the whole tree
Strong answer

"sklearn's implementation picks the first one found in the search order — essentially arbitrary. For continuous features exact ties are rare in large datasets. But in small datasets or with categorical features with few values, ties are common and arbitrary tie-breaking is a real source of instability. One principled approach is to break ties by choosing the split creating more balanced children — though sklearn doesn't do this by default. This tie-sensitivity is another reason single decision trees have high variance and why ensemble methods that build many trees are more robust."

Q27 How would you use a decision tree to detect feature interactions? ▼
Must-have points
  • A path from root to leaf implicitly encodes a feature interaction — e.g. "if Age > 30 AND Income > 50k → Approve" uses two features together
  • Features appearing together on the same root-to-leaf path are interacting
  • Shallow tree (depth 2-3) is best for interaction discovery — deeper trees become noise
  • Can extract all root-to-leaf paths as interaction rules
  • Limitation: only captures sequential interactions (first A then B), not arbitrary combinations
  • Better alternatives for interaction detection: SHAP interaction values, H-statistic
Strong answer

"A root-to-leaf path implicitly encodes interactions — if the path is 'Age > 30, then Income > 50k, then Approve,' the tree has discovered that age and income interact to determine approval. I'd train a shallow tree (depth 2-3) to find the strongest interactions without noise. Extract all paths programmatically and look for features appearing together. The limitation is trees only capture sequential, axis-aligned interactions. For richer interaction detection, SHAP interaction values are more powerful — they quantify the joint contribution of every feature pair."

Q28 What is the relationship between decision trees and XGBoost's gamma (γ) parameter? ▼
Must-have points
  • XGBoost's gamma (γ) is the minimum gain required to make a split — exactly like min_impurity_decrease in sklearn
  • XGBoost grows the full tree first, then prunes leaves where gain < γ — this is post-pruning, exactly like ccp_alpha
  • Effective alpha in cost-complexity pruning = effective gamma in XGBoost — same concept, different name
  • γ = 0 means no pruning; larger γ means more aggressive pruning
  • Combined with L1/L2 regularization in the loss function, XGBoost's pruning is more mathematically principled than sklearn's
Strong answer

"XGBoost's gamma is the minimum gain required to keep a split — any leaf with gain below gamma is pruned. This is the same concept as sklearn's ccp_alpha in cost-complexity pruning: both compare the value of a split against a complexity penalty. The difference is XGBoost also has L1 and L2 regularization built directly into the objective function, which penalises leaf weights themselves in addition to the gamma pruning. So XGBoost does post-pruning (gamma) plus regularization of leaf scores (lambda/alpha) — a more complete approach than ccp_alpha alone."

Q29 A decision tree achieves 100% training accuracy. Is this always a problem? ▼
Must-have points
  • Not always — if training data is very clean and the tree is shallow, 100% train accuracy with good test accuracy is fine
  • The real signal is the gap between train and test accuracy — not train accuracy alone
  • 100% train + 60% test = overfitting → problem
  • 100% train + 99% test = may be fine if data is genuinely linearly separable
  • Edge case: if classes are perfectly separable in the feature space, a tree can achieve 100% train and 100% test — no overfitting
  • Context matters: small dataset + deep tree + 100% train = almost certainly overfitting
Strong answer

"Not always. 100% training accuracy is only a problem if test accuracy is significantly lower — the gap is what matters. If data is genuinely separable (e.g. two classes perfectly separated by one feature), a tree can achieve 100% train and 100% test without overfitting. The concern is 100% train with poor test performance — which signals the tree memorised noise. I'd always look at train-test gap, cross-validation variance, and whether the tree depth is proportional to dataset size before concluding there's a problem."

Q30 How would you explain a single prediction from a decision tree to a non-technical stakeholder? ▼
Must-have points
  • Decision trees are uniquely interpretable — every prediction is a path of if-else rules
  • Extract the root-to-leaf path for the specific sample
  • Express each split as a plain-English condition
  • Show the leaf statistics — e.g. "of all past customers with similar profile, 87% were approved"
  • This is one of the biggest advantages over black-box models like neural networks
  • Mention: sklearn's export_graphviz or plot_tree for visual explanation
Strong answer

"This is where decision trees shine over neural networks. For any prediction I can trace the exact path: 'This customer was denied because: their credit score was below 700, AND their income was below 40k. Of all past customers with those two conditions, 89% were denied loans.' I'd visualise the path using plot_tree or export_graphviz and highlight the nodes that were traversed. A stakeholder can verify whether the logic makes business sense and challenge it — that's not possible with a neural network. This explainability is the main reason to use a decision tree even when ensembles would be more accurate."
