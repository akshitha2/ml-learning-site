1Why Plain LLM Prompting Fails for Multi-Step Tasks

A plain LLM call is a single forward pass: prompt in → text out. No mechanism to pause, call an external tool, or use a real result to decide the next step.

Concrete failure: asking a plain LLM "What's NVDA's current stock price, and is it up more than Apple's today?" — with no live data access, it either refuses or hallucinates plausible-looking numbers. Even giving both prices upfront only works because you did the multi-step fetching — the LLM itself can't decide "fetch X, then Y, then compare."

The gap ReAct fills: interleave reasoning with actions, observe real results, and adjust — not generate one static blob of text.

2Loop Mechanics — Thought → Action → Observation

Thought

LLM's internal reasoning, written out before acting — forces planning instead of jumping straight to a tool call, reducing wrong tool selection.

Action

A structured call: tool name + arguments, emitted as parseable text or structured JSON.

Observation

The real-world result, injected back into context by the harness.

Thought: I need France's population before computing 15%.
Action: search("population of France")
Observation: 68,000,000

Thought: Now compute 15% of 68,000,000.
Action: calculator("0.15 * 68000000")
Observation: 10,200,000

Thought: I have enough information.
Final Answer: ~10.2 million
Key mechanical detail: each Thought is conditioned on all prior Observations in context — this is what lets a later action depend on an earlier one's result. Impossible in a single-shot prompt.

Execution & error handling (the harness's role in the loop)

while not done:
    response = llm.generate(context)
    if response.is_final_answer():
        done = True
    else:
        tool_name, args = parse_action(response)
        try:
            result = execute_tool(tool_name, args)
            context += f"Observation: {result}"
        except Exception as e:
            context += f"Observation: Error - {e}"   # fed back, not crashed

Errors are caught and fed back as an Observation, not raised to crash the loop — this lets the LLM self-correct (retry with fixed input, try a different tool) on the next iteration. Needs a max_iterations cap so a malformed call can't loop forever.

3How the LLM Decides "Action" vs "Final Answer"

Not a separate classifier — still next-token prediction. The prompt format defines two mutually exclusive output patterns, and the harness parses which one appeared.

Format A (still working):
Thought: <reasoning>
Action: <tool_name>[<tool_input>]

Format B (done):
Thought: <reasoning>
Final Answer: <answer>
if "Final Answer:" in response:
    done = True
    return extract_final_answer(response)
else:
    action, args = parse_action(response)
    # continue loop
ApproachHow it worksReliability
Text-parsing ReAct (original paper)Plain text with Action:/Final Answer: markers; harness regex-parsesFragile — inconsistent formatting breaks the parser
Native function-calling (modern APIs)Structured JSON via a dedicated tool-call field; absence = final answerRobust — API enforces schema, no regex
Misconception: there is no special "stop token." Native function-calling has two structurally distinct output channels (tool_calls field vs plain text) the model was fine-tuned to choose between — stopping becomes a binary schema check, not string-matching inside free text. This is why text-parsing ReAct can fail to detect "Final Answer:" if phrasing drifts, even though the model effectively finished.

4Worked Example — Multi-Tool Trace

"I have a meeting with a client from Tokyo tomorrow at 9 AM their time. What time is that for me in New York, and will it clash with any of my existing meetings?"

Thought: I need to convert 9 AM Tokyo time to New York time before checking conflicts.
Action: timezone_convert["9:00 AM", "Asia/Tokyo", "America/New_York"]
Observation: 7:00 PM (previous day, due to date line + 13hr offset)

Thought: This falls on the previous calendar day for me — need to check that specific date.
Action: calendar_lookup["2026-07-01", "6:30 PM - 8:00 PM"]
Observation: Existing event: "Team Sync" 7:00-7:30 PM, 2026-07-01

Thought: Direct overlap found. I have enough information.
Final Answer: The 9 AM Tokyo meeting is 7:00 PM in New York, but on July 1st (not July 2nd).
It conflicts with your "Team Sync" (7:00-7:30 PM) that day.

Demonstrates:

  • Data dependency across tools — calendar_lookup's date couldn't be built until timezone_convert's result was known.
  • Mid-trace reasoning correction — caught that "tomorrow" ≠ same calendar date after conversion.
  • Final Answer synthesizes across observations, not just relays the last one.
If the tool returns multiple results (e.g. two overlapping events): no new tool or harness change needed — the Observation string just carries more information, and the next Thought reasons over it. A new tool is only needed if the agent must act (reschedule), not just report.

5Failure Modes

1. Infinite / non-terminating loops

  • Semantic non-convergence: keeps rephrasing a failed query, never concludes "give up."
  • Oscillation: alternates between two actions with no progress.
  • Mitigation: hard max_iterations cap + full history in context so repetition is visible to the model.

2. Hallucinated actions

  • LLM references a tool that doesn't exist or gives malformed/impossible arguments.
  • Hallucinated success: skips the Action entirely, fabricates a plausible Observation, proceeds as if real.
  • Mitigation: strict schema validation before execution; native function-calling constrains what's generatable.

3. Bad / misleading observations

  • Not a model failure — the tool itself returns technically-successful but wrong/stale data (outdated cache, wrong units).
  • LLM has no way to know it's wrong — trusts it, compounds the error into the Final Answer.
  • Why "double-checking" doesn't fix it: the LLM has no independent ground truth — re-reading the same wrong text just re-confirms it. Needs an independent verification source, freshness/provenance metadata, or confidence signaling from the tool — not a retry with the same tool (same stale source → same wrong answer).
Key distinction: failure modes 1 & 2 are about the LLM's behavior; failure mode 3 is about tool/data reliability — fixing one doesn't fix the other; each needs separate guardrails.

6ReAct vs Alternatives

Chain-of-ThoughtReActFunction-Calling Only
MechanismText reasoning, no execution — facts from training dataInterleaves reasoning + real tool executionDirect tool calls, no verbalized reasoning
StrengthGood for pure reasoning/mathGrounded, self-correcting via observed errorsFast, cheap, clean for simple tasks
WeaknessNo live data access, hallucinates factsMore latency/cost; can loop/oscillateNo reasoning trace; worse at multi-step planning
Misconception: ReAct and function-calling are not competing paradigms. ReAct is the reasoning pattern; function-calling is an implementation mechanism for the Action step. You can have function-calling without ReAct (no Thought), and ReAct via text-parsing or native function-calling.

Compliance-sensitive agents specifically prefer ReAct over function-calling-only despite the cost: function-calling logs show what was called, not why. Compliance/audit needs the causal reasoning chain — "flagged because X, Y, and Z" — which only the Thought history provides.

7Interview Traps & Misconceptions

"ReAct requires a fine-tuned model"

It's a prompting pattern, not a training method — any capable LLM can do it via few-shot/system instructions.

"More Thought steps = better"

Redundant reasoning increases cost/latency and can increase hallucination risk if the model reasons itself into a wrong conclusion without new grounding data.

Confusing ReAct with RAG

RAG is a single retrieval step before generation. ReAct is a loop — retrieval/action can happen multiple times, conditionally, with each step depending on prior observations.

"The agent knows when it's wrong"

Self-correction only works if a new Observation contradicts a prior assumption. No new grounding data = no mechanism to detect its own error.

Treating max_iterations as optional

Common take-home gap: happy-path demo works, safety net forgotten. Interviewers probe "what happens if this never converges?"

Assuming linear scaling with complexity

Cost/latency grows with iteration count, not task complexity directly — ambiguous tool schemas can spiral a "simple" task into many iterations.

8Remembering "What is the Harness?"

🧠 The LLM = The Customer

Decides what it wants ("calculator do 47×89") and reasons about the meal so far. Never touches the kitchen.
⇄

🧑‍🍳 The Harness = The Waiter

Takes the order to the kitchen (executes the tool), catches problems (error handling), carries the real dish back (Observation).
Memory hook: "The LLM only ever talks. The harness is the only thing that ever does."

In scope: tool execution, error catching, context/state management, parsing Action vs Final Answer, enforcing max_iterations, safety/guardrail validation, decision-trace logging.

Hallucinated actions

Nonexistent tool calls or fabricated Observations. Fix: schema validation + native function-calling.

Bad/stale observations trusted blindly

1130 Interview Questions

Fundamentals (1–8)
1. What problem does ReAct solve that plain prompting cannot? Easy
Plain prompting can't pause to fetch real data and use it to decide the next step — it only pattern-completes on static context, leading to hallucination on anything requiring live/external info.
2. Name the three components of the ReAct loop and which is the harness's job. Easy
Thought (LLM reasoning), Action (LLM-issued tool call), Observation (harness executes and injects the real result back). The harness executes the Action and produces the Observation.
3. Does ReAct require a fine-tuned model? Easy
No — it's a prompting pattern, achievable via system instructions or few-shot examples.
4. How does the model "decide" between an Action vs a Final Answer? Medium
Not a separate classifier — the prompt format defines two mutually exclusive patterns; the model learns to emit one, and the harness checks which appeared.
5. Why is text-parsing ReAct fragile vs native function-calling? Medium
Stopping requires the model to emit an exact string inside free text — phrasing drift can make the parser miss it. Native function-calling makes stopping a binary schema check.
6. Is ReAct the same as RAG? Easy
No. RAG is a single retrieval step before generation. ReAct is a loop where retrieval/action can happen multiple times, conditionally.
7. Difference between the ReAct "pattern" and "function-calling" mechanism? Medium
ReAct is the reasoning pattern; function-calling is one implementation of the Action step. You can have function-calling without ReAct, and ReAct via either implementation.
8. Why does each Thought "see" prior Observations, and why does it matter? Medium
Full history is resent each turn — this lets a later action depend on an earlier tool's result, impossible in a single-shot prompt.
Harness & Mechanics (9–16)
9. Define "harness" precisely. Easy
Everything around the LLM that isn't the forward pass: tool execution, error catching, context/state management, parsing, loop control, guardrails, logging.
10. If a tool call throws an exception, what should the harness do? Medium
Catch it and feed the error back as an Observation so the LLM can self-correct next turn — not crash the loop.
11. Why is max_iterations necessary even with perfect error handling? Medium
Error handling only catches execution failures; a model can succeed at every step while never semantically converging.
12. Risk of the harness blindly retrying a failed action with the same input? Medium
The harness has no semantic understanding of why it failed — only the LLM can decide the right adaptation.
13. What is a "decision trace" and why does it matter for compliance? Medium
The logged Thought/Action/Observation history — used for auditability, e.g. "why did the agent deny this loan."
14. Give an example of a harness guardrail before executing an Action. Medium
Validating tool arguments — e.g. blocking a delete_file action on a restricted path, or rejecting an unknown tool name.
15. Does the loop structure change if a tool returns multiple results? Hard
No — only the Observation's content changes. The harness's job and loop mechanics stay identical.
16. Why must context include full history, not just the latest Observation? Hard
Needed for the model to detect its own repetition/oscillation and reason over multi-step dependencies.
Failure Modes & Reliability (17–23)
17. Name three distinct failure categories in ReAct agents. Medium
Infinite/non-terminating loops, hallucinated actions (including hallucinated success), and bad/stale observations trusted blindly.
18. Why can't "double-checking" fix a stale/wrong Observation? Hard
The LLM has no independent ground truth — it can only re-reason over the same possibly-wrong text, catching internal inconsistencies at best.
19. Why doesn't re-fetching with the same tool fix stale data? Medium
If the tool is the stale source, retrying returns the same wrong answer — needs a different information channel.
20. What is "hallucinated success" vs a hallucinated action? Hard
Hallucinated action calls a nonexistent tool/bad args. Hallucinated success skips the tool entirely and fabricates a plausible Observation.
21. What causes "oscillation," distinct from simple non-convergence? Hard
The model alternates between two actions where each result triggers reverting to the other — no progress despite each step "succeeding."
22. Does fixing execution errors also fix infinite loops? Hard
No — execution-error handling and loop-control are separate guardrail layers.
23. Why does cost scale with iteration count, not task complexity directly? Hard
Ambiguous tool schemas or misread Observations can spiral a "simple" task into many iterations — the cost driver is loop turns, not inherent difficulty.
Comparisons & System Design (24–30)
24. When would you choose function-calling-only over full ReAct? Medium
Simple, single-tool lookups where reasoning-transparency overhead isn't needed.
25. Why is ReAct preferred over function-calling-only for compliance-sensitive agents? Medium
Function-calling logs show what was called, not why. Compliance needs the causal reasoning chain from the Thought history.
26. Design: your agent must both report and resolve a scheduling conflict — what changes? Hard
Reporting needs no change (richer Observation). Resolving requires a new tool (e.g. reschedule_event) since the agent needs a new capability to act.
27. Why prefer a locally-hosted LLM over an API-based one for agentic workflows? Medium
Data sovereignty — sensitive compliance/healthcare data can't leave enterprise infrastructure; supports regulatory compliance and full audit control.
28. Red flag if a candidate skips max_iterations/error feedback in a design? Medium
Signals they've only seen a happy-path demo, not a production-grade agent.
29. Why must the Final Answer synthesize across all Observations, not just the last? Hard
Earlier Observations may carry information the last step doesn't restate — dropping them loses context the user needs.
30. Trade-off of adding more Thought steps? Hard
Gains: more auditable, sometimes catches mid-trace errors. Loses: higher cost/latency, and can increase hallucination risk without new grounding data.
