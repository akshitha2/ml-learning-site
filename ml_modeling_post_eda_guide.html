Step 1

Train/Test Split & Cross-Validation

from sklearn.model_selection import train_test_split X_train, X_test, y_train, y_test = train_test_split( X, y, test_size=0.2, # keyword, not positional -- 3rd positional arg IS test_size stratify=y, # preserves class ratio -- essential for imbalanced data (fraud, churn) random_state=42 # reproducibility )

cross_val_score vs GridSearchCV

ToolPurposeReturns
cross_val_scoreEvaluate ONE model config across k folds — no tuningArray of k scores
GridSearchCVSearch a param grid, using CV internally to pick the best comboBest estimator + best params
from sklearn.model_selection import cross_val_score from sklearn.ensemble import RandomForestClassifier scores = cross_val_score(RandomForestClassifier(random_state=42), X_train, y_train, cv=5, scoring='roc_auc') print(f"Mean AUC: {scores.mean():.3f}, Std: {scores.std():.3f}")
Quick Check
Q. Why stratify on y when splitting fraud data?
A. Fraud is rare (often <1%). Without stratify, a random split could put almost no fraud cases in the test set, making evaluation unreliable.
Q. When would you use GridSearchCV over cross_val_score?
A. When you need to compare multiple hyperparameter combinations, not just measure one fixed model's stability.
Interview Trap

Passing test_size as a positional argument: train_test_split(X, y, 0.8, stratify=y) sets test_size=0.8, silently inverting your train/test ratio. Always use test_size= as a keyword.

Step 2

Preprocessing — Scaling, Encoding, ColumnTransformer

from sklearn.preprocessing import StandardScaler, OneHotEncoder scaler = StandardScaler() X_train_scaled = scaler.fit_transform(X_train) # fit + transform on TRAIN only X_test_scaled = scaler.transform(X_test) # transform (not fit) on TEST

ColumnTransformer — mixing numeric & categorical columns

from sklearn.compose import ColumnTransformer # NOT sklearn.pipeline preprocessor = ColumnTransformer([ ('num', StandardScaler(), ['txn_amount', 'txn_hour']), # (name, transformer, columns) ('cat', OneHotEncoder(handle_unknown='ignore'), ['merchant_category']) ]) preprocessor.fit(X_train) # fit on the FULL feature frame -- columns are picked internally
💡

Each ColumnTransformer entry is a 3-item tuple: (name, transformer, column_list). Forgetting the column list, or forgetting commas between tuples, is the most common syntax slip.

Quick Check
Q. Why call .transform() (not .fit_transform()) on the test set?
A. Fitting on test data means the scaler's mean/std (or encoder's categories) would be influenced by data the model shouldn't see yet — a form of data leakage.
Q. Why use handle_unknown='ignore' in OneHotEncoder?
A. If a category appears in test/production data that wasn't seen during training (e.g., a new merchant category), the encoder would otherwise throw an error. 'ignore' encodes it as all-zeros instead.
Interview Trap

ColumnTransformer lives in sklearn.compose, not sklearn.pipeline — a very common import mix-up.

Step 3

Pipelines

from sklearn.pipeline import Pipeline from sklearn.preprocessing import StandardScaler from sklearn.linear_model import LogisticRegression pipe = Pipeline([ ('scaler', StandardScaler()), ('model', LogisticRegression()) ]) pipe.fit(X_train, y_train) pipe.predict(X_test)

Tuning a pipeline with GridSearchCV — the step__param convention

from sklearn.model_selection import GridSearchCV param_grid = {'model__C': [0.01, 0.1, 1, 10]} # stepname__paramname -- double underscore grid = GridSearchCV(pipe, param_grid, cv=5, scoring='recall') grid.fit(X_train, y_train) grid.best_params_ grid.best_estimator_
💡

Pipeline.fit() calls fit_transform() on every step except the last, and fit() on the last. Pipeline.predict() calls transform() on every step except the last, then predict() on the last. This is what guarantees train-only fitting.

Quick Check
Q. Why does GridSearchCV over a pipeline prevent leakage during cross-validation, but manual scaling before CV doesn't?
A. Inside GridSearchCV, the pipeline is refit on each fold's training portion only. If you scale the whole dataset once before CV, every fold's "training" data has already seen the full dataset's statistics.
Interview Trap

Forgetting the stepname__ prefix in the param grid — plain {'C': [...]} will raise an error because GridSearchCV doesn't know which pipeline step C belongs to.

Step 4

Linear Regression

from sklearn.linear_model import LinearRegression model = LinearRegression() model.fit(X_train, y_train) model.coef_ # one weight per feature model.intercept_ # the bias term (b) y_pred = model.predict(X_test)

Intuition: predicts y = b + w1*x1 + w2*x2 + ... + wn*xn. Each coef_ tells you how much y changes for a 1-unit change in that feature, holding others constant.

Key assumptions (commonly asked)

  • Linearity — relationship between X and y is linear
  • Independence — residuals aren't correlated with each other
  • Homoscedasticity — residual variance is constant across all predicted values
  • Normality of residuals — errors are roughly normally distributed
  • No multicollinearity — features aren't highly correlated with each other

Regularized variants

from sklearn.linear_model import Ridge, Lasso, ElasticNet ridge = Ridge(alpha=1.0) # L2 penalty -- shrinks coefficients, keeps all features lasso = Lasso(alpha=1.0) # L1 penalty -- can shrink coefficients to exactly 0 (feature selection) elastic = ElasticNet(alpha=1.0, l1_ratio=0.5) # mix of L1 + L2
Quick Check
Q. Why must you scale features before Ridge/Lasso but it doesn't matter for plain LinearRegression?
A. Regularization penalizes the size of coefficients directly. Unscaled features with larger numeric ranges get artificially smaller coefficients and are penalized less — scaling puts every feature on equal footing before the penalty is applied.
Q. Multicollinearity doesn't hurt prediction accuracy much — so why does it matter?
A. It makes individual coefficients unstable and hard to interpret (they can flip sign or swing wildly with small data changes), even though overall model fit/prediction may stay fine.
Interview Trap

Fitting Ridge/Lasso on unscaled data and then trying to compare coefficient magnitudes to judge feature importance — without scaling, this comparison is meaningless.

Step 5

Regression Metrics

from sklearn.metrics import mean_absolute_error, mean_squared_error, r2_score import numpy as np mae = mean_absolute_error(y_test, y_pred) mse = mean_squared_error(y_test, y_pred) rmse = np.sqrt(mse) # or mean_squared_error(y_test, y_pred, squared=False) in older versions r2 = r2_score(y_test, y_pred)
MetricFormula (intuition)When to prefer it
MAEAverage of |actual − predicted|Errors treated equally; robust to outliers
MSEAverage of (actual − predicted)²Penalizes big errors more; used as a training loss
RMSE√MSE — same units as yInterpretable version of MSE; still outlier-sensitive
R²1 − (SS_residual / SS_total) — fraction of variance explainedQuick "goodness of fit" summary, 0 to 1 (can go negative)
Adjusted R²R² penalized for number of predictorsComparing models with different numbers of features

Numerical example: actual = [10, 20, 30], predicted = [12, 18, 33]. Errors = [2, -2, 3]. MAE = (2+2+3)/3 = 2.33. MSE = (4+4+9)/3 = 5.67. RMSE = √5.67 ≈ 2.38.

💡

R² always increases (or stays the same) when you add more features, even useless ones — that's why Adjusted R² exists: it subtracts a penalty proportional to the number of predictors, so it can actually decrease if a new feature doesn't help.

Quick Check
Q. Your model has MAE=5 and RMSE=15 on the same dataset. What does that gap tell you?
A. RMSE >> MAE signals a few large errors (outliers) are inflating RMSE, since squaring amplifies big residuals much more than small ones.
Interview Trap

Reporting R² alone as "model quality" without checking residual plots — a high R² can still hide a model that systematically over/under-predicts in certain ranges (non-random residual pattern).

Step 6

Decision Trees

from sklearn.tree import DecisionTreeClassifier, DecisionTreeRegressor clf = DecisionTreeClassifier( criterion='gini', # or 'entropy' -- both measure node impurity max_depth=5, # limits tree depth -- main lever against overfitting min_samples_split=10, # min samples required to split a node min_samples_leaf=5, # min samples required at a leaf random_state=42 ) clf.fit(X_train, y_train)

Gini vs Entropy

CriterionFormulaNotes
Gini1 − Σ(pᵢ²)Faster to compute (no log); default in sklearn
Entropy−Σ(pᵢ · log₂ pᵢ)Information-theoretic; slightly more computationally expensive, usually similar results

Numerical example (Gini): a node with 70% class A, 30% class B → Gini = 1 − (0.7² + 0.3²) = 1 − (0.49+0.09) = 0.42. A pure node (100% one class) has Gini = 0.

Quick Check
Q. Why does an unpruned decision tree tend to overfit?
A. Without depth/leaf-size limits, it keeps splitting until nodes are pure (often 1 sample per leaf), effectively memorizing the training data including its noise.
Q. Do decision trees need feature scaling?
A. No — trees split on thresholds per feature independently, so the relative scale of one feature vs another doesn't affect the splits.
Interview Trap

Assuming a single deep decision tree will generalize well because it fits training data perfectly — a 100% training accuracy on an unpruned tree is itself a red flag for overfitting, not a good sign.

Step 7

Random Forest & XGBoost

from sklearn.ensemble import RandomForestClassifier model = RandomForestClassifier( n_estimators=200, max_depth=8, class_weight='balanced', # handles imbalance -- reweights classes inversely to frequency random_state=42 ) model.fit(X_train, y_train)
from xgboost import XGBClassifier xgb = XGBClassifier( n_estimators=200, learning_rate=0.1, scale_pos_weight=10, # ~ (negative count / positive count) for imbalance eval_metric='auc', random_state=42 ) xgb.fit(X_train, y_train) y_proba = xgb.predict_proba(X_test)[:, 1] # probability of the POSITIVE (fraud) class

Quick Check
Q. Why use predict_proba instead of predict for a fraud model?
A. Probabilities let you tune the decision threshold based on business cost — e.g., flag as fraud at prob > 0.3 instead of the default 0.5 if missing fraud (false negatives) is costlier than false alarms.
Interview Trap

Printing raw model.feature_importances_ — it's an unlabeled numpy array. Without pairing it with X_train.columns, you can't say which importance belongs to which feature.

Step 8

Classification Metrics

from sklearn.metrics import ( confusion_matrix, ConfusionMatrixDisplay, precision_score, recall_score, f1_score, roc_auc_score, classification_report ) # convention: TRUE label always comes first cm = confusion_matrix(y_test, y_pred) precision = precision_score(y_test, y_pred) recall = recall_score(y_test, y_pred) f1 = f1_score(y_test, y_pred) auc = roc_auc_score(y_test, y_proba) # needs PROBABILITIES, not hard labels disp = ConfusionMatrixDisplay(cm) disp.plot()
MetricFormulaInterpretation
PrecisionTP / (TP + FP)Of everything flagged fraud, how much really was
RecallTP / (TP + FN)Of all real fraud, how much did we catch
F12·(P·R)/(P+R) — harmonic mean, not geometricBalances precision and recall in one number

Numerical example: Precision = 0.8, Recall = 0.6 → F1 = 2·(0.8·0.6)/(0.8+0.6) = 0.96/1.4 ≈ 0.686. (Geometric mean would give √(0.8·0.6) ≈ 0.693 — close but not the actual formula; memorize "harmonic.")

💡

Accuracy is misleading on fraud data: predicting "not fraud" for everything on a 99%-legitimate dataset still gives 99% accuracy while catching zero fraud. Use Precision/Recall/F1/AUC-PR instead. For imbalanced data, AUC-PR is preferred over AUC-ROC, since ROC's false-positive-rate denominator (all negatives) is huge and makes the curve look deceptively good.

Quick Check
Q. In fraud detection, would you usually prioritize precision or recall?
A. Recall — missing real fraud (false negative) is typically costlier than a false alarm (false positive), which a human review layer can filter out.
Interview Trap

Swapping argument order — f1_score(y_pred, y_test) instead of f1_score(y_test, y_pred). Since precision/recall aren't symmetric, this silently returns a wrong (but plausible-looking) number. True label always goes first.

Step 9

Hyperparameter Tuning

from sklearn.model_selection import GridSearchCV, RandomizedSearchCV param_grid = { 'n_estimators': [100, 200, 300], 'max_depth': [4, 6, 8] } grid = GridSearchCV(RandomForestClassifier(random_state=42), param_grid, cv=5, scoring='f1', n_jobs=-1) grid.fit(X_train, y_train) grid.best_params_, grid.best_score_
GridSearchCV
  • Tries every combination exhaustively
  • Guaranteed to find the best in the grid
  • Expensive as grid size grows (combinatorial)
RandomizedSearchCV
  • Samples a fixed number of random combinations
  • Much faster on large search spaces
  • May miss the true optimum, usually close enough
Interview Trap

Using accuracy as scoring= for an imbalanced fraud dataset during tuning — this optimizes toward the majority class. Use scoring='f1', 'recall', or 'roc_auc' instead.

Step 10

Model Persistence

import joblib joblib.dump(pipe, 'fraud_pipeline.pkl') # save the WHOLE pipeline, not just the model loaded_pipe = joblib.load('fraud_pipeline.pkl') loaded_pipe.predict(X_test)
💡

Save the full Pipeline (preprocessing + model), not just the classifier — otherwise you'd have to remember and re-implement the exact same scaling/encoding steps separately at inference time.

Interview Trap

joblib.dump(model, ...) just creates a display/result object if misused with the wrong reference — always confirm you're dumping the fitted object (post .fit()), and that you call .load() before calling .predict() on it.

30 Interview Questions

Q1Easy
Write code to split X, y 80/20, stratified, random_state=42.
+
Answer
from sklearn.model_selection import train_test_split X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2, stratify=y, random_state=42)
Q2Easy
Why is accuracy a poor metric for fraud detection?
+
Answer

With ~1% fraud, predicting "not fraud" always still gives ~99% accuracy while catching zero actual fraud — accuracy hides complete failure on the minority class.

Q3Easy
What's the difference between .fit(), .transform(), and .fit_transform()?
+
Answer

.fit() learns parameters (e.g. mean/std) from data. .transform() applies already-learned parameters. .fit_transform() does both in one call — use it on train only; use .transform() alone on test to avoid leakage.

Q4Easy
Write code to train a LogisticRegression model and print its coefficients.
+
Answer
from sklearn.linear_model import LogisticRegression model = LogisticRegression() model.fit(X_train, y_train) print(model.coef_, model.intercept_)
Q5Easy
What does class_weight='balanced' do?
+
Answer

It reweights the loss function inversely proportional to class frequency, so the minority class (fraud) contributes more per-sample penalty, discouraging the model from ignoring it.

Q6Medium
Build a Pipeline with StandardScaler + RandomForestClassifier.
+
Answer
from sklearn.pipeline import Pipeline from sklearn.preprocessing import StandardScaler from sklearn.ensemble import RandomForestClassifier pipe = Pipeline([('scaler', StandardScaler()), ('clf', RandomForestClassifier())]) pipe.fit(X_train, y_train)
Q7Medium
Why does a Pipeline prevent data leakage during cross-validation, while manually scaling once before CV does not?
+
Answer

Inside CV, the pipeline refits the scaler on each fold's training portion only. Manual scaling before CV computes statistics from the full dataset once, so every fold's "unseen" validation data has already influenced the scaling.

Q8Medium
Write a ColumnTransformer applying OneHotEncoder to 'city' and StandardScaler to 'amount' and 'age'.
+
Answer
from sklearn.compose import ColumnTransformer from sklearn.preprocessing import OneHotEncoder, StandardScaler ct = ColumnTransformer([ ('cat', OneHotEncoder(handle_unknown='ignore'), ['city']), ('num', StandardScaler(), ['amount', 'age']) ])
Q9Medium
Explain Gini impurity with a numerical example.
+
Answer

Gini = 1 − Σ(pᵢ²). For a node with 70% class A / 30% class B: Gini = 1 − (0.49 + 0.09) = 0.42. A pure node (100% one class) has Gini = 0 — lower Gini means purer, better split.

Q10Medium
What's the difference between Ridge and Lasso regression?
+
Answer

Ridge (L2) shrinks all coefficients toward zero but rarely to exactly zero. Lasso (L1) can shrink coefficients to exactly zero, effectively performing feature selection.

Q11Medium
Compute MAE, MSE, and RMSE given y_test and y_pred.
+
Answer
from sklearn.metrics import mean_absolute_error, mean_squared_error import numpy as np mae = mean_absolute_error(y_test, y_pred) mse = mean_squared_error(y_test, y_pred) rmse = np.sqrt(mse)
Q12Medium
Why does R² always increase (or stay flat) as you add more features, and what do you use instead?
+
Answer

More features give the model more ways to fit training data noise, so residual sum of squares can only shrink or stay the same. Use Adjusted R², which penalizes the score for the number of predictors.

Q13Medium
What is the F1 score's formula, and why is it the "harmonic" mean and not a simple average?
+
Answer

F1 = 2·(P·R)/(P+R). The harmonic mean punishes large gaps between precision and recall more than a simple average would — a model with precision=1.0 and recall=0.1 gets F1≈0.18, not 0.55, correctly reflecting poor overall performance.

Q14Medium
Write code for GridSearchCV tuning C in [0.01, 0.1, 1, 10] over a Pipeline with step name 'model'.
+
Answer
from sklearn.model_selection import GridSearchCV param_grid = {'model__C': [0.01, 0.1, 1, 10]} grid = GridSearchCV(pipe, param_grid, cv=5, scoring='recall') grid.fit(X_train, y_train)
Q15Medium
Why is AUC-PR preferred over AUC-ROC for imbalanced data like fraud?
+
Answer

ROC-AUC's x-axis (false positive rate) divides by the total negative count, which is huge when negatives dominate — so even a model producing many false positives in absolute terms can still show a good-looking ROC curve. Precision-Recall AUC is more sensitive to performance on the rare positive class.

Q16Medium
Get sorted, labeled feature importances from a fitted RandomForestClassifier.
+
Answer
import pandas as pd importances = pd.Series(model.feature_importances_, index=X_train.columns) importances.sort_values(ascending=False, inplace=True)
Q17Medium
Why don't decision trees require feature scaling?
+
Answer

Trees split on threshold comparisons per feature independently (e.g. amount > 500), so the absolute scale of a feature doesn't change which split is chosen — unlike distance-based or gradient-based models.

Q18Medium
Save a fitted pipeline to disk and reload it to make predictions.
+
Answer

Q19Hard
Random Forest vs XGBoost — walk through the full trade-off for a fraud model.
+
Answer

RF bags parallel trees (faster training, fewer hyperparameters, more robust to noisy/mislabeled points, out-of-the-box stable). XGBoost boosts sequentially, correcting residuals (typically higher raw accuracy on tabular data via regularization, but more tuning needed and more sensitive to noisy labels since it keeps re-fitting errors). In banking, also weigh regulatory explainability — SHAP works with both but is more commonly paired with XGBoost in practice, and model risk teams often want feature-level justification for any flagged transaction.

Q20Hard
Describe an end-to-end AWS architecture for a real-time fraud scoring pipeline using S3, Redshift, SageMaker, and Lambda.
+
Answer

S3 stores raw/processed data and model artifacts. Redshift runs batch SQL for feature engineering (e.g. 30-day transaction aggregates). SageMaker trains the model (Training Jobs) and hosts it for low-latency inference (real-time Endpoint). Lambda is the orchestration/glue layer: triggered by an incoming transaction event, it fetches features and calls the SageMaker endpoint, then returns the fraud decision. SageMaker Model Monitor + CloudWatch handle drift detection and logging; retraining is triggered on a schedule or on drift via EventBridge.

Q21Hard
Why can Redshift not serve as a real-time inference endpoint?
+
Answer

Redshift is a columnar data warehouse optimized for large analytical batch queries, with latency in seconds — not the sub-100ms latency needed for real-time transaction scoring. Real-time serving needs a dedicated low-latency layer like a SageMaker Endpoint.

Q22Hard
List 3 fraud-specific feature engineering ideas beyond raw transaction fields.
+
Answer

(1) Velocity features — number/amount of transactions in the last 1hr/24hr per account. (2) Deviation features — how much this transaction's amount deviates from the account's historical average. (3) Network/graph features — shared device ID, IP, or merchant across multiple accounts (common fraud ring signal).

Q23Hard
Besides resampling and scale_pos_weight/class_weight, what's another way to handle class imbalance?
+
Answer

Threshold tuning post-training — instead of the default 0.5 cutoff on predict_proba, choose a threshold (e.g. via the precision-recall curve) that matches the business cost trade-off between false positives and false negatives. Also: framing fraud as anomaly/outlier detection (e.g. Isolation Forest) rather than pure classification.

Q24Hard
Trace through this code — what's wrong, if anything?
+
scaler = StandardScaler() X_scaled = scaler.fit_transform(X) # fit on FULL dataset X_train, X_test, y_train, y_test = train_test_split(X_scaled, y, test_size=0.2)
Answer

Leakage bug: the scaler is fit on the full dataset (including what becomes the test set) before splitting. The test set's statistics influence the scaling parameters. Correct order: split first, then fit_transform on X_train only, transform on X_test.

Q25Hard
You get 99% recall but only 20% precision on a fraud model. Is that acceptable? What would you check?
+
Answer

Depends on operational cost: 20% precision means 4 out of 5 flagged transactions are false alarms, which could overwhelm a manual review team. Check the cost of a review (cheap → may be fine) vs the cost of a missed fraud (expensive → high recall may still be worth it). Consider adjusting the threshold and adding a review-priority tier rather than treating all flags equally.

Q26Hard
What's the difference between L1 and L2 regularization in terms of the geometry of the constraint region?
+
Answer

L1's constraint region is a diamond (has corners on the axes) — the optimal solution often lands exactly on a corner, zeroing out some coefficients. L2's constraint region is a circle/sphere (smooth, no corners) — the optimum rarely lands exactly on an axis, so coefficients shrink but rarely hit zero.

Q27Hard
How would you explain a single fraud flag to a compliance/regulatory team (explainability)?
+
Answer

Use SHAP values to decompose the model's prediction for that specific transaction into per-feature contributions (e.g. "flagged mainly due to unusually high transaction velocity + new device ID"), giving a human-readable, case-specific justification rather than relying on aggregate feature importances.

Q28Hard
Why might a model with high training accuracy but poor test recall on the fraud class still pass a naive review?
+
Answer

If reviewed only via overall accuracy, an overfit or imbalance-blind model can look excellent while completely failing the minority class — this is why fraud models must always be evaluated per-class (recall/precision on the fraud class specifically), not on aggregate metrics.

Q29Hard
How does SageMaker Model Monitor help after deployment, and why is retraining not just "run it again on a schedule"?
+
Answer

Model Monitor tracks data/prediction drift against a training baseline (e.g. feature distributions shifting as fraud patterns evolve). Pure scheduled retraining can miss sudden drift between schedules or waste resources retraining when nothing changed — drift-triggered retraining (via CloudWatch alarms → EventBridge → SageMaker Pipeline) responds to actual model degradation instead of a fixed calendar.

Q30Hard
Write a full pipeline: ColumnTransformer (scale numeric, one-hot categorical) → RandomForestClassifier, then GridSearchCV over max_depth.
+
Answer
from sklearn.compose import ColumnTransformer from sklearn.preprocessing import StandardScaler, OneHotEncoder from sklearn.ensemble import RandomForestClassifier from sklearn.pipeline import Pipeline from sklearn.model_selection import GridSearchCV preprocessor = ColumnTransformer([ ('num', StandardScaler(), ['txn_amount', 'txn_hour']), ('cat', OneHotEncoder(handle_unknown='ignore'), ['merchant_category']) ]) pipe = Pipeline([ ('prep', preprocessor), ('clf', RandomForestClassifier(random_state=42)) ]) param_grid = {'clf__max_depth': [4, 6, 8, 10]} grid = GridSearchCV(pipe, param_grid, cv=5, scoring='f1') grid.fit(X_train, y_train)
