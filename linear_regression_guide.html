Topic 1

Problem Setup & Intuition

Linear regression predicts a continuous output by fitting the best straight line (or hyperplane) through the data. It doesn't try to hit every point exactly — it captures the general trend.

House size (sqft)Price ($1000s)
1000200
1500280
2000340

A line like ŷ = 0.14x + 60 roughly fits this — for every extra sq ft, price rises ~$140; the slope is the marginal effect of x on y.

💡

1 feature → line in 2D. 2 features → plane in 3D. n features → hyperplane in (n+1)-D space. A line only has 2 degrees of freedom (slope + intercept), so it mathematically can't pass through 3+ non-collinear points — and even when a more flexible model could fit every training point, doing so means fitting the noise, not the true signal.

Quick Check — Intuition
Q. Why does a model that achieves zero training error not necessarily perform well on new data?
A. Because it likely memorized noise specific to the training sample — noise that won't repeat in unseen data — so test error rises even as training error hits zero. This is overfitting.
Q. What does the slope coefficient represent, intuitively?
A. The marginal effect — how much ŷ changes for a one-unit increase in x, holding other features constant.
Interview Trap

Assuming a more flexible model is always better because it fits training data more closely. Flexibility trades training fit for generalization — the real goal is minimizing test error, not training error.

Topic 2

Model Representation

Once we know linear regression is fitting a line/hyperplane, we need a precise way to write down what that line actually is in terms of numbers we can compute with. For a single feature, that's simply:

ŷ = θ₀ + θ₁x

Here θ₀ is the intercept — the predicted value when x=0 — and θ₁ is the slope — how much ŷ changes per unit increase in x. With multiple features (say, house size AND number of bedrooms), the same idea extends by simply adding a term per feature:

ŷ = θ₀ + θ₁x₁ + θ₂x₂ + ... + θₙxₙ

Writing this out term-by-term gets unwieldy once you have dozens of features, so we compress it into a single matrix multiplication — this is the form actually used in code and in every derivation from here on:

ŷ = Xθ
  • X — the design matrix, shape (m, n+1). Each of the m rows is one training example; each of the n+1 columns is one feature — plus one extra column of all 1s, prepended specifically to carry the intercept term (more on why below).
  • θ — the parameter vector, shape (n+1, 1). θ₀ sits at index 0 (the intercept), followed by θ₁...θₙ for each real feature.
  • ŷ — the resulting prediction vector, shape (m, 1) — one predicted value per training example, all computed in a single matrix multiply.

This matrix form is what makes linear regression scale cleanly: whether you have 1 feature or 10,000, it's still just one Xθ operation, and it's the exact form we'll reuse when deriving the gradient (Topic 4) and the Normal Equation (Topic 6).

Worked example

Take two houses, at 1000 and 1500 sqft, with θ₀=60 and θ₁=0.14. Building the design matrix (note the leading column of 1s) and multiplying it by θ gives:

X = [[1, 1000], θ = [60, Xθ = [1(60) + 1000(0.14) = [200, [1, 1500]] 0.14] 1(60) + 1500(0.14)] 270]

So the model predicts $200K for the 1000 sqft house and $270K for the 1500 sqft one — each row of X, multiplied against θ, produces exactly one entry of ŷ.

Common Misconception

"Linear" means linear in the parameters θ, not that the x–y relationship must look like a straight line visually. ŷ = θ₀ + θ₁x + θ₂x² is still linear regression — it's linear in θ₀, θ₁, θ₂, even though it fits a curve.

Quick Check — Model Representation
Q. Why prepend a column of 1s to X instead of handling the intercept separately?
A. It folds θ₀ into the same matrix multiplication as every other parameter — one clean Xθ operation. Without it, the model is forced through the origin (ŷ = 0 whenever all features = 0), which is a real structural constraint, not just a missed constant — the fit gets worse across the whole range of data.
Q. Is polynomial regression (ŷ = θ₀ + θ₁x + θ₂x²) still "linear regression"?
A. Yes — it's linear in the parameters, which is all the "linear" in linear regression requires.
Interview Trap

Forgetting the intercept column and assuming the model is just "off by a constant everywhere." In reality it removes a full degree of freedom, forcing the line through the origin — a structural handicap, not a fixable offset.

Topic 3

Cost Function (Mean Squared Error)

Model Representation gave us a way to produce predictions from any θ — but we still have no way to tell whether one θ is "better" than another. The cost function fixes that: it turns "how good is this line" into a single number we can actually minimize.

J(θ) = (1 / 2m) · Σᵢ (ŷ⁽ⁱ⁾ − y⁽ⁱ⁾)²
  • m — number of training samples
  • ŷ⁽ⁱ⁾ − y⁽ⁱ⁾ — the residual for sample i: how far off the prediction is from the actual value
  • squaring each residual before summing means large errors get penalized far more than small ones, and negative/positive errors can't cancel each other out
  • the leading ½ is purely a convenience constant — it cancels neatly with the "2" that pops out when we differentiate this in Topic 4, and doesn't change which θ minimizes the function

Worked example

Take predictions ŷ = [200, 270] against actual values y = [200, 280] (m=2 samples). The residuals are [0, −10]. Squaring and averaging (with the ½ factor):

J(θ) = (1/2·2)[(200−200)² + (270−280)²] = (1/4)[0 + 100] = 25

The units here are squared (e.g. thousands of dollars, squared), which is why this raw number isn't very interpretable on its own — J(θ)=25 doesn't directly tell you "off by how many dollars." That's exactly what RMSE (Topic 10) fixes, by taking the square root to bring the units back to something meaningful. Here, J(θ) is mainly useful as the thing gradient descent and the Normal Equation are built to minimize, not as a metric you'd report to a stakeholder.

Why squared error, not absolute error (MAE)?

  • Squaring penalizes large errors far more than small ones (residual 10 → contributes 100; residual 2 → contributes 4)
  • MSE is differentiable everywhere (smooth) — MAE has a non-differentiable kink at 0, which breaks gradient-based optimization
  • MSE has a clean closed-form solution (Normal Equation); MAE does not
Prefer MSE when
  • Outliers are genuine, important signal you want the model to capture
Prefer MAE when
  • Outliers are noise/data errors you don't want the model chasing — MAE is more robust
Quick Check — Cost Function
Q. Why isn't "removing negative signs" the real reason MSE is preferred over MAE?
A. Absolute value also removes negative signs — the real reasons are differentiability everywhere and the resulting convex optimization landscape, which guarantees gradient descent finds the global minimum.
Q. A house price dataset has one entry with a decimal-point typo, inflating it 50x. MSE or MAE?
A. MAE — it won't let that single corrupted point disproportionately warp the fit the way MSE would.
Interview Trap

Assuming MSE is "always more sensitive, therefore worse." Sensitivity to large errors is a double-edged sword — good when outliers are real signal, bad when they're noise.

Topic 4

Gradient Descent

The cost function J(θ) tells us how bad a given θ is, but doesn't hand us the best θ directly. Because J(θ) is convex — a smooth, single-bottomed bowl shape over the parameter space — we can find the minimum by starting somewhere and repeatedly taking small steps downhill. That iterative process is gradient descent, and it's the workhorse algorithm behind not just linear regression but most of modern machine learning.

The key tool is the gradient — the multi-dimensional generalization of "slope." At any point θ, the gradient points in the direction of steepest ascent (the way that increases cost fastest). So to go downhill, we simply move in the opposite direction, scaled by how big a step we want to take:

θⱼ := θⱼ − α · ∂J(θ)/∂θⱼ
  • θⱼ — the parameter being updated (θ₀, θ₁, etc.) — every parameter gets its own update, computed simultaneously
  • α (alpha) — the learning rate: a single number controlling how big a step to take on each iteration. Too large and you overshoot the minimum; too small and progress crawls
  • ∂J(θ)/∂θⱼ — the partial derivative of the cost with respect to that one parameter — literally "how steep is the bowl, in this parameter's direction, right now"

Working through the calculus (applying the chain rule to J(θ)) gives a clean, vectorized form that's what actually gets implemented in code — updating every θⱼ at once via a single matrix operation:

∇J(θ) = (1/m) · Xᵀ(Xθ − y)

You repeat this update — compute the gradient, take a step, recompute — for many iterations, watching the cost J(θ) fall each time, until it flattens out (converges) near the minimum.

Batch vs. Stochastic vs. Mini-batch

VariantSamples per updateTrade-off
Batch GDAll m samplesStable but slow for large datasets
Stochastic GD1 random sampleFast, noisy convergence
Mini-batch GDSmall batch (e.g. 32)Practical standard in ML/DL
💡

Unscaled features cause wildly different gradient magnitudes across parameters, making one learning rate hard to tune for all of them — this is why feature scaling (standardization) matters before running gradient descent.

Quick Check — Gradient Descent
Q. Cost is increasing each iteration instead of decreasing. What's the likely cause and fix?
A. Learning rate α is too large, causing updates to overshoot the minimum and diverge. Fix: reduce α.
Q. Is a very small learning rate always "safe"?
A. No — too small means painfully slow convergence. It's a tuning trade-off, not a one-directional safety knob.
Interview Trap

Believing smaller α is always safer. Too small → extremely slow convergence; too large → divergence. Both are failure modes.

Topic 5

Newton's Method & IRLS

Gradient descent only asks "which way is downhill?" — it has no sense of how sharply the bowl is curving at the current point, so it can take many small, somewhat inefficient steps even on a simple convex surface. Newton's method adds that missing piece: it also uses curvature (the 2nd derivative) to estimate not just the direction downhill, but roughly how far to jump to land near the bottom in one move.

For a single parameter, this looks like θ := θ − J'(θ)/J''(θ). With multiple parameters, the 2nd derivative generalizes to a matrix of all pairwise second partial derivatives — the Hessian — and the update becomes:

θ := θ − H⁻¹ ∇J(θ)   where H = (1/m) XᵀX (for MSE — constant, independent of θ)

The remarkable part is what happens when you apply this specifically to MSE: because ŷ=Xθ is a direct linear function of θ, the Hessian works out to (1/m)XᵀX — a matrix built only from the data, with no θ in it at all. Since the curvature never changes no matter where you currently stand on the bowl, there's nothing left to "discover" iteratively — Newton's method computes the exact minimum in a single jump for linear regression.

Key Insight

Because the Hessian for MSE is constant (doesn't depend on θ), Newton's method reaches the exact minimum in a single step for linear regression — and algebraically, this single step reduces exactly to the Normal Equation: θ = (XᵀX)⁻¹Xᵀy.

Why isn't Newton's method commonly used at scale?

  • Computing H⁻¹ costs O(n³) where n = number of features — infeasible for large n (e.g. 10,000+ features)
  • Gradient descent variants scale far better to large n and m

IRLS (Iteratively Reweighted Least Squares)

Linear regression's one-step shortcut is a special case — most models don't get this lucky. Logistic regression, for instance, passes its linear combination Xθ through a sigmoid to produce probabilities, and that nonlinearity means the Hessian no longer stays constant — it now depends on the current θ. IRLS is the algorithm that adapts Newton's method to handle this.

Plain least squares treats every data point equally when fitting a line. Weighted least squares instead minimizes Σwᵢ(yᵢ−ŷᵢ)², where each point gets its own weight wᵢ — some points are allowed to matter more than others. IRLS uses this idea at every iteration: it computes a weight for each point based on the model's current prediction, solves a weighted least-squares problem to get an updated θ, then recomputes the weights using the new predictions, repeating until convergence (hence "iteratively reweighted").

For logistic regression specifically, the weight for point i is wᵢ = p̂ᵢ(1−p̂ᵢ), where p̂ᵢ is the model's current predicted probability for that point. This value peaks at ≈0.25 when p̂ᵢ≈0.5 — meaning the model is genuinely unsure about that point, so it gets more say in how θ updates next. It shrinks toward 0 as p̂ᵢ approaches 0 or 1 — meaning the model is already confident about that point, so it gets less influence on the next update, even if that confident prediction happens to be wrong. Applying IRLS to plain linear regression's MSE would collapse to exactly one iteration, since we already showed its Hessian never changes — there's simply nothing left to reweight after the first (and only) step.

Quick Check — Newton's Method
Q. Why does linear regression's Hessian not depend on θ, while logistic regression's does?
A. Linear regression's prediction ŷ=Xθ is a direct linear function of θ, so θ cancels out in the 2nd derivative. Logistic regression's prediction passes through a nonlinear sigmoid, so θ stays baked into the curvature term via p̂(1−p̂).
Q. Why would anyone use gradient descent instead of Newton's method if Newton's converges in fewer iterations?
A. Each Newton iteration costs O(n³) to invert the Hessian — for large n, gradient descent's cheaper O(n) per-iteration cost wins overall despite needing more iterations.
Interview Trap

Thinking Newton's method is "strictly better" because it converges in fewer iterations. Fewer iterations ≠ less total compute — each iteration is far more expensive at large n.

Topic 6

Normal Equation (Closed-Form Solution)

Gradient descent walks toward the minimum step by step; Newton's method jumps there using curvature. But there's an even more direct route available specifically for MSE: since J(θ) is convex, the point where it's minimized is exactly the point where its slope is zero in every direction. Instead of iterating at all, we can set the gradient equal to zero and solve the resulting equation algebraically for θ.

Starting from the gradient we already derived in Topic 4, ∇J(θ)=(1/m)Xᵀ(Xθ−y), setting it to zero and simplifying (multiplying through by m, then isolating θ) leads directly to:

θ = (XᵀX)⁻¹ Xᵀy

Notice that XᵀX here is exactly the same Hessian we computed in Topic 5 — which is precisely why the Normal Equation is often described as "the one Newton step, starting from θ=0" written out in closed form. There's no learning rate to tune, no convergence to wait for, and no risk of overshooting — you compute this formula once and you have the exact global minimum.

When to use vs. Gradient Descent

Normal EquationGradient Descent
ComplexityO(n³) — matrix inversionO(kn) per iteration
Feature scaling needed?NoYes
Large n (100k+ features)?InfeasibleWorks
XᵀX singular?FailsUnaffected

Why might XᵀX be non-invertible?

  • Multicollinearity — two or more features are exact linear combinations of each other
  • More features than samples (n > m) — underdetermined system
💡

With 50 samples and 100 features, XᵀX has shape (101,101) but rank limited to min(m, n+1)=51 — not full rank, so not invertible. Infinitely many θ's would fit the training data equally well, including some with zero training error — a strong overfitting signal.

Quick Check — Normal Equation
Q. Why is the Normal Equation not "always better" just because it's exact and non-iterative?
A. It's O(n³) — infeasible for large n. In practice, for large-scale ML with many features, gradient descent (or variants) is what's actually used; the Normal Equation is mostly a small-scale/teaching tool.
Q. What guarantees the Normal Equation gives the global minimum, not just a local one?
A. J(θ) for MSE is convex — a single bowl shape with one minimum — so setting the gradient to zero finds the global minimum directly.
Interview Trap

Assuming the Normal Equation always works. It fails outright when XᵀX is singular — you need Ridge regularization or the pseudo-inverse in that case.

Topic 7

Assumptions of Linear Regression

AssumptionWhat it means / how to check
LinearityCorrect functional form chosen — check residuals vs. fitted values for patterns (should be random scatter around 0)
IndependenceErrors aren't correlated across observations — common violation in time series; check via Durbin-Watson (≈2 = fine)
HomoscedasticityConstant variance of errors across all predictions — cone-shaped residual plot = violated
Normality of residualsNeeded for valid p-values/CIs (not for the coefficients themselves) — check via Q-Q plot or Jarque-Bera
No multicollinearityFeatures shouldn't be highly correlated — needed for reliable coefficient interpretation, not for predictions
Linearity ≠ "no curves allowed"

ŷ = θ₀+θ₁x+θ₂x² still satisfies the linearity assumption if x² genuinely belongs in the model. The assumption is about correctly specifying which terms belong — it isn't broken by using transformed features like x² or log(x).

Why heteroscedasticity matters

Standard error formulas assume constant residual variance. If violated, coefficients stay unbiased, but standard errors are typically underestimated — making p-values look more significant than they really are.

Q-Q Plot

Compares your residuals' quantiles against a theoretical normal distribution's quantiles. Points on the 45° diagonal = normal residuals. Curving away at the tails = heavier tails than normal; systematic deviation on one side = skewness.

Quick Check — Assumptions
Q. Residual plot shows a "cone" shape — narrow on the left, wide on the right. Which assumption is violated, and what specifically becomes untrustworthy?
A. Homoscedasticity is violated (heteroscedasticity present). Standard errors, p-values, and confidence intervals become unreliable — not the coefficients themselves, which stay unbiased.
Q. Does linear regression require the raw X features to be normally distributed?
A. No — it's the residuals (errors) that need approximate normality, not the raw features or the target y.
Interview Trap

A single RMSE number can't reveal heteroscedasticity — two datasets can share identical RMSE while one has a uniform error band and the other has a widening funnel shape. You need a residual plot, not just a summary statistic.

Topic 8

Multicollinearity & VIF

Occurs when features carry overlapping/redundant information (e.g. "size in sqft" and "size in sqm"). Predictions (ŷ) usually stay fine — but individual coefficients become unstable and hard to interpret.

VIFⱼ = 1 / (1 − Rⱼ²)

Regress feature xⱼ against all other features; Rⱼ² is how well they predict xⱼ. Rⱼ²=0.9 → VIF=10.

VIF valueInterpretation
1No multicollinearity concern
5Rⱼ²=0.8 — moderate overlap, worth investigating
>10Severe — problematic, consider dropping/combining features
💡

Perfect multicollinearity makes XᵀX singular (Subtopic 6) — one column is a linear combination of another, reducing rank. Near-perfect multicollinearity doesn't break invertibility outright but makes (XᵀX)⁻¹ numerically unstable.

Quick Check — Multicollinearity
Q. Feature A has VIF=50. You drop unrelated feature B instead. Does this fix A's problem?
A. No — VIF for A is driven by whichever feature(s) it's actually correlated with. Dropping an unrelated feature does nothing to that relationship.
Q. Does high multicollinearity mean the model's predictions are bad?
A. Not necessarily — ŷ can still be accurate. The damage is to coefficient interpretability and statistical significance testing, not predictive accuracy.
Interview Trap

Interviewers often ask "if VIF is high, is your model wrong?" expecting: no — predictions are fine, only coefficient interpretation/significance is compromised.

Topic 9

Regularization — Ridge, Lasso, Elastic Net

Everything so far minimizes MSE and nothing else — which means the model is free to push coefficients as large as it wants if that reduces training error, even when that's really just chasing noise or leaning too hard on a handful of correlated features (Topic 8). Regularization adds a second term to the cost function that directly penalizes large coefficients, forcing the model to trade a little training fit for a simpler, more stable, better-generalizing set of parameters.

Ridge (L2): J(θ) = MSE + λ Σθⱼ²  |  Lasso (L1): J(θ) = MSE + λ Σ|θⱼ|

In both cases, λ is a hyperparameter controlling how strongly to penalize — higher λ shrinks coefficients more aggressively. Crucially, the sum in the penalty starts at j=1, not j=0 — the intercept θ₀ is deliberately excluded, since penalizing it would just shift every prediction up or down rather than control model complexity.

Ridge happens to have its own closed-form solution, obtained the same way as the plain Normal Equation but with the penalty term folded in: θ = (XᵀX + λI)⁻¹Xᵀy. The +λI addition is what makes this so useful in practice — even if XᵀX itself is singular (from multicollinearity or having more features than samples), adding λI (a full-rank matrix) to it guarantees the whole expression is invertible, sidestepping the exact failure mode we saw in Topic 6.

Ridge (L2)Lasso (L1)Elastic Net
Coefficients → exactly 0?NoYesYes
Feature selection?NoYesYes
Handles multicollinearityYes (shares penalty)Poorly (picks one arbitrarily)Yes
Closed-form solution?YesNo (iterative)No (iterative)
Best whenMany small/medium correlated predictorsFew truly relevant predictors (sparse)Correlated + sparse mix
Why L1 zeroes coefficients but L2 doesn't

Geometrically, the L1 penalty region is a diamond with sharp corners on the axes; L2's is a smooth circle. The optimum is where cost contours first touch the penalty boundary — the diamond's corners make landing exactly on an axis (θⱼ=0) likely; the circle has no corners, so coefficients shrink but rarely hit exactly 0.

Quick Check — Regularization
Q. 200 features, you suspect only ~10 are truly relevant. Which regularization first?
A. Lasso — its ability to zero out coefficients performs automatic feature selection, matching the expected sparsity.
Q. Is a higher λ always "more regularization = better"?
A. No — λ is tuned via cross-validation. Too high underfits (excess bias); too low barely regularizes (overfitting risk remains).
Interview Trap

Assuming Lasso is strictly better because it does feature selection. With groups of correlated features, Lasso arbitrarily picks one and zeroes the rest — unstable. Elastic Net or Ridge handle correlated groups more gracefully.

Topic 10

Evaluation Metrics — RMSE, MAE, R², Adjusted R²

Once a model is fit — via gradient descent, Newton's method, the Normal Equation, or a regularized variant — the natural next question is: how good is it, really, and how do I explain that to someone who isn't going to look at J(θ) directly? That's what this set of metrics is for. RMSE and MAE describe the size of typical errors in real, interpretable units; R² and Adjusted R² describe how much of the pattern in y the model actually captures.

RMSE = √[(1/m)Σ(y−ŷ)²]  |  MAE = (1/m)Σ|y−ŷ|

Both bring the error back into the same units as y (unlike MSE, which is in squared units and therefore hard to interpret directly). RMSE, because it squares errors before averaging and then takes a square root, still penalizes large errors more heavily — so it's more sensitive to outliers, echoing the same MSE-vs-MAE trade-off from Topic 3. MAE treats every error linearly, making it the more robust choice when a few extreme errors shouldn't dominate your sense of "typical" performance.

R² answers a different question: not "how big are the errors," but "how much of the variation in y did the model actually explain, compared to the simplest possible baseline of just guessing the mean every time?"

R² = 1 − SSres/SStot
  • SSres — sum of squared residuals: the total squared error your actual model makes
  • SStot — total sum of squares: the total squared error you'd get if you ignored every feature and simply predicted the average ȳ for every single observation
  • R²=1 means residuals are zero (perfect fit); R²=0 means your model is doing no better than that mean-only baseline; R² can even go negative on test data, which happens when a model is bad enough (usually overfit) that it performs worse than the naive baseline would have

Plain R² has an awkward quirk, though: it can never decrease when you add more features, even completely random, meaningless ones — because the model always has at least as much flexibility to shave off a little more residual error. Adjusted R² corrects for this by explicitly penalizing the number of features relative to the sample size, so it only goes up when a new feature improves the fit by more than pure chance would predict:

R²adj = 1 − (1−R²)·(m−1)/(m−n−1)
💡

Plain R² never decreases when you add more features — even useless ones — because the model always has at least as much flexibility to reduce SSres. Adjusted R² penalizes this, only rising if a new feature improves fit more than chance would predict.

Quick Check — Metrics
Q. Model A: R²=0.85, 3 features. Model B: R²=0.86, 15 features. Same 100 samples. Which do you prefer, and what would you check?
A. Likely Model A — check Adjusted R². Adding 12 extra features for only +0.01 R² is a classic overfitting signal; Adjusted R² would likely favor the simpler model despite B's higher raw R².
Q. Why should you also check test error, not just training R²?
A. R²/training error alone can't distinguish a genuinely good fit from a model that has simply memorized training noise — only held-out performance reveals that.
Interview Trap

Treating R²=0.99 as automatically "great." It doesn't detect overfitting, omitted-variable bias, or wrong functional form, and doesn't confirm predictions are practically useful for the business need.

Topic 11

Bias-Variance Tradeoff

Every choice we've discussed so far — how many features to include, how much to regularize, whether to trust a high R² — ultimately comes back to one underlying tension. If you decompose a model's expected error on new, unseen data, it splits cleanly into three separate sources:

Expected Test Error = Bias² + Variance + Irreducible Error

Bias is error from the model being too simple to represent the true pattern — it makes strong, often wrong assumptions (like forcing a straight line through clearly curved data), and it makes essentially the same mistake no matter which training sample it happened to see. Variance is the opposite failure: the model is flexible enough to chase the specific quirks and noise of whatever training sample it was given, so its predictions swing wildly if you retrained it on a slightly different sample of the same underlying data. Irreducible error is simply the noise baked into the data itself — measurement error, genuine randomness — that no model, however good, can ever eliminate.

The "tradeoff" part is that these two error sources pull in opposite directions as you change model complexity: simplifying a model (fewer features, more regularization) tends to raise bias but lower variance; making it more flexible (more features, less regularization) tends to lower bias but raise variance. You can't drive both to zero simultaneously — the practical goal is finding the complexity level where their sum is smallest, not chasing either one in isolation.

High Bias (Underfitting)
  • Model too simple, consistently wrong the same way regardless of training sample
  • Low variance across resamples, but misses the true pattern
High Variance (Overfitting)
  • Model too sensitive to the specific training sample seen
  • Wildly different predictions across resamples — memorizes noise
LeverEffect on BiasEffect on Variance
Fewer features↑ (may miss curvature)↓ (more stable)
More features / polynomial terms↓↑ (sensitive to noise)
Regularization (λ↑)↑ (shrinks real signal too)↓ (less noise sensitivity)
More training dataUnchanged↓
💡

Reducing bias (more features, less regularization) typically increases variance and vice versa — you're not eliminating error, you're shifting where it comes from. The goal is minimizing total error (bias² + variance), not either term alone.

Quick Check — Bias-Variance
Q. Model X: low training error, high test error. Model Y: moderate training error, similarly moderate/slightly-better test error. Which suffers more from variance, and which do you deploy?
A. Model X suffers from variance (overfitting). Deploy Model Y — smaller train/test gap and comparable-or-better test error signal it generalizes better.
Q. Why is regularization described as "trading bias for variance"?
A. It deliberately shrinks coefficients away from their true OLS values (adding bias) in exchange for a larger reduction in variance — a net win when the unregularized model was overfitting.
Interview Trap

"Low training error = good model." Very low training error is often a symptom of high variance/overfitting — always compare against held-out test error.

Topic 12

statsmodels OLS Output — Full Breakdown

Everything covered so far — the fitted coefficients, R²/Adjusted R², the assumptions, multicollinearity — all shows up together the moment you actually fit a model in Python. Running statsmodels' OLS produces a single dense summary table that packs in overall model quality, per-coefficient significance, and residual diagnostics all at once. Learning to read every line of it is what turns "I ran a regression" into "I can actually defend what this model is telling me."

import statsmodels.api as sm X = sm.add_constant(X) # adds the 1s intercept column model = sm.OLS(y, X).fit() print(model.summary())

Quick Check — statsmodels Output
Q. Overall F-statistic is significant, but every individual coefficient's p-value is non-significant. What's the likely cause?
A. Multicollinearity — correlated features "share credit," making each individually look weak in isolation even though jointly they explain real variance.
Q. What would you re-check before trusting the p-values if Cond. No. is very high?
A. Check VIF for individual features and consider dropping/combining correlated ones or using Ridge — high condition number signals numerically unstable coefficient estimates.
Interview Trap

Reading "Covariance Type: nonrobust" as harmless boilerplate. It's a real assumption — if heteroscedasticity is present, every p-value and CI in the table is computed under a false premise unless you switch to robust standard errors.

Topic 13

Common Interview Traps — Concept Recap

  • Correlation ≠ causation — a significant coefficient shows association, not that x causes y (confounders can produce spurious relationships)
  • Outlier vs. leverage vs. influence — outlier = unusual y; leverage = unusual x; influence = removing the point changes coefficients substantially (needs both); measured via Cook's Distance
  • R² misuse — high R² doesn't detect overfitting, omitted variables, or wrong functional form
  • Heteroscedasticity — corrupts standard errors/p-values/CIs, not the coefficients themselves
  • "X must be normal" is false — it's the residuals that need approximate normality
  • Extrapolation danger — a model trained on [1000,3000] sqft shouldn't be trusted on a 10,000 sqft mansion, even with a high R² inside the training range
  • Multicollinearity hurts interpretation, not prediction — ŷ can remain accurate even with unstable, hard-to-interpret coefficients
  • Simpson's Paradox — a trend within each subgroup can reverse when subgroups are pooled together, due to systematic differences in average x and y across groups
The Meta-Trap

"R²=0.95, all p-values <0.001 — can we trust it for houses way outside our training range?" No — high in-range fit says nothing about validity outside the observed data range; the linear relationship and the assumptions behind those p-values were only verified where you had data.

Final

30 Linear Regression Interview Questions

Q1Easy
What is the objective linear regression is trying to minimize?
+

The Mean Squared Error between predictions and actual values: J(θ) = (1/2m)Σ(ŷ−y)². It finds the θ that minimizes this convex cost function.

Q2Easy
Why is a column of 1s added to the design matrix X?
+

To represent the intercept term θ₀ within the same Xθ matrix multiplication. Without it, the model is forced through the origin.

Q3Easy
What does the slope coefficient in a simple linear regression represent?
+

The marginal effect — the expected change in y for a one-unit increase in x, holding other variables constant.

Q4Easy
What does "linear" in linear regression actually refer to?
+

Linearity in the parameters θ, not in x. A model can include x², log(x), etc. as features and still be linear regression.

Q5Easy
Why is MSE preferred over MAE for gradient-based optimization?
+

MSE is differentiable everywhere (smooth), while MAE has a non-differentiable kink at zero, which breaks gradient computations at that point.

Q6Easy
What does R²=0 mean?
+

The model performs no better than simply predicting the mean of y for every observation.

Q7Easy
Can R² be negative? What does that imply?
+

Yes — it implies the model performs worse than just predicting the mean, which can happen on test data for a badly overfit or misspecified model.

Q8Easy
What units are RMSE and MAE expressed in, compared to MSE?
+

RMSE and MAE are in the same units as y. MSE is in squared units, which is why RMSE (its square root) is preferred for interpretability.

Q9Easy
Does linear regression require the raw feature X or target y to be normally distributed?
+

No — it's the residuals (errors) that need approximate normality, and even that mainly matters for valid inference (p-values, CIs), not for the coefficient estimates themselves.

Q10Easy
What's the difference between R² and Adjusted R²?
+

R² never decreases when features are added, even useless ones. Adjusted R² penalizes added features that don't meaningfully improve fit, making it suitable for comparing models with different feature counts.

Q11Medium
Derive the gradient of the MSE cost function with respect to θⱼ.
+

∂J(θ)/∂θⱼ = (1/m)Σ(ŷ⁽ⁱ⁾−y⁽ⁱ⁾)·xⱼ⁽ⁱ⁾, derived via the chain rule on J(θ)=(1/2m)Σ(ŷ−y)² — the ½ cancels with the power rule's 2. Vectorized: ∇J(θ)=(1/m)Xᵀ(Xθ−y).

Q12Medium
Derive the Normal Equation from the gradient of J(θ).
+

Set ∇J(θ)=0: Xᵀ(Xθ−y)=0 → XᵀXθ = Xᵀy → θ = (XᵀX)⁻¹Xᵀy, valid when XᵀX is invertible.

Q13Medium
Why does gradient descent require feature scaling but the Normal Equation doesn't?
+

Unscaled features cause gradient magnitudes to differ wildly across parameters, making a single learning rate hard to tune and slowing convergence. The Normal Equation solves algebraically in one step, so scale doesn't affect convergence — only numerical conditioning at extreme scales.

Q14Medium
What does VIF measure, and how is it computed?
+

VIFⱼ = 1/(1−Rⱼ²), where Rⱼ² comes from regressing feature xⱼ on all other features. It measures how much of xⱼ's variance is explained by the other features — high VIF (>5–10) flags multicollinearity.

Q15Medium
How does Ridge regression's closed-form solution fix the invertibility problem of the Normal Equation?
+

θ=(XᵀX+λI)⁻¹Xᵀy — adding λI (a full-rank matrix) to XᵀX guarantees the sum is invertible even when XᵀX alone is singular (from multicollinearity or n>m).

Q16Medium
Why can Lasso zero out coefficients but Ridge cannot?
+

Geometrically, L1's penalty region is a diamond with sharp corners on the axes; L2's is a smooth circle. The optimum touches the penalty boundary where cost contours first meet it — the diamond's corners make landing exactly on an axis (θⱼ=0) likely; the circle has no corners, so L2 shrinks but rarely hits exactly zero.

Q17Medium
What's the practical difference between detecting outliers via IQR vs. Z-score?
+

IQR is robust for skewed data since it doesn't assume normality. Z-score assumes roughly normal data and is itself sensitive to the outliers it's trying to detect, since they inflate the mean/std used in its computation.

Q18Medium
A feature has a very low p-value but a tiny coefficient. Is its effect practically important?
+

Not necessarily — check the feature's scale (a tiny coefficient on a large-range feature can still mean a big swing in ŷ) and consider standardized coefficients. Also, with large samples, even trivial effects can become statistically significant since standard error shrinks as 1/√m.

Q19Medium
What's the difference between an outlier, a leverage point, and an influential point?
+

Outlier = unusual y value (large residual). Leverage = unusual x value (far from mean of other x's). Influence = removing the point substantially changes the coefficients — requires both high leverage and being an outlier; measured via Cook's Distance.

Q20Medium
Why does a low Durbin-Watson value matter, and when is it most likely to occur?
+

A value near 0 signals strong positive autocorrelation in residuals — violating the independence assumption. Most common in time-series data, where today's error correlates with yesterday's.

Q21Hard
Prove that for MSE, the Hessian used in Newton's method is constant and equals (1/m)XᵀX.
+

Gradient: ∇J(θ)=(1/m)Xᵀ(Xθ−y). Taking the derivative again w.r.t. θ: ∂²J/∂θ² = (1/m)XᵀX — since Xθ is linear in θ, this term has no θ dependence left, so the Hessian is constant. This is why Newton's method converges to the exact minimum in one step for linear regression, and why that single step algebraically equals the Normal Equation.

Q22Hard
Why does logistic regression's Hessian depend on θ, requiring IRLS, while linear regression's does not?
+

Logistic regression's prediction p̂=σ(Xθ) passes through a nonlinear sigmoid, so θ doesn't cancel out of the second derivative — it appears via the curvature term p̂(1−p̂), which depends on the current θ. Linear regression's ŷ=Xθ is directly linear in θ, so θ cancels entirely, leaving a constant Hessian.

Q23Hard
Why might overall F-statistic be significant while every individual coefficient's p-value is not?
+

Multicollinearity — correlated features "share credit" for explaining y, so each one's isolated marginal contribution (and thus its individual p-value) looks weak, even though jointly the group of features is genuinely predictive (reflected in the significant F-statistic).

Q24Hard
Explain Simpson's Paradox in the context of a regression, and how you'd detect/correct for it.
+

A trend that holds within each subgroup can reverse when subgroups are pooled, because the groups differ systematically in average x and y levels. Detect it by checking whether including a categorical group variable (or running per-group regressions) changes the sign of a coefficient. Correct by including the group variable in the model rather than pooling blindly.

Q25Hard
Why is Adjusted R² not sufficient on its own to compare two models of very different complexity, and what would you check alongside it?
+

Adjusted R² is still computed on training data — it can't fully protect against overfitting the way held-out test performance can, and it doesn't reveal assumption violations (heteroscedasticity, non-normal residuals) that affect inference validity. Cross-validated test error and residual diagnostics should be checked alongside it.

Q26Hard
Why does heteroscedasticity leave coefficients unbiased but corrupt inference, and what's the practical fix?
+

OLS coefficient estimation doesn't require constant residual variance to be unbiased — only the standard error formula does. When violated, standard errors (and thus p-values/CIs) are miscalculated, typically underestimated, inflating apparent significance. Fix: use robust standard errors (e.g. statsmodels' cov_type='HC3') rather than re-estimating the coefficients themselves.

Q27Hard
Walk through why Elastic Net handles correlated feature groups better than Lasso alone.
+

Lasso's L1 penalty, applied to a group of highly correlated features, tends to arbitrarily select one feature and zero out the rest — the choice is unstable and can flip with small data changes. Elastic Net adds an L2 term alongside L1, which encourages correlated features to share the coefficient weight (Ridge's behavior) rather than one dominating arbitrarily, while still retaining L1's ability to zero out truly irrelevant features.

Q28Hard
A model has R²=0.95 and highly significant p-values on training data spanning x∈[1000,3000]. Your manager wants to predict for x=10,000. What's your answer and why?
+

Advise caution — high R² and significant p-values only validate the model within the observed range of training data. The linear relationship, and the assumptions underlying those p-values/CIs, were never verified outside [1000,3000]. The prediction may be directionally reasonable but its uncertainty is unquantifiable without new data covering that range — this is the extrapolation trap.

Q29Hard
Why is the condition number in statsmodels' output related to, but different from, VIF?
+

Both detect multicollinearity, but VIF is computed per-feature by regressing it on the others, while the condition number is a single dataset-wide statistic — the ratio of the largest to smallest eigenvalue of XᵀX. A high condition number signals general numerical instability in the whole coefficient estimation, without pinpointing which specific feature(s) are responsible the way VIF does.

Q30Hard
You add 12 low-signal features to a 3-feature model. Training R² rises slightly (0.85→0.86), but you suspect overfitting. Walk through every check you'd run before deciding to keep them.
+

1) Compute Adjusted R² for both models — a marginal R² gain from 12 extra parameters likely lowers it. 2) Compare cross-validated/test error, not just training R² — overfitting shows up as a widening train-test gap. 3) Check VIF among the new features and against the original 3, since more features raise multicollinearity risk. 4) Check AIC/BIC as a complexity-penalized comparison. 5) Consider Lasso/Elastic Net to let regularization decide which of the 12 genuinely earn a non-zero coefficient, rather than keeping all of them by default.
