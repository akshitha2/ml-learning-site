Forecasting Metrics

Forecasting metrics answer three questions: how large is my error (MAE, RMSE), how large relative to the actual (MAPE, SMAPE, WAPE), and am I systematically biased (MFE). Picking the wrong one for your data type will produce misleading production numbers.

MAE — Mean Absolute Error
"On average, how far off am I?" — the interpretable baseline.
MAE = (1/n) × Σ |yᵢ − ŷᵢ|
// yᵢ = actual, ŷᵢ = predicted, n = number of periods

Every error contributes equally regardless of size. A 100-unit error and a 1-unit error have proportional influence on MAE — there's no penalty multiplier for being very wrong.

Numerical Example

Actual:    [100, 200, 300]
Predicted: [110, 180, 320]
Errors:    [ 10,  20,  20]
MAE = (10 + 20 + 20) / 3 = 16.7 units
→ On average, predictions are off by 16.7 units

Advantages

  • Same units as the target — directly interpretable
  • Robust to outliers — extreme errors don't dominate
  • Optimal for median prediction (L1 loss)

Disadvantages

  • Not differentiable at zero — gradient-based optimization harder
  • Doesn't penalize large errors — misses costly spikes
  • Scale-dependent — can't compare across different series
Use WhenAvoid When
Outlier errors are acceptable; robust baseline neededLarge errors are disproportionately costly (e.g., revenue spikes)
Actuals vary in a small, predictable rangeComparing models across different scales/series
RMSE — Root Mean Squared Error
"How bad are my worst predictions?" — outlier-sensitive error.
RMSE = √[ (1/n) × Σ (yᵢ − ŷᵢ)² ]
// Squaring penalizes large errors disproportionately

Squaring means a 10-unit error contributes 100 to the sum, while a 1-unit error contributes only 1. RMSE amplifies large errors — making it the right metric when big mistakes are expensive.

Numerical Example

Actual:    [100, 200, 300]
Predicted: [110, 180, 320]
Errors²:   [100, 400, 400]
MSE  = 900 / 3 = 300
RMSE = √300 = 17.3 units
→ RMSE (17.3) > MAE (16.7) — gap reveals mild outlier errors
→ Large gap between RMSE and MAE = large outlier errors present
Interview Insight: The gap between RMSE and MAE is diagnostic. If RMSE ≫ MAE, large individual errors dominate. In IPL forecasting, a rain delay causes a single extreme hour error — spikes RMSE but barely moves MAE. This gap tells you the error distribution is heavy-tailed.

Advantages

  • Penalizes large errors — right for high-stakes predictions
  • Differentiable everywhere — ideal for gradient optimization (MSE loss)
  • Same units as target after square root

Disadvantages

  • Sensitive to outliers — one bad prediction skews the metric
  • Scale-dependent — can't compare across series
  • Harder to interpret directly vs MAE
MAPE — Mean Absolute Percentage Error
"How wrong am I as a fraction of the actual?" — the go-to for scale-free comparison.
MAPE = (100/n) × Σ |(yᵢ − ŷᵢ) / yᵢ|
// Expressed as a percentage. Denominator = actual value yᵢ

Scale-independent — this is why you can report "1.3% MAPE" as a meaningful number regardless of whether the revenue is ₹1L or ₹100Cr. Allows comparison across completely different time series.

Numerical Example

Actual:    [100, 200]
Predicted: [110, 180]
% Errors:  [10/100, 20/200] = [10%, 10%]
MAPE = 10%
→ On average, predictions are 10% away from actuals
⚠️ Critical Failure Mode: When yᵢ → 0, MAPE → ∞. Early IPL hours (9-10 AM) have near-zero revenue. MAPE explodes. This is why reporting "MAPE across hours 10-18" is statistically correct, not cherry-picking. The denominator becomes unstable near zero.
Asymmetry Trap: MAPE is asymmetric. An over-prediction of 50% gives MAPE = 50%. But an under-prediction of 50% of the same actual also gives 50%. However, actual=100, predict=200 (over by 100%) vs actual=200, predict=100 (under by 50%) get different MAPE values even though the absolute error is the same.

Advantages

  • Scale-independent — compare across different series
  • Widely understood by business stakeholders
  • Intuitive: "we were X% off on average"

Disadvantages

  • Undefined/infinite when actual = 0
  • Asymmetric — penalizes over-forecast more than under-forecast
  • Biased toward under-forecasting models
SMAPE — Symmetric MAPE
"MAPE but fairer when actuals are near zero."
SMAPE = (100/n) × Σ |yᵢ − ŷᵢ| / ((|yᵢ| + |ŷᵢ|) / 2)
// Denominator = average of actual + predicted (not just actual)

The key fix: denominator uses the average of actual and predicted instead of just actual. This bounds SMAPE between 0% and 200%, preventing the infinity problem when actuals are near zero.

MAPE vs SMAPE — Near-Zero Actual

Actual = 10, Predicted = 20
MAPE  = |10−20| / 10          = 100%   ← explodes
SMAPE = |10−20| / ((10+20)/2) = 66.7%  ← bounded

Actual = 100, Predicted = 200
MAPE  = 100%
SMAPE = 66.7%  ← same — symmetric treatment
Caveat: SMAPE has its own asymmetry when actual = 0 exactly. If actual=0 and predicted=10: SMAPE = 200%. If actual=10 and predicted=0: SMAPE = 200%. Symmetric in this edge case, but the value (200%) is hard to interpret. Better than MAPE but not perfect.
WAPE — Weighted Absolute Percentage Error
"The right metric when your segments have very different scales."
WAPE = Σ|yᵢ − ŷᵢ| / Σyᵢ
// Total absolute error divided by total actual — volume-weighted

WAPE weights each error by the actual volume of that period or segment. Large-volume segments dominate the metric — which is exactly what business stakeholders want. A 30% error in a ₹1K segment shouldn't penalize your system as much as a 5% error in a ₹10Cr segment.

Why WAPE Beats Average MAPE for Hierarchical Forecasting

Segment A (large): Actual=1000, Predicted=1100 → Error=100
Segment B (small): Actual=10,   Predicted=20   → Error=10

Average MAPE = (10% + 100%) / 2 = 55% ← misleading
WAPE         = (100+10)/(1000+10) = 10.9% ← correct picture

→ Segment B's 100% MAPE is irrelevant — it's ₹10 of error.
→ WAPE surfaces what actually matters to the business.
AOP Engine Application: 370+ models across 4 segments with different scales. Average MAPE equally weights low-volume cohort (high MAPE, low revenue impact) with high-volume cohort (low MAPE, high revenue impact). WAPE fixes this. "How is your system doing overall?" → WAPE is the answer.
MFE — Mean Forecast Error (Bias)
"Am I systematically over- or under-predicting?" — the directional check.
MFE = (1/n) × Σ (yᵢ − ŷᵢ)
// No absolute value — sign carries the diagnostic signal

Interpretation

MFE > 0 → Model systematically UNDER-predicts (actuals > predictions)
MFE < 0 → Model systematically OVER-predicts (actuals < predictions)
MFE = 0 → Unbiased (but could still have high MAE/RMSE!)

Example: MFE = -50K in IPL revenue forecasting
→ Model consistently over-predicts by ₹50K/hour
→ Causes over-allocation of marketing budget every match
→ A/B campaigns are triggered unnecessarily
Key Insight: MFE = 0 doesn't mean a good model. You could have errors of [+100, -100, +100, -100] → MFE = 0 but MAE = 100. Always report MFE alongside MAE/MAPE, not instead of them.

Business Translation — High-Value User Prediction (Dream11)

TP → Correctly identified depositor → Show targeted offer ✓
TN → Correctly identified non-depositor → No wasted spend ✓
FP → Non-depositor flagged as depositor → Wasted campaign budget
FN → Depositor missed → Lost revenue opportunity
Precision, Recall, F1
Precision = TP / (TP + FP) // Quality of positive predictions
Recall = TP / (TP + FN) // Coverage of actual positives
F1 = 2 × (Precision × Recall) / (Precision + Recall)

Worked Example — 1000 users, 90 actual depositors

TP=80, FP=20, FN=10, TN=890

Precision = 80 / (80+20)   = 80.0%  ← 1 in 5 flagged users is wrong
Recall    = 80 / (80+10)   = 88.9%  ← missed 10 real depositors
F1        = 2×(0.8×0.889)/(0.8+0.889) = 84.2%
Accuracy  = (80+890)/1000  = 97.0%  ← looks great, hides the full picture

Precision — Minimize FP

  • Use when false alarms are costly
  • Spam filter — don't block real emails
  • Targeted ad spend — don't waste budget

Recall — Minimize FN

  • Use when missing positives is costly
  • Cancer screening — don't miss a case
  • Churn prediction — don't miss at-risk user
The Accuracy Paradox — Why Accuracy Fails
The #1 interview trap on imbalanced data.

200M Users, 1% Churn Rate

Total users:     200,000,000
Churners:          2,000,000  (1%)
Non-churners:    198,000,000  (99%)

Model: "Predict nobody churns"
TP=0, FP=0, FN=2M, TN=198M
Accuracy = 198M / 200M = 99%  ← looks excellent
Recall = 0/2M = 0%  ← catches zero churners

→ Useless model. Perfect accuracy. Never use accuracy on imbalanced data.
F-beta Score — Asymmetric Cost
Fβ = (1 + β²) × (Precision × Recall) / (β² × Precision + Recall)
// β=1: equal weight | β=2: recall 2x | β=0.5: precision 2x

When to Use Which β

β = 1.0 → F1 — equal cost for FP and FN
β = 2.0 → F2 — recall weighted higher (missing positives more costly)
          Use: churn prediction, fraud detection, disease screening
β = 0.5 → F0.5 — precision weighted higher (false alarms more costly)
          Use: spam filters, content moderation, targeted campaigns

ROC-AUC vs PR-AUC

ROC Curve
Plots TPR vs FPR across all thresholds. AUC = probability of ranking a random positive above a random negative.
TPR (Y-axis) = TP / (TP + FN) ← Recall FPR (X-axis) = FP / (FP + TN) ← 1 - Specificity AUC = P(score(positive) > score(negative))
Why ROC-AUC Misleads on Imbalanced Data: FPR = FP / (FP + TN). When TN is massive (198M non-churners), even thousands of false positives produce a tiny FPR. The ROC curve looks excellent while your model is spamming false alarms. The large TN masks the problem.
PR Curve (Precision-Recall)
Plots Precision vs Recall. TN never appears — not distorted by majority class.
Precision (Y-axis) = TP / (TP + FP) Recall (X-axis) = TP / (TP + FN) Average Precision (AP) = Σ (Rₙ − Rₙ₋₁) × Pₙ ← use sklearn's average_precision_score
Baseline Difference: ROC-AUC random baseline = always 0.5. PR-AUC random baseline = class prevalence (e.g. 0.02 for 2% positive class). A PR-AUC of 0.72 on 2% positive class is excellent. Always state baseline when reporting PR-AUC.

Regression Metrics

R² — Coefficient of Determination
"How much variance does my model explain vs just predicting the mean?"
R² = 1 − (SS_res / SS_tot)
SS_res = Σ(yᵢ − ŷᵢ)² ← residual sum of squares SS_tot = Σ(yᵢ − ȳ)² ← total sum of squares (baseline: predict mean)

Interpretation

R² = 0.85 → Model explains 85% of variance in the target
R² = 0.0  → Model is no better than predicting the mean
R² < 0    → Model is WORSE than predicting the mean (possible!)
⚠️ R² always increases as you add features — can overfit silently
→ Use Adjusted R² when comparing models with different feature counts
RMSLE — Root Mean Squared Log Error
"RMSE for targets that span orders of magnitude." — revenue, user counts, prices.
RMSLE = √[ (1/n) × Σ (log(yᵢ+1) − log(ŷᵢ+1))² ]
// +1 avoids log(0). Works in log space → relative errors matter equally

When RMSLE Beats RMSE

Revenue can be ₹1K (small match) or ₹100Cr (final).
RMSE: Error of ₹10K on ₹1K prediction = same as ₹10K on ₹100Cr → wrong
RMSLE: Error of 10× on ₹1K = same penalty as 10× on ₹100Cr → correct

→ Use RMSLE when % error matters equally across scale
→ Use RMSE when absolute error matters (predict exact units)

Ranking & Retrieval Metrics

Used in recommendation systems, search, and RAG pipelines. The core question: not just "did you find the relevant items?" but "did you rank them correctly?"

Precision@K
"Of the top K results I returned, how many are relevant?"
Precision@K = (# relevant items in top K) / K

Example — Top 5 retrieved events in Eventify RAG

Retrieved: [✓, ✓, ✗, ✓, ✗]   (✓ = relevant)
P@5 = 3/5 = 60%
P@3 = 2/3 = 66.7%
→ Doesn't penalize for rank position — item 1 same as item 5
MRR — Mean Reciprocal Rank
"Where does the FIRST relevant result appear?" — for single-answer queries.
MRR = (1/|Q|) × Σ 1/rank_i
// rank_i = position of first relevant result for query i

Example

Query 1: First relevant result at rank 1 → 1/1 = 1.0
Query 2: First relevant result at rank 3 → 1/3 = 0.33
Query 3: First relevant result at rank 2 → 1/2 = 0.5
MRR = (1.0 + 0.33 + 0.5) / 3 = 0.61
→ Use for QA systems, Text-to-SQL (first valid query matters most)
MAP — Mean Average Precision
"Average P@K at every relevant document position, averaged across queries."
AP = (1/R) × Σₖ P@k × rel(k)
MAP = mean(AP) across all queries
// R = total relevant items, rel(k) = 1 if item at rank k is relevant

Example — 3 relevant items total

Retrieved: [✓, ✗, ✓, ✗, ✓]
P@1=1/1=1.0, P@3=2/3=0.67, P@5=3/5=0.60
AP = (1.0 + 0.67 + 0.60) / 3 = 0.756
→ MAP rewards finding relevant items early, not just finding them
NDCG — Normalized Discounted Cumulative Gain
"The gold standard for graded relevance — position and relevance score both matter."
DCG@K = Σᵢ (2^relᵢ − 1) / log₂(i+1)
NDCG@K = DCG@K / IDCG@K
// IDCG = DCG of perfect ranking. NDCG ∈ [0,1]

Example — Relevance scores [3, 2, 3, 0, 1] at ranks 1-5

DCG@5 = (2³-1)/log₂(2) + (2²-1)/log₂(3) + (2³-1)/log₂(4) + 0 + (2¹-1)/log₂(6)
      = 7/1 + 3/1.58 + 7/2 + 0 + 1/2.58
      = 7 + 1.90 + 3.5 + 0 + 0.39 = 12.79

Ideal order: [3,3,2,1,0] → IDCG = 13.85
NDCG@5 = 12.79/13.85 = 0.92
→ Use when results have graded relevance (not just relevant/irrelevant)
→ Used in Eventify: top-50 Qdrant retrieval quality

Advantages

  • Handles graded relevance (not just binary)
  • Position-aware — top result matters more
  • Normalized — compare across queries

Disadvantages

  • Requires relevance labels — expensive to obtain
  • Logarithmic discount may not match user behavior
  • Harder to explain to non-technical stakeholders
