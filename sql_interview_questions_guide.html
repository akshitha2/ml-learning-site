Round 1

Warm-up

W1
Get all employees earning more than the average salary in their department.
employees(employee_id, name, salary, department_id)
› SQL Solution
SELECT e.employee_id, e.name, e.salary, e.department_id FROM employees e JOIN ( SELECT department_id, AVG(salary) AS avg_salary FROM employees GROUP BY department_id ) d ON e.department_id = d.department_id WHERE e.salary > d.avg_salary; -- Window function alternative (no self-join needed): SELECT employee_id, name, salary, department_id FROM ( SELECT *, AVG(salary) OVER (PARTITION BY department_id) AS avg_salary FROM employees ) t WHERE salary > avg_salary;
› Python Solution
df["avg_salary"] = df.groupby("department_id")["salary"].transform("mean") result = df[df["salary"] > df["avg_salary"]]
W2
Find duplicate emails in a customer table.
customers(customer_id, email, name)
› SQL Solution
SELECT email, COUNT(*) AS cnt FROM customers GROUP BY email HAVING COUNT(*) > 1;
› Python Solution
dupe_emails = ( df.groupby("email") .size() .reset_index(name="cnt") .query("cnt > 1") )
Pitfall: HAVING filters on the aggregate; WHERE cannot reference COUNT(*).
W3
List customers who have never placed an order.
customers(customer_id, name) · orders(order_id, customer_id, order_date)
› SQL Solution
SELECT c.customer_id, c.name FROM customers c LEFT JOIN orders o ON c.customer_id = o.customer_id WHERE o.order_id IS NULL; -- Equivalent: SELECT customer_id, name FROM customers WHERE customer_id NOT IN (SELECT customer_id FROM orders WHERE customer_id IS NOT NULL);
› Python Solution
never_ordered = customers[~customers["customer_id"].isin(orders["customer_id"])]
Pitfall: NOT IN silently returns nothing if the subquery contains a NULL — filter NULLs out or use NOT EXISTS.
W4
Count orders per month for the last year, including months with zero orders.
orders(order_id, customer_id, order_date)
› SQL Solution
WITH months AS ( SELECT generate_series( date_trunc('month', CURRENT_DATE - INTERVAL '11 months'), date_trunc('month', CURRENT_DATE), INTERVAL '1 month' ) AS month_start ) SELECT m.month_start, COUNT(o.order_id) AS order_count FROM months m LEFT JOIN orders o ON date_trunc('month', o.order_date) = m.month_start GROUP BY m.month_start ORDER BY m.month_start;
› Python Solution
orders["month"] = orders["order_date"].dt.to_period("M").dt.to_timestamp() counts = orders.groupby("month").size() all_months = pd.date_range(end=pd.Timestamp.today().normalize().replace(day=1), periods=12, freq="MS") result = counts.reindex(all_months, fill_value=0)
Tests whether the candidate knows to generate a calendar/spine table — plain GROUP BY silently drops zero-count months.
W5
Difference between WHERE and HAVING — give an example where using the wrong one breaks the query.
› SQL Solution
-- WRONG: WHERE cannot reference an aggregate SELECT department_id, COUNT(*) FROM employees WHERE COUNT(*) > 5 -- ERROR GROUP BY department_id; -- CORRECT: SELECT department_id, COUNT(*) FROM employees GROUP BY department_id HAVING COUNT(*) > 5;
› Python Solution
# WHERE-equivalent: filter rows first filtered = df[df["salary"] > 50000] # HAVING-equivalent: filter after aggregation grouped = df.groupby("department_id").size().reset_index(name="cnt") grouped = grouped[grouped["cnt"] > 5]
Round 2

Core

C1
Find the 2nd (or Nth) highest salary per department.
employees(employee_id, name, salary, department_id)
› SQL Solution
SELECT department_id, employee_id, name, salary FROM ( SELECT *, DENSE_RANK() OVER (PARTITION BY department_id ORDER BY salary DESC) AS rnk FROM employees ) t WHERE rnk = 2;
› Python Solution
df["rnk"] = df.groupby("department_id")["salary"].rank(method="dense", ascending=False) result = df[df["rnk"] == 2]
Pitfall: use DENSE_RANK, not ROW_NUMBER, if tied salaries should count as one rank.
C2
Find users active on 3+ consecutive days.
logins(user_id, login_date)
› SQL Solution
WITH grp AS ( SELECT user_id, login_date, login_date - (ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_date))::int AS grp_key FROM (SELECT DISTINCT user_id, login_date FROM logins) d ) SELECT user_id, MIN(login_date) AS streak_start, COUNT(*) AS streak_len FROM grp GROUP BY user_id, grp_key HAVING COUNT(*) >= 3;
› Python Solution
d = logins.drop_duplicates(["user_id","login_date"]).sort_values(["user_id","login_date"]) d["rn"] = d.groupby("user_id").cumcount() + 1 d["grp_key"] = d["login_date"] - pd.to_timedelta(d["rn"], unit="D") streaks = ( d.groupby(["user_id","grp_key"]) .agg(streak_start=("login_date","min"), streak_len=("login_date","count")) .reset_index() ) result = streaks[streaks["streak_len"] >= 3]
Same "date minus row_number" pattern as Gaps & Islands G1 — the single most-tested SQL pattern in DS interviews.
C3
Calculate a 7-day rolling average of daily revenue.
daily_revenue(date, revenue)
› SQL Solution
SELECT date, revenue, AVG(revenue) OVER ( ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS rolling_7d_avg FROM daily_revenue ORDER BY date;
› Python Solution
df = df.sort_values("date") df["rolling_7d_avg"] = df["revenue"].rolling(window=7, min_periods=1).mean()
Pitfall: ROWS counts physical rows (assumes no missing dates); RANGE counts by date value — ask which the candidate would pick if dates have gaps.
C4
Find each employee's manager's manager (2 levels up).
employees(employee_id, name, manager_id)
› SQL Solution
SELECT e.employee_id, e.name AS employee, m1.name AS manager, m2.name AS skip_level_manager FROM employees e LEFT JOIN employees m1 ON e.manager_id = m1.employee_id LEFT JOIN employees m2 ON m1.manager_id = m2.employee_id;
› Python Solution
m1 = df.merge(df, left_on="manager_id", right_on="employee_id", suffixes=("", "_m1"), how="left") result = m1.merge(df, left_on="manager_id_m1", right_on="employee_id", suffixes=("", "_m2"), how="left") result = result[["employee_id","name","name_m1","name_m2"]]
For N levels instead of a fixed 2, this needs a recursive CTE (see Self-Join S3).
C5
Build a funnel: % of users who go from signup → activation → purchase.
users(user_id, signup_date) · activations(user_id, activation_date) · purchases(user_id, purchase_date)
› SQL Solution
WITH funnel AS ( SELECT u.user_id, 1 AS signed_up, CASE WHEN a.user_id IS NOT NULL THEN 1 ELSE 0 END AS activated, CASE WHEN p.user_id IS NOT NULL THEN 1 ELSE 0 END AS purchased FROM users u LEFT JOIN activations a ON u.user_id = a.user_id LEFT JOIN purchases p ON u.user_id = p.user_id ) SELECT SUM(signed_up) AS signups, SUM(activated) AS activations, SUM(purchased) AS purchases, ROUND(100.0 * SUM(activated) / SUM(signed_up), 1) AS pct_activated, ROUND(100.0 * SUM(purchased) / SUM(activated), 1) AS pct_purchased_of_activated FROM funnel;
› Python Solution
funnel = users[["user_id"]].copy() funnel["activated"] = funnel["user_id"].isin(activations["user_id"]).astype(int) funnel["purchased"] = funnel["user_id"].isin(purchases["user_id"]).astype(int) signups = len(funnel) activated = funnel["activated"].sum() purchased = funnel["purchased"].sum() pct_activated = round(100 * activated / signups, 1) pct_purchased_of_activated = round(100 * purchased / activated, 1)
Ask: should purchase require activation first, or just "ever done"? Ambiguity is intentional — good candidates ask.
C6
Month-over-month % change in revenue.
monthly_revenue(month, revenue)
› SQL Solution
SELECT month, revenue, LAG(revenue) OVER (ORDER BY month) AS prev_month_revenue, ROUND( 100.0 * (revenue - LAG(revenue) OVER (ORDER BY month)) / NULLIF(LAG(revenue) OVER (ORDER BY month), 0), 1 ) AS pct_change FROM monthly_revenue ORDER BY month;
› Python Solution
df = df.sort_values("month") df["prev_month_revenue"] = df["revenue"].shift(1) df["pct_change"] = (df["revenue"] / df["prev_month_revenue"] - 1) * 100
Pitfall: first row is NULL after LAG — candidate should state how they'd handle it.
C7
Find each customer's first purchase date, then classify into cohorts by signup month.
customers(customer_id, signup_date) · orders(order_id, customer_id, order_date)
› SQL Solution
WITH first_purchase AS ( SELECT customer_id, MIN(order_date) AS first_order_date FROM orders GROUP BY customer_id ) SELECT c.customer_id, date_trunc('month', c.signup_date) AS cohort_month, fp.first_order_date FROM customers c LEFT JOIN first_purchase fp ON c.customer_id = fp.customer_id;
› Python Solution
first_purchase = orders.groupby("customer_id")["order_date"].min().reset_index(name="first_order_date") result = customers.merge(first_purchase, on="customer_id", how="left") result["cohort_month"] = result["signup_date"].dt.to_period("M").dt.to_timestamp()
C8
Find the running total per account, resetting when balance goes negative.
transactions(account_id, txn_date, amount)
› SQL Solution
-- Plain window functions can't reference the running value mid-calculation, -- so this needs a recursive CTE. WITH RECURSIVE ordered AS ( SELECT *, ROW_NUMBER() OVER (PARTITION BY account_id ORDER BY txn_date) AS rn FROM transactions ), running AS ( SELECT account_id, txn_date, amount, rn, amount AS running_total FROM ordered WHERE rn = 1 UNION ALL SELECT o.account_id, o.txn_date, o.amount, o.rn, CASE WHEN r.running_total + o.amount < 0 THEN 0 ELSE r.running_total + o.amount END FROM ordered o JOIN running r ON o.account_id = r.account_id AND o.rn = r.rn + 1 ) SELECT account_id, txn_date, amount, running_total FROM running ORDER BY account_id, txn_date;
› Python Solution
def running_with_reset(group): total = 0 out = [] for amt in group: total += amt if total < 0: total = 0 out.append(total) return out df = df.sort_values(["account_id","txn_date"]) df["running_total"] = df.groupby("account_id")["amount"].transform(running_with_reset)
Senior-level trap: each step depends on the previous *computed* value, not just raw rows — needs recursion or a procedural loop.
Round 3

Self-Joins

S1
Find all pairs of employees who have the same manager.
employees(employee_id, name, manager_id)
› SQL Solution
SELECT a.employee_id AS emp1, b.employee_id AS emp2, a.manager_id FROM employees a JOIN employees b ON a.manager_id = b.manager_id AND a.employee_id < b.employee_id; -- avoids mirrored pairs + self-pairs
› Python Solution
m = df.merge(df, on="manager_id", suffixes=("1","2")) pairs = m[m["employee_id1"] < m["employee_id2"]][["employee_id1","employee_id2","manager_id"]]
Pitfall: forgetting the "<" condition gives duplicate mirrored pairs and matches employees with themselves.
S2
Find employees who earn more than their manager.
employees(employee_id, name, salary, manager_id)
› SQL Solution
SELECT e.employee_id, e.name AS employee, e.salary, m.name AS manager, m.salary AS manager_salary FROM employees e JOIN employees m ON e.manager_id = m.employee_id WHERE e.salary > m.salary;
› Python Solution
m = df.merge(df, left_on="manager_id", right_on="employee_id", suffixes=("","_mgr")) result = m[m["salary"] > m["salary_mgr"]]
S3
Find the org chart depth for each employee (levels from the CEO).
employees(employee_id, name, manager_id) -- CEO has manager_id IS NULL
› SQL Solution
WITH RECURSIVE org AS ( SELECT employee_id, name, manager_id, 0 AS depth FROM employees WHERE manager_id IS NULL UNION ALL SELECT e.employee_id, e.name, e.manager_id, o.depth + 1 FROM employees e JOIN org o ON e.manager_id = o.employee_id ) SELECT * FROM org ORDER BY depth, employee_id;
› Python Solution
depth = {} def get_depth(emp_id): if emp_id in depth: return depth[emp_id] mgr = df.loc[df.employee_id==emp_id, "manager_id"].iloc[0] if pd.isna(mgr): depth[emp_id] = 0 else: depth[emp_id] = get_depth(mgr) + 1 return depth[emp_id] df["depth"] = df["employee_id"].apply(get_depth)
A plain self-join breaks down here (fixed joins can't handle arbitrary depth) — tests if the candidate reaches for a recursive CTE.
S4
Find products that were bought together in the same order (market basket pairs).
order_items(order_id, product_id)
› SQL Solution
SELECT a.product_id AS product_a, b.product_id AS product_b, COUNT(*) AS times_bought_together FROM order_items a JOIN order_items b ON a.order_id = b.order_id AND a.product_id < b.product_id GROUP BY a.product_id, b.product_id ORDER BY times_bought_together DESC;
› Python Solution
m = df.merge(df, on="order_id", suffixes=("_a","_b")) m = m[m["product_id_a"] < m["product_id_b"]] pairs = ( m.groupby(["product_id_a","product_id_b"]) .size() .reset_index(name="times_bought_together") .sort_values("times_bought_together", ascending=False) )
S5
Find consecutive orders by the same customer and calculate days between them.
orders(order_id, customer_id, order_date)
› SQL Solution
-- Window-function approach (preferred over a self-join): SELECT customer_id, order_date, LAG(order_date) OVER (PARTITION BY customer_id ORDER BY order_date) AS prev_order_date, order_date - LAG(order_date) OVER (PARTITION BY customer_id ORDER BY order_date) AS days_between FROM orders ORDER BY customer_id, order_date;
› Python Solution
df = df.sort_values(["customer_id","order_date"]) df["prev_order_date"] = df.groupby("customer_id")["order_date"].shift(1) df["days_between"] = (df["order_date"] - df["prev_order_date"]).dt.days
Good discussion point: ask why LAG() is usually preferable to a self-join here (performance, readability).
Round 4

Gaps & Islands

G1
Find users active on 3+ consecutive days.
logins(user_id, login_date)
› SQL Solution
WITH grp AS ( SELECT user_id, login_date, login_date - (ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_date))::int AS grp_key FROM (SELECT DISTINCT user_id, login_date FROM logins) d ) SELECT user_id, MIN(login_date) AS streak_start, MAX(login_date) AS streak_end, COUNT(*) AS streak_len FROM grp GROUP BY user_id, grp_key HAVING COUNT(*) >= 3;
› Python Solution
d = logins.drop_duplicates(["user_id","login_date"]).sort_values(["user_id","login_date"]) d["rn"] = d.groupby("user_id").cumcount() + 1 d["grp_key"] = d["login_date"] - pd.to_timedelta(d["rn"], unit="D") streaks = d.groupby(["user_id","grp_key"]).agg( streak_start=("login_date","min"), streak_end=("login_date","max"), streak_len=("login_date","count") ).reset_index() result = streaks[streaks["streak_len"] >= 3]
The core trick: date − row_number() is constant within a consecutive run. Memorize this pattern.
G2
Find the longest streak of consecutive login days per user.
logins(user_id, login_date)
› SQL Solution
WITH grp AS ( SELECT user_id, login_date, login_date - (ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_date))::int AS grp_key FROM (SELECT DISTINCT user_id, login_date FROM logins) d ), streaks AS ( SELECT user_id, grp_key, COUNT(*) AS streak_len FROM grp GROUP BY user_id, grp_key ) SELECT user_id, MAX(streak_len) AS longest_streak FROM streaks GROUP BY user_id ORDER BY longest_streak DESC;
› Python Solution
d = logins.drop_duplicates(["user_id","login_date"]).sort_values(["user_id","login_date"]) d["rn"] = d.groupby("user_id").cumcount() + 1 d["grp_key"] = d["login_date"] - pd.to_timedelta(d["rn"], unit="D") streak_len = d.groupby(["user_id","grp_key"]).size().reset_index(name="streak_len") longest = streak_len.groupby("user_id")["streak_len"].max().reset_index(name="longest_streak")
G3
Find gaps of 7+ days with no activity for each user (inverse problem).
activity(user_id, activity_date)
› SQL Solution
WITH ordered AS ( SELECT user_id, activity_date, LEAD(activity_date) OVER (PARTITION BY user_id ORDER BY activity_date) AS next_date FROM (SELECT DISTINCT user_id, activity_date FROM activity) d ) SELECT user_id, activity_date AS gap_start, next_date AS gap_end, next_date - activity_date AS gap_length FROM ordered WHERE next_date - activity_date >= 7 ORDER BY user_id, activity_date;
› Python Solution
d = activity.drop_duplicates(["user_id","activity_date"]).sort_values(["user_id","activity_date"]) d["next_date"] = d.groupby("user_id")["activity_date"].shift(-1) d["gap_length"] = (d["next_date"] - d["activity_date"]).dt.days gaps = d[d["gap_length"] >= 7][["user_id","activity_date","next_date","gap_length"]] gaps.columns = ["user_id","gap_start","gap_end","gap_length"]
Flips the pattern: use LEAD (not the row-number trick) since you're measuring distance between consecutive rows, not grouping runs.
G4
Given subscription start/end dates per user, merge overlapping or adjacent date ranges into single periods.
subscriptions(user_id, start_date, end_date)
› SQL Solution
WITH ordered AS ( SELECT user_id, start_date, end_date, MAX(end_date) OVER ( PARTITION BY user_id ORDER BY start_date ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING ) AS running_max_end FROM subscriptions ), flagged AS ( SELECT *, CASE WHEN start_date <= COALESCE(running_max_end, start_date) + INTERVAL '1 day' THEN 0 ELSE 1 END AS is_new_group FROM ordered ), grouped AS ( SELECT *, SUM(is_new_group) OVER (PARTITION BY user_id ORDER BY start_date) AS grp_id FROM flagged ) SELECT user_id, MIN(start_date) AS merged_start, MAX(end_date) AS merged_end FROM grouped GROUP BY user_id, grp_id ORDER BY user_id, merged_start;
› Python Solution
d = subscriptions.sort_values(["user_id","start_date"]).copy() d["running_max_end"] = d.groupby("user_id")["end_date"].cummax().shift(1) d["is_new_group"] = ( d["start_date"] > d["running_max_end"] + pd.Timedelta(days=1) ) | d["running_max_end"].isna() d["grp_id"] = d.groupby("user_id")["is_new_group"].cumsum() merged = d.groupby(["user_id","grp_id"]).agg( merged_start=("start_date","min"), merged_end=("end_date","max") ).reset_index(drop=False).drop(columns="grp_id")
Harder variant: ranges instead of single dates. Needs a running MAX of end_date, not just a row count, to detect overlap.
G5
Find machines with 5+ consecutive failed status checks that ended in a "resolved" event.
machine_checks(machine_id, check_time, status) -- status: 'failed' | 'resolved' | 'ok'
› SQL Solution
WITH ordered AS ( SELECT *, ROW_NUMBER() OVER (PARTITION BY machine_id ORDER BY check_time) AS rn FROM machine_checks ), failed_only AS ( SELECT *, ROW_NUMBER() OVER (PARTITION BY machine_id ORDER BY check_time) - ROW_NUMBER() OVER (PARTITION BY machine_id, (status='failed') ORDER BY check_time) AS grp_key FROM ordered WHERE status = 'failed' ), streaks AS ( SELECT machine_id, grp_key, COUNT(*) AS streak_len, MIN(check_time) AS streak_start, MAX(check_time) AS streak_end FROM failed_only GROUP BY machine_id, grp_key HAVING COUNT(*) >= 5 ) SELECT s.* FROM streaks s JOIN machine_checks mc ON mc.machine_id = s.machine_id AND mc.check_time = ( SELECT MIN(check_time) FROM machine_checks mc2 WHERE mc2.machine_id = s.machine_id AND mc2.check_time > s.streak_end ) WHERE mc.status = 'resolved';
› Python Solution
d = machine_checks.sort_values(["machine_id","check_time"]).copy() d["rn_all"] = d.groupby("machine_id").cumcount() d["rn_failed"] = d[d.status=="failed"].groupby("machine_id").cumcount() d["grp_key"] = d["rn_all"] - d["rn_failed"] failed = d[d.status == "failed"] streaks = failed.groupby(["machine_id","grp_key"]).agg( streak_len=("status","count"), streak_start=("check_time","min"), streak_end=("check_time","max") ).reset_index() streaks = streaks[streaks["streak_len"] >= 5] def next_status_is_resolved(row): after = d[(d.machine_id==row.machine_id) & (d.check_time > row.streak_end)] if after.empty: return False return after.sort_values("check_time").iloc[0]["status"] == "resolved" streaks = streaks[streaks.apply(next_status_is_resolved, axis=1)]
Combines gaps-and-islands grouping with a conditional lookup after the streak — a strong senior-level filter question.
Round 5

Retention / Cohort

R1
Calculate Day-1, Day-7, Day-30 retention for users grouped by signup cohort.
users(user_id, signup_date) · activity(user_id, activity_date)
› SQL Solution
WITH cohort AS ( SELECT user_id, date_trunc('month', signup_date) AS cohort_month, signup_date FROM users ), act AS ( SELECT a.user_id, c.cohort_month, (a.activity_date - c.signup_date) AS days_since_signup FROM activity a JOIN cohort c ON a.user_id = c.user_id ) SELECT c.cohort_month, COUNT(DISTINCT c.user_id) AS cohort_size, COUNT(DISTINCT CASE WHEN a.days_since_signup = 1 THEN a.user_id END) AS day1_retained, COUNT(DISTINCT CASE WHEN a.days_since_signup = 7 THEN a.user_id END) AS day7_retained, COUNT(DISTINCT CASE WHEN a.days_since_signup = 30 THEN a.user_id END) AS day30_retained FROM cohort c LEFT JOIN act a ON c.user_id = a.user_id GROUP BY c.cohort_month ORDER BY c.cohort_month;
› Python Solution
users["cohort_month"] = users["signup_date"].dt.to_period("M").dt.to_timestamp() act = activity.merge(users[["user_id","signup_date","cohort_month"]], on="user_id") act["days_since_signup"] = (act["activity_date"] - act["signup_date"]).dt.days def retained_on(day): ids = act.loc[act.days_since_signup==day, "user_id"].unique() return users.assign(retained=users.user_id.isin(ids)) cohort_size = users.groupby("cohort_month")["user_id"].nunique() day1 = retained_on(1).groupby("cohort_month")["retained"].sum() day7 = retained_on(7).groupby("cohort_month")["retained"].sum() day30 = retained_on(30).groupby("cohort_month")["retained"].sum() result = pd.DataFrame({"cohort_size": cohort_size, "day1_retained": day1, "day7_retained": day7, "day30_retained": day30})
Pitfall: the denominator must always be cohort_size (fixed), not a shifting count — a common mistake that inflates retention.
R2
Build a retention curve: % of cohort still active in each week after signup (week 0–8).
users(user_id, signup_date) · activity(user_id, activity_date)
› SQL Solution
WITH cohort AS ( SELECT user_id, date_trunc('week', signup_date) AS cohort_week, signup_date FROM users ), act AS ( SELECT a.user_id, c.cohort_week, FLOOR((a.activity_date - c.signup_date) / 7) AS week_number FROM activity a JOIN cohort c ON a.user_id = c.user_id WHERE a.activity_date >= c.signup_date ), weeks AS (SELECT generate_series(0,8) AS week_number), sizes AS ( SELECT cohort_week, COUNT(DISTINCT user_id) AS cohort_size FROM cohort GROUP BY cohort_week ) SELECT s.cohort_week, w.week_number, COUNT(DISTINCT a.user_id) AS active_users, ROUND(100.0 * COUNT(DISTINCT a.user_id) / s.cohort_size, 1) AS pct_retained FROM sizes s CROSS JOIN weeks w LEFT JOIN act a ON a.cohort_week = s.cohort_week AND a.week_number = w.week_number GROUP BY s.cohort_week, w.week_number, s.cohort_size ORDER BY s.cohort_week, w.week_number;
› Python Solution
users["cohort_week"] = users["signup_date"].dt.to_period("W").dt.start_time act = activity.merge(users[["user_id","signup_date","cohort_week"]], on="user_id") act = act[act["activity_date"] >= act["signup_date"]] act["week_number"] = ((act["activity_date"] - act["signup_date"]).dt.days // 7) cohort_size = users.groupby("cohort_week")["user_id"].nunique() rows = [] for cw, grp in act.groupby("cohort_week"): for w in range(0, 9): active = grp.loc[grp.week_number==w, "user_id"].nunique() rows.append({"cohort_week": cw, "week_number": w, "active_users": active, "pct_retained": round(100*active/cohort_size[cw],1)}) result = pd.DataFrame(rows)
Needs a cross join / cartesian spine of weeks 0–8 so weeks with zero active users still show 0%, not a missing row.
R3
Classify users into New / Retained / Resurrected / Churned for each month.
activity(user_id, activity_month) -- one row per user per active month
› SQL Solution
WITH months AS (SELECT DISTINCT activity_month FROM activity), user_months AS (SELECT user_id, activity_month FROM activity), flagged AS ( SELECT um.user_id, m.activity_month, EXISTS (SELECT 1 FROM user_months u WHERE u.user_id=um.user_id AND u.activity_month = m.activity_month) AS active_this_month, EXISTS (SELECT 1 FROM user_months u WHERE u.user_id=um.user_id AND u.activity_month = m.activity_month - INTERVAL '1 month') AS active_prev_month, EXISTS (SELECT 1 FROM user_months u WHERE u.user_id=um.user_id AND u.activity_month < m.activity_month) AS active_before FROM (SELECT DISTINCT user_id FROM activity) um CROSS JOIN months m ) SELECT activity_month, CASE WHEN active_this_month AND NOT active_before THEN 'New' WHEN active_this_month AND active_prev_month THEN 'Retained' WHEN active_this_month AND NOT active_prev_month AND active_before THEN 'Resurrected' WHEN NOT active_this_month AND active_prev_month THEN 'Churned' END AS status, COUNT(DISTINCT user_id) AS user_count FROM flagged WHERE active_this_month OR active_prev_month GROUP BY activity_month, status ORDER BY activity_month, status;
› Python Solution
all_months = sorted(activity["activity_month"].unique()) all_users = activity["user_id"].unique() active_set = activity.groupby("activity_month")["user_id"].apply(set).to_dict() rows = [] seen_before = {u: False for u in all_users} prev_month_active = set() for m in all_months: this_month_active = active_set.get(m, set()) for u in all_users: is_active = u in this_month_active was_active_prev = u in prev_month_active was_active_before = seen_before[u] if is_active and not was_active_before: status = "New" elif is_active and was_active_prev: status = "Retained" elif is_active and not was_active_prev and was_active_before: status = "Resurrected" elif not is_active and was_active_prev: status = "Churned" else: status = None if status: rows.append({"activity_month": m, "user_id": u, "status": status}) prev_month_active = this_month_active for u in this_month_active: seen_before[u] = True result = pd.DataFrame(rows).groupby(["activity_month","status"]).size().reset_index(name="user_count")
Senior-level filter: candidate must define "New" vs "Resurrected" themselves — reveals how they handle ambiguity.
R4
Calculate rolling 28-day active users (a DAU/MAU-style metric).
activity(user_id, activity_date)
› SQL Solution
SELECT a.activity_date, COUNT(DISTINCT b.user_id) AS active_last_28d FROM (SELECT DISTINCT activity_date FROM activity) a JOIN activity b ON b.activity_date BETWEEN a.activity_date - INTERVAL '27 days' AND a.activity_date GROUP BY a.activity_date ORDER BY a.activity_date;
› Python Solution
dates = sorted(activity["activity_date"].unique()) rows = [] for d in dates: window_start = d - pd.Timedelta(days=27) active = activity.loc[ (activity.activity_date >= window_start) & (activity.activity_date <= d), "user_id" ].nunique() rows.append({"activity_date": d, "active_last_28d": active}) result = pd.DataFrame(rows)
Pitfall: this is a RANGE-based window (by date value), not ROWS-based — ROWS BETWEEN 27 PRECEDING breaks if dates have gaps or duplicates.
R5
Given a cohort's revenue table, calculate cumulative LTV per cohort by month since signup.
users(user_id, signup_date) · revenue(user_id, revenue_date, revenue)
› SQL Solution
WITH cohort AS ( SELECT user_id, date_trunc('month', signup_date) AS cohort_month FROM users ), monthly AS ( SELECT c.cohort_month, (EXTRACT(YEAR FROM r.revenue_date) - EXTRACT(YEAR FROM c.cohort_month)) * 12 + (EXTRACT(MONTH FROM r.revenue_date) - EXTRACT(MONTH FROM c.cohort_month)) AS month_since_signup, r.revenue FROM revenue r JOIN cohort c ON r.user_id = c.user_id ), cohort_size AS ( SELECT cohort_month, COUNT(DISTINCT user_id) AS n_users FROM cohort GROUP BY cohort_month ), agg AS ( SELECT cohort_month, month_since_signup, SUM(revenue) AS month_revenue FROM monthly GROUP BY cohort_month, month_since_signup ) SELECT a.cohort_month, a.month_since_signup, SUM(a.month_revenue) OVER ( PARTITION BY a.cohort_month ORDER BY a.month_since_signup ) / cs.n_users AS cumulative_ltv_per_user FROM agg a JOIN cohort_size cs ON a.cohort_month = cs.cohort_month ORDER BY a.cohort_month, a.month_since_signup;
› Python Solution
users["cohort_month"] = users["signup_date"].dt.to_period("M") rev = revenue.merge(users[["user_id","cohort_month"]], on="user_id") rev["revenue_month"] = rev["revenue_date"].dt.to_period("M") rev["month_since_signup"] = (rev["revenue_month"] - rev["cohort_month"]).apply(lambda x: x.n) cohort_size = users.groupby("cohort_month")["user_id"].nunique() monthly = rev.groupby(["cohort_month","month_since_signup"])["revenue"].sum().reset_index() monthly = monthly.sort_values(["cohort_month","month_since_signup"]) monthly["cum_revenue"] = monthly.groupby("cohort_month")["revenue"].cumsum() monthly["cumulative_ltv_per_user"] = monthly.apply( lambda row: row["cum_revenue"] / cohort_size[row["cohort_month"]], axis=1 )
Combines cohort bucketing + running SUM window function + per-user normalization — a good "tie it all together" closing question.

The recurring theme: SQL's PARTITION BY almost always maps to pandas' groupby(), and OVER (ORDER BY ...) window logic maps to a method chained after groupby() — .cumsum(), .shift(), .rank(), .rolling(). If you can say the SQL window function out loud, you can usually guess its pandas equivalent from this pattern alone.
