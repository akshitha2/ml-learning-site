Agent Architecture Fundamentals

Study guide compiled from Deep Dive + Grill Me session · 9 subtopics

1. Agent vs. Pipeline 2. Loop Mechanics 3. Tool Calling Internals 4. Memory Systems 5. Planning Strategies 6. State Management & Control Flow 7. Multi-Agent Architectures 8. Failure Modes & Reliability 9. Evaluation of Agents ⚠ Gaps to Review Interview Questions

1. Agent vs. Pipeline

Defining criterion: An agent is a system where the LLM's output determines the next action taken by the system — with results fed back before the next decision. A fixed pipeline (retrieve→generate, always) is not an agent even if an LLM is involved.

Loop: Perceive → Reason → Act → (feed result back into Perceive) → repeat until termination.

Perceive re-runs every iteration — it reads the current accumulated state, not just the original query.

3. Tool Calling Internals

Schema does two jobs: tool selection signal (via description) and argument constraint (type, enum, required).

Constrained decoding: token sampling is restricted at each step to stay valid against the schema — this is why native function calling beats ReAct parsing structurally, not just "tries harder."

FailureExample
Argument hallucinationRight tool, wrong value (NYC → NJ)
Wrong tool selectionIrrelevant/no results — entirely wrong tool called
Type mismatch"5" (string) passed where int expected

Sequential vs. parallel calls: sequential iff call 2's arguments can't be resolved without call 1's output.

4. Memory Systems

TypeScopeFailure mode
Working memoryCurrent loop onlyDies with the task, by design
Short-term (sliding window)Recent N turnsTotal loss of anything before the window — silent
Short-term (summarization)Whole conversation, compressedLossy — flattens strict constraints into soft approximations
Long-termAcross sessionsStale/contradicted memories need invalidation
Fix for hard constraints: pull them out of summarization entirely into a pinned, always-injected structured slot (e.g. constraints: {budget_max: 4200}). Don't let hard constraints get compressed like narrative context.

Memory vs. RAG: RAG = external knowledge base. Agent memory = the agent's own past experience (autobiographical).

6. State Management & Control Flow

Control flowImplicit — model decidesExplicit — code defines edges
Invariant enforcementRequested via prompt (not guaranteed)Hard-enforced via edge conditions
Best forOpen-ended, exploratoryCompliance/safety-critical production
Not automatically solved by graphs: state resumption after a human-approval node fires — resume same node? restart the step? roll back further? This is a separate design decision.

7. Multi-Agent Architectures

Orchestrator-Worker: orchestrator decomposes + routes to narrow-scoped workers, then synthesizes.

Hierarchical: orchestrators of orchestrators — more modularity, compounding latency, harder failure tracing.

Debate/Consensus: independent agents cross-critique — avoids the shared-blind-spot problem that single-agent Reflection has (different model/context ≠ same systematic error).

Key design question: shared vs. isolated context. Isolated + explicit hand-off is the default in production — avoids context bloat and cross-contamination between workers.

8. Failure Modes & Reliability

FailureFix
Infinite/runaway loopsMax-iteration cap, "no progress" detection
Hallucinated tool callsStrict schema validation before execution
Error propagationStructured error schema + explicit "don't assume success" instruction
Context poisoningReflection/validation nodes that can correct earlier turns
GuardrailsCode-enforced (approval gates, rate limits) — never model-only

9. Evaluation of Agents

Task success rate alone hides inefficient/unsafe paths — need trajectory analysis: efficiency, redundancy, mid-trajectory constraint adherence, recovery behavior.

Offline vs. online: offline = fast/reproducible but may miss real edge cases; online = real signal, real cost.

LLM-as-judge: scalable but can share the agent's blind spots — mitigate with a different/stronger judge + concrete rubrics.

Latency/step-count belongs in trajectory metrics — a correct-but-slow run passes a pure success/fail eval silently.

⚠ Gaps to Review (from your Grill Me answers)

Gap 1 — Agent vs. pipeline definition

You said: "When LLM makes decisions instead of human, it's an agent."

Missing: A plain LLM call also "decides" what text to output — that's not sufficient. The defining piece is the feedback loop: the LLM's output determines the next action, and the result of that action feeds back in before the next decision. No loop = not an agent, regardless of whether a decision was made.

Gap 2 — Why native function calling beats ReAct

You said: function calling gives finite/valid cases; ReAct can hallucinate.

Missing: the mechanism — constrained decoding restricts token sampling itself to stay schema-valid at each step. It's not that the model "tries harder," the decoding process structurally cannot emit invalid output. ReAct has no such constraint; it's free text your harness parses after the fact.

Gap 3 — What graphs don't solve automatically

You said (after 2 attempts): guessed "ask human to proceed" / "human intervention or budget agent" — didn't land on it.

Answer: state resumption after a human-approval node fires. The graph guarantees the check happens; it says nothing about whether execution resumes at the same node, restarts the step, or rolls back further. That's a design decision you own separately.

Gap 4 — Isolated vs. shared context in multi-agent

You said: shared context "might confuse" workers.

Missing: be concrete — two distinct risks: (1) context bloat, irrelevant info competing for attention, and (2) cross-contamination, a worker acting on another worker's tool calls/results that aren't relevant to its own task.

Gap 5 (Deep Dive) — Empty tool payload handling

You said: handle at the observation step.

Missing: location is right but incomplete — you also need explicit reasoning/prompt instructions telling the model what "empty" means and what to do about it (retry broader? report not-found? fallback tool?). Schema alone doesn't guarantee correct interpretation.

Gap 6 (Deep Dive) — 504 error propagation fix

You said: schema checks instead of plain string fix it.

Missing: schema is necessary but not sufficient — pair it with an explicit system-prompt instruction ("never claim success without a success-shaped observation"), since a model can still misinterpret a well-structured error if not told how to treat it.

Interview Questions

What's the architectural difference between an agent and a RAG pipeline? Easy

RAG has fixed control flow (always retrieve → generate) determined by code. An agent's control flow is determined dynamically by the LLM's own output at each step — whether to retrieve, what to retrieve, whether to retrieve again.

Why does context length grow O(n²) in tokens processed across an agent trajectory, not O(n)? Medium

Each iteration re-sends the full accumulated history (all prior tool calls/results) in the prompt. At step k you resend k-1 prior turns, so total tokens processed across a full n-step trajectory sums to O(n²).

Explain constrained decoding and why it makes native function calling more reliable than ReAct-style parsing. Medium

Constrained decoding restricts the token sampling process itself so only schema-valid tokens can be emitted at each generation step — validity is structurally guaranteed, not hoped for. ReAct relies on free-text generation parsed post-hoc by the harness, which breaks on malformed output even when intent was correct.

Design a memory strategy for an agent handling long-running sessions with hard user constraints (e.g., budget caps, exclusions). Hard

Separate hard constraints from narrative history. Use summarization/sliding window for general context, but extract constraints into a structured, always-injected slot (not subject to compression) — e.g. a constraints object re-added to every prompt verbatim, independent of the rolling summary.

When would you choose a graph-based (state machine) architecture over a single-agent loop, and what's the cost? Medium

Choose graph-based when you need to hard-enforce invariants (approval gates, spending limits, compliance checks) — code-level edges guarantee enforcement vs. prompted requests that can be missed. Cost: reduced flexibility — rigid graphs struggle with genuinely open-ended tasks that don't fit predefined nodes.

Your agent has 98% task success on your offline eval set but users complain in production. Diagnose. Hard

Classic train/serve skew applied to agent evals — the eval set doesn't represent the real production distribution of tasks/edge cases. Also check if success rate is masking trajectory issues (latency, redundant calls, mid-trajectory constraint violations that get silently corrected) that never surface in a pass/fail metric.

Why can LLM-as-judge evaluation fail silently, and how do you mitigate it? Medium

The judge model can share the same training biases/blind spots as the agent model being evaluated, rubber-stamping systematic errors. Mitigate with a stronger or architecturally different judge model, plus concrete fact-checkable rubrics (e.g. exact budget compliance) rather than vague quality scoring.

Multi-agent orchestrator-worker: should the Budget Agent enforce the spending limit, or should the Orchestrator? Hard

Orchestrator, via a graph edge — code-level enforcement guarantees the check fires every time. If left to the Budget Agent's LLM reasoning, it's a request, not a guarantee, and can be silently missed. Split responsibility: Budget Agent tracks/reports the running total (data), Orchestrator enforces the gate (control).
