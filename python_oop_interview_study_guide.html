Mental Model

Classes & Objects

A class is a blueprint; an object is a specific instance built from it. Every attribute lookup on an instance falls back to the class, then to its parents — this single fact underlies almost every OOP "gotcha" in Python.

class Dog: species = "Canis familiaris" # class attribute — shared by ALL instances def __init__(self, name, age): self.name = name # instance attribute — unique per object self.age = age def bark(self): return f"{self.name} says woof!" d1 = Dog("Rex", 3) d2 = Dog("Fido", 5) print(d1.species, d2.species) # both share the same class attribute
💡

Attribute lookup order is: instance __dict__ → class __dict__ → parent class(es) __dict__ (following MRO). This is why reassigning a class attribute via an instance (d1.species = "x") creates a new instance attribute instead of mutating the shared one.

Pillar 1

Encapsulation

Bundling data and the methods that operate on it, and restricting direct access to internal state. Python has no true "private" — it relies on naming conventions and one bit of automatic name mangling.

ConventionMeaning
publicNo prefix — freely accessible, part of the public API
_protectedSingle underscore — "internal use" by convention only, nothing stops access
__privateDouble underscore — triggers name mangling to _ClassName__attr
__dunder__Reserved for Python's own magic methods — don't invent new ones

@property — controlled access

class BankAccount: def __init__(self, balance): self._balance = balance @property def balance(self): return self._balance @balance.setter def balance(self, value): if value < 0: raise ValueError("Balance cannot be negative") self._balance = value acc = BankAccount(100) acc.balance = 50 # calls the setter, validated print(acc.balance) # calls the getter — looks like plain attribute access
Trap — "Private" isn't private

__balance is just renamed to _ClassName__balance at compile time. It's still fully accessible — name mangling exists to avoid accidental collisions in subclasses, not to enforce security.

Pillar 2

Inheritance & MRO (Diamond Problem)

Single Inheritance
  • One parent class
  • super().__init__() calls the parent's constructor
Multiple Inheritance
  • A class inherits from 2+ parents
  • Resolved via MRO (C3 linearization)

super() and the Diamond Problem

class Animal: def speak(self): return "..." class Dog(Animal): def speak(self): return "Woof" class Robot: def speak(self): return "Beep" class RoboDog(Dog, Robot): pass print(RoboDog().speak()) # "Woof" — MRO checks Dog before Robot print(RoboDog.__mro__) # (RoboDog, Dog, Animal, Robot, object)
💡

Python resolves multiple inheritance with MRO (Method Resolution Order), computed via the C3 linearization algorithm — left-to-right, depth-first, but never visiting a class before its own children. Check it any time with ClassName.__mro__ or ClassName.mro().

Trap — super() in multiple inheritance

super() does not mean "my direct parent" — it means "the next class in the MRO chain." With multiple inheritance and cooperative super().__init__() calls in every class, all parent constructors run in MRO order, not just the first one.

Pillar 3

Polymorphism & Duck Typing

The same interface (method name) behaves differently depending on the object's actual class. Python leans on duck typing — "if it walks like a duck and quacks like a duck" — over strict type checks.

class Cat: def speak(self): return "Meow" class Duck: def speak(self): return "Quack" for animal in [Cat(), Duck()]: print(animal.speak()) # works for ANY object with a .speak() method
💡

Duck typing means Python never checks isinstance() before calling .speak() — if the object has that method, it works. This is what makes len() work on lists, strings, and dicts alike (each defines __len__).

Pillar 4

Abstraction — ABCs

Hiding implementation details behind a required interface. Python enforces this with the abc module — a class with an abstract method cannot be instantiated until every abstract method is overridden.

from abc import ABC, abstractmethod class Model(ABC): @abstractmethod def fit(self, X, y): ... @abstractmethod def predict(self, X): ... class LinearModel(Model): def fit(self, X, y): # training logic return self def predict(self, X): # prediction logic return X # Model() -> TypeError: Can't instantiate abstract class lm = LinearModel() # OK — all abstract methods implemented
Trap — forgetting to inherit ABC

@abstractmethod only enforces anything if the class inherits from ABC (or uses ABCMeta as its metaclass). Slap it on a plain class and it's just a decorator with no effect — the class instantiates fine, silently.

Worked Example — Vector class with operator overloading

class Vector: def __init__(self, x, y): self.x, self.y = x, y def __add__(self, other): return Vector(self.x + other.x, self.y + other.y) def __repr__(self): return f"Vector({self.x}, {self.y})" v1, v2 = Vector(1, 2), Vector(3, 4) print(v1 + v2) # Vector(4, 6) — calls __add__ under the hood
Trap — __repr__ vs __str__

If __str__ is missing, Python falls back to __repr__ for print() and str(). The reverse is not true. Always implement __repr__ at minimum — ideally something that could be pasted back into Python to recreate the object.

Method Types

Instance vs @classmethod vs @staticmethod

class Employee: raise_pct = 1.05 def __init__(self, salary): self.salary = salary def give_raise(self): # instance method — needs self self.salary *= Employee.raise_pct @classmethod def from_string(cls, data_str): # classmethod — needs cls, not self salary = float(data_str) return cls(salary) # alternative constructor pattern @staticmethod def is_valid_salary(salary): # staticmethod — needs neither return salary > 0
TypeFirst argUse case
instance methodselfReads/writes instance state
@classmethodclsAlternative constructors, factory methods, touches class state
@staticmethodnoneUtility/helper logic that's related but needs no self/cls
Trap — classmethod as constructor + inheritance

cls(salary) inside a @classmethod — not Employee(salary) — so that from_string correctly returns a subclass instance if called on a subclass. Hardcoding the class name breaks polymorphic construction.

DS/ML Application

OOP Patterns in ML Code

scikit-learn's entire API is a consistent OOP contract: every estimator implements fit(), most implement predict() or transform(). Interviewers love asking you to replicate this.

Custom sklearn-style estimator

from sklearn.base import BaseEstimator, TransformerMixin class ClipOutliers(BaseEstimator, TransformerMixin): def __init__(self, lower=0.01, upper=0.99): self.lower = lower self.upper = upper def fit(self, X, y=None): self.lo_ = X.quantile(self.lower) self.hi_ = X.quantile(self.upper) return self # fit MUST return self — chaining relies on it def transform(self, X): return X.clip(self.lo_, self.hi_, axis=1)
💡

BaseEstimator gives free get_params()/set_params() (needed for GridSearchCV); TransformerMixin gives a free fit_transform() built by composing your fit and transform. This is mixin composition, not deep inheritance — a recurring interview theme.

Composition vs Inheritance

Inheritance ("is-a")
  • Use when a subclass truly specializes a base behavior
  • Risk: deep hierarchies get fragile and hard to test
Composition ("has-a")
  • A class holds other objects as attributes and delegates to them
  • Preferred in ML pipelines — e.g. a Pipeline holding a list of transformer objects
Interview framing
"Favor composition over inheritance" is a real design principle, not just a soundbite — be ready to justify it with an ML example, like a Pipeline class composed of independent fit/transform steps rather than one giant class inheriting from many mixins.
Summary

Gotchas Summary

G1Mutable default arguments

Default argument values are evaluated ONCE, at function definition time, and reused across every call. Never use def __init__(self, items=[]) — use items=None then self.items = items or [] inside.

G2Class attribute mutation vs instance attribute creation

instance.attr = value always creates/overwrites an instance attribute, never mutates the class attribute — even if the names match. But instance.some_list.append(x) on a shared mutable class attribute DOES mutate it for every instance, since it's the same list object.

G3Shallow copy vs deep copy

copy.copy() duplicates the outer object only — nested mutable objects (lists, dicts, custom objects) are still shared references. copy.deepcopy() recursively duplicates everything.

G4__slots__ and memory

Normal instances store attributes in a per-instance __dict__, which has real memory overhead. __slots__ = ['x', 'y'] removes that dict, saving memory at scale — but also blocks adding new attributes dynamically and complicates multiple inheritance.

G5Private isn't private

__attr is name-mangled to _ClassName__attr, not hidden. It exists to avoid subclass naming collisions, not for access control — anyone can still reach it if they know the mangled name.

G6Calling overridden methods from __init__

If a base class's __init__ calls self.setup() and a subclass overrides setup(), the subclass version runs — even during the base class's own construction. This can break if the subclass version depends on attributes not yet set.

Practice

30 Interview Questions

Q1
What are the four pillars of OOP, and give one Python-specific example of each.
+
Answer
  • Encapsulation — bundling state + behavior, using @property for controlled access
  • Inheritance — class Dog(Animal), reusing/extending behavior
  • Polymorphism — duck typing, e.g. any object with .speak() works in the same loop
  • Abstraction — ABC + @abstractmethod to enforce an interface
Q2
Is a "private" attribute in Python actually private? Explain name mangling.
+
Answer

No — Python has no enforced access control. __attr gets renamed to _ClassName__attr at compile time (name mangling), mainly to prevent accidental overwrites in subclasses, not to block access. It's still reachable via the mangled name.

Q3
Difference between a class attribute and an instance attribute — with a mutable-default trap.
+
Answer
class Cart: def __init__(self, items=[]): # BUG: shared across all instances self.items = items c1 = Cart(); c1.items.append("apple") c2 = Cart() print(c2.items) # ['apple'] — unexpectedly shared!
Trap

Default arguments are evaluated once at function definition. Fix with items=None and self.items = items if items is not None else [].

Q4
What's the difference between __new__ and __init__?
+
Answer

__new__ is the actual object constructor — it allocates and returns the new instance, called before __init__. __init__ just initializes that already-created instance's attributes and returns None. __new__ is rarely overridden except for immutable types (like subclassing tuple/str) or singleton patterns.

Q5
@staticmethod vs @classmethod — when would you use each?
+
Answer
  • @classmethod receives cls — used for alternative constructors (e.g. from_json) or logic touching class-level state
  • @staticmethod receives neither self nor cls — a namespaced utility function that's logically related but needs no class/instance data
Q6
Explain duck typing and why Python favors it over explicit interface checks.
+
Answer

"If it walks like a duck and quacks like a duck, it's a duck" — Python cares whether an object supports the needed method/attribute, not its declared type. This is why len(obj) works on any object defining __len__, regardless of class hierarchy.

Q7
What does super() actually do? Is it always "my parent class"?
+
Answer

No — super() returns a proxy to the next class in the MRO, not necessarily the direct parent. In single inheritance they're the same thing, but under multiple inheritance, MRO can route through a sibling branch first.

Q8
Difference between __str__ and __repr__.
+
Answer

__str__ is for a human-readable, end-user-facing string (used by print()/str()). __repr__ is for an unambiguous, developer-facing representation — ideally valid Python that recreates the object — used by the REPL and as a fallback when __str__ is missing.

Q9
How do you make a custom class sortable with sorted() or the < operator?
+
Answer
class Employee: def __init__(self, salary): self.salary = salary def __lt__(self, other): return self.salary < other.salary sorted(employees) # uses __lt__ under the hood

Implement __lt__ (and optionally __eq__); for full ordering support use functools.total_ordering to auto-derive the rest from just __eq__ + one comparison method.

Q10
What is method overriding, and how is it different from overloading (which Python doesn't really have)?
+
Answer

Overriding = a subclass redefines a method with the same name/signature as its parent, replacing the behavior — this is core polymorphism. Python has no true method overloading (same name, different signatures) like Java/C++; the last-defined method with that name simply wins. Default arguments or *args/**kwargs simulate it.

Q11
Can you instantiate a class with an abstract method that isn't overridden? What error do you get?
+
Answer

No — TypeError: Can't instantiate abstract class X with abstract method(s) Y, but only if the class inherits from ABC/uses ABCMeta. Without that, @abstractmethod is decorative and has no enforcement.

Q12
What's the difference between composition and inheritance? When would you prefer one?
+
Answer

Inheritance models "is-a" relationships by extending a base class; composition models "has-a" by holding other objects as attributes and delegating to them. Prefer composition when behavior is better assembled from independent, swappable parts (e.g. a pipeline of transformer steps) — it avoids deep, fragile hierarchies.

Q13
What does __call__ do, and give a practical use case.
+
Answer
class Multiplier: def __init__(self, factor): self.factor = factor def __call__(self, x): return x * self.factor double = Multiplier(2) double(5) # 10 — instance used like a function

Defining __call__ makes instances callable like functions. Common in ML for stateful "function objects" — e.g. a configured preprocessing step or a custom loss function holding hyperparameters.

Q14
What's the difference between a shallow copy and a deep copy of an object?
+
Answer

copy.copy() creates a new outer object but nested mutable attributes still point to the same objects as the original. copy.deepcopy() recursively clones every nested object too, so mutating a copy never affects the original.

Q15
How do you make a class iterable? What's the difference between __iter__ and __next__?
+
Answer
class Countdown: def __init__(self, start): self.n = start def __iter__(self): return self def __next__(self): if self.n <= 0: raise StopIteration self.n -= 1 return self.n + 1

__iter__ must return an iterator object (often self); __next__ returns the next value each call and raises StopIteration when exhausted. This pair is what powers for x in obj.

Q16
Explain the diamond problem and how Python's MRO resolves it.
+
Answer

When class D inherits from both B and C, which both inherit from A, calling a method defined on A could be ambiguous about which path to follow. Python resolves this deterministically via C3 linearization: check ClassName.__mro__ for the exact left-to-right, depth-first (but child-before-parent) order Python will search.

Trap

Assuming inheritance order in class D(B, C) doesn't matter — it directly determines MRO priority; class D(C, B) gives a different resolution order.

Q17
What are mixins, and how are they used in scikit-learn's design?
+
Answer

A mixin is a small class meant to be combined via multiple inheritance to add specific reusable behavior, not to stand alone. TransformerMixin adds a free fit_transform() by composing your own fit and transform; ClassifierMixin adds a default score(). This lets sklearn compose behavior without deep, rigid hierarchies.

Q18
Why must fit() return self in a custom sklearn-compatible transformer?
+
Answer

Sklearn's method-chaining convention (model.fit(X, y).predict(X)) and pipeline internals rely on fit returning the fitted estimator itself. If it returns None instead, chained calls and Pipeline composition break.

Q19
What is __slots__ and what trade-off does it introduce?
+
Answer
class Point: __slots__ = ('x', 'y') def __init__(self, x, y): self.x, self.y = x, y

Declaring __slots__ removes the per-instance __dict__, cutting memory overhead significantly at scale (millions of small objects). Trade-off: you can no longer add arbitrary new attributes dynamically, and combining __slots__ with multiple inheritance from classes that also define slots gets tricky.

Q20
Design a Singleton pattern in Python.
+
Answer
class Singleton: _instance = None def __new__(cls, *args, **kwargs): if cls._instance is None: cls._instance = super().__new__(cls) return cls._instance a, b = Singleton(), Singleton() print(a is b) # True — same object

Override __new__ to check for and return an existing instance instead of always creating a new one. Common alternative: a module-level object, since Python modules are already singletons by import caching.

Q21
What happens if a subclass's __init__ doesn't call super().__init__()?
+
Answer

The parent class's initialization logic never runs — any attributes the parent would have set are missing, which typically surfaces later as an AttributeError when a parent method expects that state to exist.

Trap

This is easy to miss in multiple inheritance chains where several __init__s must cooperatively call super().__init__() for every parent to actually run.

Q22
Implement a custom context manager class (the __enter__/__exit__ protocol).
+
Answer
class Timer: def __enter__(self): import time self.start = time.time() return self def __exit__(self, exc_type, exc_val, exc_tb): import time print(f"Elapsed: {time.time() - self.start:.2f}s") return False # False/None re-raises any exception with Timer(): do_work()

__exit__'s return value matters: returning a truthy value suppresses an exception raised inside the with block; returning False/None lets it propagate.

Q23
How would you build a custom ML model class that mimics the sklearn fit/predict API from scratch?
+
Answer
class SimpleLinearRegression: def __init__(self, lr=0.01, epochs=100): self.lr, self.epochs = lr, epochs def fit(self, X, y): self.w_ = 0.0 self.b_ = 0.0 n = len(X) for _ in range(self.epochs): y_pred = self.w_ * X + self.b_ dw = (-2/n) * sum(X * (y - y_pred)) db = (-2/n) * sum(y - y_pred) self.w_ -= self.lr * dw self.b_ -= self.lr * db return self def predict(self, X): return self.w_ * X + self.b_

Interviewers check for: fit returning self, learned parameters stored with a trailing underscore (sklearn convention signaling "fitted state"), and hyperparameters stored unmodified in __init__.

Q24
What's the difference between an abstract base class and a Protocol (structural typing)?
+
Answer

ABC requires explicit, nominal inheritance ("this class IS-A subclass of the interface") and enforces it at instantiation. typing.Protocol (PEP 544) allows structural typing — a class doesn't need to inherit anything, it just needs to implement the right methods, checked by static type checkers rather than at runtime.

Q25
Why does Python evaluate default arguments once, and how does that interact with class-level mutable attributes?
+
Answer

Default argument objects are created once when the def statement executes, and that same object is reused on every call unless a new value is passed. It's the same root cause as class attributes holding mutable objects (like a list) — all instances that don't override it share and can mutate the one object.

Q26
How do you implement operator overloading so a custom Vector class supports both v1 + v2 and v1 + 5?
+
Answer
class Vector: def __init__(self, x, y): self.x, self.y = x, y def __add__(self, other): if isinstance(other, Vector): return Vector(self.x + other.x, self.y + other.y) elif isinstance(other, (int, float)): return Vector(self.x + other, self.y + other) return NotImplemented

Branch on the type of other inside __add__, and return NotImplemented (not raise an error) for unsupported types — this lets Python fall back to trying other.__radd__(self).

Q27
What's a metaclass, in one or two sentences, and why would sklearn use ABCMeta?
+
Answer

A metaclass is "the class of a class" — it controls how classes themselves are created, similar to how a class controls how instances are created. ABCMeta is the metaclass behind ABC; it's what actually implements the runtime check blocking instantiation of classes with unfulfilled abstract methods.

Q28
Explain the difference between == and is, and how __eq__ interacts with it.
+
Answer

is checks identity (same object in memory); == checks equality, which calls __eq__ — defaulting to identity comparison if __eq__ isn't overridden. Overriding __eq__ without also overriding __hash__ makes the object unhashable by default in Python 3 (or breaks hash consistency), since equal objects are expected to hash identically.

Trap

Defining __eq__ silently sets __hash__ to None unless you also define __hash__ explicitly — instances become unusable as dict keys or set members.

Q29
How would you design a class hierarchy for different ML model types (e.g. LinearModel, TreeModel) that share a common evaluate() method but differ in fit()/predict()?
+
Answer
  • Define an abstract BaseModel(ABC) with abstract fit()/predict()
  • Implement a shared, concrete evaluate(X, y) on the base class that calls self.predict(X) — works for every subclass via polymorphism
  • Each subclass (LinearModel, TreeModel) only needs to implement its own fit/predict

This is the Template Method pattern: the base class defines the algorithm's skeleton (evaluate), subclasses fill in the model-specific steps.

Q30
You're asked to explain "favor composition over inheritance" to a non-technical stakeholder. What do you say?
+
Answer

"Instead of building one big class that tries to be everything by inheriting from many parents, we build small, focused pieces and plug them together — like assembling Lego blocks instead of carving one complicated statue. It's easier to swap, test, and reuse individual pieces."
