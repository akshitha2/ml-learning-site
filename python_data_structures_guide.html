Mutability & Hashing — The Two Ideas Behind Everything

Before comparing lists/tuples/sets/dicts, you need two ideas nailed down, because almost every rule about these structures traces back to them.

Mutability — can you change it after creating it?

Mutable means you can change the object's contents in place after it's created (e.g. add an item, without making a brand new object). Immutable means once created, it can never be changed — any "modification" actually creates a whole new object.

my_list = [1, 2, 3] my_list.append(4) # changes the SAME list object in place — lists are mutable my_tuple = (1, 2, 3) # my_tuple.append(4) --> AttributeError! Tuples have no append — they're immutable new_tuple = my_tuple + (4,) # this creates a BRAND NEW tuple, doesn't touch the old one

Hashing — what does "hashable" mean?

A hash is a fixed-size number Python computes from a value, used to find things quickly (like a locker number computed from your name). Something is hashable if that number never changes during its lifetime — which is only guaranteed if the object is immutable. This is why sets and dictionary keys require hashable items: Python uses the hash to instantly locate where to look, and if the value could change, the hash would become wrong and lookups would silently break.

int, float, str, boolNoYesYes

💡

This single table answers a huge chunk of "why can't I do X" questions: "why can't I use a list as a dict key" → because lists are mutable, so unhashable. "Why can a tuple be a dict key but not a list" → same reason, opposite conclusion.

Structure 1

Lists

A list is an ordered, mutable collection — items keep the position you put them in, and you can add, remove, or change items after creation. Think of it as a numbered row of boxes you can rearrange freely.

nums = [10, 20, 30, 40, 50] print(nums[0]) # 10 -- indexing starts at 0 print(nums[-1]) # 50 -- negative index counts from the end print(nums[1:3]) # [20, 30] -- slicing: start included, end excluded print(nums[::2]) # [10, 30, 50] -- step of 2 nums.append(60) # adds to the end nums.insert(0, 5) # inserts at a specific position nums.remove(30) # removes the FIRST matching value (not by position)

List comprehensions

A compact way to build a new list from an existing iterable in a single readable line, instead of writing a full for-loop with .append().

squares = [x**2 for x in range(5)] # [0, 1, 4, 9, 16] evens = [x for x in range(10) if x % 2 == 0] # the "if" filters which items are included

sort() vs sorted()

FunctionBehavior
list.sort()Sorts the list IN PLACE, returns None — you don't reassign it
sorted(list)Returns a brand NEW sorted list, leaves the original untouched — works on any iterable
names = ["Charlie", "Alice", "Bob"] sorted_names = sorted(names, key=len) # key= tells Python what to sort BY (here, string length)
💡

key= takes a function — Python calls it once per item and sorts based on what that function returns, not the item itself. key=len sorts by length; key=lambda x: x.lower() sorts case-insensitively.

Quick Check — Lists
Q. What does names.sort() return, and what's the common mistake beginners make with it?
A. It returns None — the common mistake is writing names = names.sort(), which wipes out the list by assigning None to it instead of the sorted list.
Q. What's the output of [1, 2, 3][1:]?
A. [2, 3] — leaving the end index blank means "go to the end of the list."
Interview Trap — Aliasing vs Copying

b = a does NOT copy a list — it just makes b point to the exact same list object as a. Changing b changes a too. To actually copy: use b = a[:] or b = a.copy() (shallow copy — nested lists inside are still shared) or copy.deepcopy(a) for a fully independent copy.

Structure 2

Tuples

A tuple is like a list's immutable twin — ordered but you can never add, remove, or change items after creation. Once built, it's locked.

point = (3, 4) x, y = point # "unpacking" -- assigns 3 to x and 4 to y in one line # point[0] = 99 --> TypeError! Tuples don't support item assignment single = (5,) # the trailing comma is REQUIRED -- (5) is just an int in parentheses

Why bother with tuples if lists can do everything?

  • Tuples are hashable (if their contents are) → usable as dictionary keys or set members, unlike lists
  • Signals intent: "this data shouldn't change" — e.g. a fixed (latitude, longitude) pair
  • Slightly faster and more memory-efficient than lists for fixed-size data

Named tuples — tuples with labeled fields

from collections import namedtuple Point = namedtuple("Point", ["x", "y"]) # defines a new tuple-like type with named fields p = Point(3, 4) print(p.x, p.y) # 3 4 -- access by name, not just position print(p[0]) # 3 -- still works like a regular tuple too
Quick Check — Tuples
Q. Can a tuple ever be unhashable?
A. Yes — if it contains a mutable element, like (1, [2, 3]), the tuple itself becomes unhashable, because Python can't guarantee the hash would stay valid if that inner list changes.
Q. What does a, b = b, a do, and what concept enables it?
A. Swaps the values of a and b — Python builds a temporary tuple (b, a) on the right, then unpacks it into a, b on the left, in one step.
Interview Trap — The single-element tuple

(5) is just the integer 5 — parentheses alone don't make a tuple. You need a trailing comma: (5,). This trips up almost everyone the first time.

Structure 3

Sets

A set is an unordered collection of unique items — no duplicates allowed, and there's no concept of "position" or indexing. Think of it like a bag: you can check "is this in the bag?" instantly, but you can't ask "what's in slot 2?"

nums = {1, 2, 2, 3, 3, 3} print(nums) # {1, 2, 3} -- duplicates are automatically dropped nums.add(4) # adds a single item print(3 in nums) # True -- membership check, very fast (O(1) average)

Set operations

a = {1, 2, 3} b = {2, 3, 4} print(a | b) # union: {1, 2, 3, 4} -- everything in either print(a & b) # intersection: {2, 3} -- only in both print(a - b) # difference: {1} -- in a but not in b print(a ^ b) # symmetric difference: {1, 4} -- in exactly one, not both

frozenset — the immutable version

A frozenset behaves exactly like a set but can't be changed after creation — which makes it hashable, so (unlike a regular set) it can be placed inside another set or used as a dictionary key.

💡

The most common real-world use of sets in data work: deduplication (set(my_list) instantly removes duplicates) and fast membership testing — checking "have I seen this ID before?" in a set is O(1) on average, versus O(n) for a list.

Quick Check — Sets
Q. Can you put a list inside a set?
A. No — TypeError: unhashable type: 'list'. Lists are mutable, so they can't be hashed, and sets require hashable elements.
Q. What does set([3, 1, 2, 1]) return, and does it preserve the original order?
A. {1, 2, 3} — sets don't preserve insertion order or guarantee any particular ordering; never rely on the order you'll see when printing a set.
Interview Trap — Empty set syntax

{} creates an empty dictionary, not an empty set — a very common beginner mistake. You must write set() explicitly to get an empty set.

Structure 4

Dictionaries

A dictionary stores key-value pairs — instead of looking things up by position (like a list), you look them up by a unique key, similar to a real dictionary where you look up a word to find its definition.

person = {"name": "Alice", "age": 30} print(person["name"]) # "Alice" -- direct access; KeyError if the key doesn't exist print(person.get("email")) # None -- .get() returns None (or a default) instead of erroring print(person.get("email", "N/A")) # "N/A" -- custom fallback value person["email"] = "a@x.com" # adds a new key, or overwrites if it already exists person.setdefault("age", 99) # does NOTHING here -- "age" already exists, keeps 30

Iterating over a dict

for key in person: # iterates over KEYS only, by default print(key) for key, value in person.items(): # .items() gives you BOTH key and value together print(key, value)

Dict comprehensions

squares = {x: x**2 for x in range(5)} # {0:0, 1:1, 2:4, 3:9, 4:16}
💡

Since Python 3.7, dictionaries officially preserve insertion order (the order you added keys). Before that, order wasn't guaranteed — many people still mistakenly think dicts are always "unordered," so this is a common interview clarification point.

Quick Check — Dictionaries
Q. What's the difference between person["email"] and person.get("email") if the key doesn't exist?
A. person["email"] raises a KeyError; person.get("email") returns None (or a custom default you supply) without crashing.
Q. Can a dictionary key be a list?
A. No — same hashability rule as sets. Keys must be hashable (immutable), so lists are rejected but tuples are fine.
Interview Trap — Iterating and mutating at the same time

Never add or remove keys from a dictionary while directly looping over it (for k in d: del d[k]) — this raises RuntimeError: dictionary changed size during iteration. Fix: loop over list(d.keys()) instead, which makes a separate list first.

Why "in" is slow on a list but fast on a set/dict
Checking x in my_list must scan item by item until it finds a match (or reaches the end) — O(n). Checking x in my_set or x in my_dict computes a hash and jumps almost directly to where it would be — O(1) on average. If you're checking membership repeatedly in a loop, converting a list to a set first is a classic optimization.
Power Tools

The collections Module

Python's built-in collections module has specialized versions of lists/dicts that solve extremely common interview patterns.

Counter — counting things

from collections import Counter words = ["apple", "banana", "apple", "cherry", "apple"] counts = Counter(words) print(counts) # Counter({'apple': 3, 'banana': 1, 'cherry': 1}) print(counts.most_common(2)) # [('apple', 3), ('banana', 1)] -- top 2 by count

defaultdict — dictionaries with automatic default values

from collections import defaultdict groups = defaultdict(list) # if a key doesn't exist, auto-create it as an empty list groups["fruits"].append("apple") # no KeyError, even though "fruits" didn't exist yet print(groups) # defaultdict(, {'fruits': ['apple']})

deque — fast additions/removals from both ends

from collections import deque dq = deque([1, 2, 3]) dq.appendleft(0) # O(1) -- adding to the front of a normal list would be O(n) dq.append(4) dq.popleft() # O(1) removal from the front
💡

Use a regular list when you mostly add/remove from the END. Use a deque when you need fast operations at BOTH ends (e.g. implementing a queue, a sliding window, or BFS in a graph problem).

Quick Reference

Glossary of Every Term Used

mutable
Can be changed in place after creation (lists, sets, dicts).
immutable
Cannot be changed after creation; any "change" produces a new object (tuples, strings, ints).
hashable
Has a hash value that never changes during its lifetime — required to be a dict key or set member. Generally true only for immutable types.
hash / hash table
A fixed-size number computed from a value, used to jump almost directly to where that value is stored — the mechanism behind O(1) set/dict lookups.
slicing
Extracting a sub-portion of a sequence with [start:stop:step] notation; start is included, stop is excluded.
unpacking
Assigning multiple variables at once from a tuple/list, e.g. x, y = (1, 2).
comprehension
A compact one-line syntax for building a new list/dict/set from an existing iterable, e.g. [x**2 for x in range(5)].
aliasing
When two variable names point to the exact same object in memory, so changing one appears to change the other.
shallow copy
Copies the outer container but nested mutable objects inside are still shared with the original.
deep copy
Recursively copies everything, including nested objects, so the copy is fully independent.
O(1) / O(n)
Big-O notation for how an operation's time grows with input size — O(1) means constant time (doesn't grow), O(n) means it grows proportionally with the number of items.
Practice

50 Interview Questions

Conceptual Questions (Explain in Words)

C1Easy
What's the difference between a list and a tuple?
+
Answer

Lists are mutable and use []; tuples are immutable and use (). Both are ordered, but only tuples (if their contents are hashable) can be used as dict keys or set members.

C2Easy
What's the difference between a set and a list?
+
Answer

A set holds only unique values and has no ordering or indexing, while a list allows duplicates and preserves order/position. Membership checks are much faster on a set (O(1) average) than a list (O(n)).

C3Easy
What does it mean for something to be "hashable"?
+
Answer

It has a hash value (a number Python computes from it) that never changes during its lifetime — required for use as a dict key or set element. This is only reliable for immutable objects.

C4Easy
Why can't you use a list as a dictionary key?
+
Answer

Lists are mutable, so they're unhashable — Python can't guarantee the key's hash stays the same if the list's contents change later, which would break the dictionary's internal lookup.

C5Easy
What's the difference between dict.get() and dict[key]?
+
Answer

dict[key] raises a KeyError if the key is missing; dict.get(key) returns None (or a supplied default) instead, without crashing.

C6Medium
Are Python dictionaries ordered? Explain the historical nuance.
+
Answer

Since Python 3.7, dicts officially preserve insertion order. Before that (3.6 and earlier), order was an implementation detail (worked in CPython 3.6 but wasn't guaranteed by the language spec). Many people still assume dicts are inherently unordered.

C7Medium
What's the difference between a shallow copy and a deep copy of a list of lists?
+
Answer

A shallow copy duplicates the outer list, but the inner lists are still the same shared objects as the original — mutating a nested list affects both. A deep copy recursively duplicates every nested list too, giving full independence.

C8Medium
Why is membership testing (x in ...) faster on a set than a list?
+
Answer

A set uses a hash table — it computes a hash of x and jumps almost directly to where it would be stored, O(1) on average. A list has no such shortcut and must scan sequentially until it finds a match or exhausts the list, O(n).

C9Medium
What's a named tuple, and why would you use one over a plain tuple?
+
Answer

A named tuple lets you access fields by name (point.x) instead of only by position (point[0]), improving readability, while still behaving like a regular immutable tuple underneath.

C10Medium
What's the difference between list.sort() and sorted(list)?
+
Answer

list.sort() sorts in place and returns None; sorted(list) returns a brand new sorted list and leaves the original unchanged. sorted() also works on any iterable, not just lists.

C11Medium
What is a frozenset, and when would you use one?
+
Answer

A frozenset is an immutable version of a set — same operations (union, intersection) but can't be modified after creation, which makes it hashable and therefore usable as a dict key or nested inside another set.

C12Medium
What does aliasing mean in Python, and why does it matter for mutable objects?
+
Answer

Aliasing is when two variable names refer to the exact same underlying object. For mutable objects (like lists), changes made through one name are visible through the other — surprising if you assumed b = a created an independent copy.

C13Hard
Explain why average-case dict/set lookup is O(1) but worst-case can degrade to O(n).
+
Answer

Hash tables map keys to "buckets" based on their hash. If many keys hash to the same bucket (a collision), Python must fall back to scanning within that bucket — with enough collisions, lookup degrades toward O(n). This is rare in practice due to Python's hash function quality and automatic resizing.

C14Hard
Why does defining __eq__ on a custom class without __hash__ break its use in a set?
+
Answer

Python requires that two objects considered equal (__eq__) must also hash identically. Defining __eq__ alone makes Python set __hash__ to None by default, making the object unhashable — so it can't go in a set until you also define a matching __hash__.

C15Hard
When would you choose a deque over a list?
+
Answer

When you need fast additions/removals from BOTH ends of the sequence — a deque does this in O(1), while a list is O(1) only at the end and O(n) at the front (since every other element must shift). Common in queue implementations, sliding window problems, and BFS.

C16Hard
What's the practical difference between using a Counter and manually building a dict of counts?
+
Answer

Functionally similar, but Counter avoids KeyError boilerplate (missing keys default to 0), and provides convenience methods like .most_common(n), arithmetic between counters (+, -), which a plain dict doesn't offer out of the box.

C17Hard
Why does list concatenation with + inside a loop become a performance problem?
+
Answer

Each + creates an entirely new list and copies all existing elements into it — O(n) per operation. Doing this n times in a loop becomes O(n²) overall. Using .append() in a loop (or a list comprehension) is O(n) total, since it grows the same list in place.

C18Hard
How would you deduplicate a list while preserving the original order (sets alone don't do this)?
+
Answer

Use a dict (or dict.fromkeys()) instead of a set, since dicts preserve insertion order in modern Python: list(dict.fromkeys(my_list)) removes duplicates while keeping the first-seen order, unlike set(my_list) which loses ordering.

C19Hard
Why can tuples sometimes be mutable "in effect" even though they're immutable?
+
Answer

A tuple itself can't be changed (you can't reassign its slots), but if it contains a mutable object like a list, that inner list can still be mutated in place — t = (1, [2, 3]); t[1].append(4) works fine, even though t is a tuple.

C20Hard
How would you explain choosing the right data structure to a non-technical stakeholder?
+
Answer

"If order and duplicates matter, and I'll change it a lot — list. If it should never change — tuple. If I just need to know 'have I seen this before' quickly, with no duplicates — set. If I need to look things up by a name or ID instead of position — dictionary."

T21Hard
Given a list of numbers, find two numbers that sum to a target, in O(n) using a dict (the "two-sum" problem).
+
Answer
def two_sum(nums, target): seen = {} for i, n in enumerate(nums): complement = target - n if complement in seen: return [seen[complement], i] seen[n] = i return None print(two_sum([2, 7, 11, 15], 9)) # [0, 1]

The dict lets you check "have I seen the complement?" in O(1) instead of nested loops (O(n²)).

T22Hard
Implement a sliding window using a deque to find the maximum of every k-sized window in a list.
+
Answer
from collections import deque def max_sliding_window(nums, k): dq = deque() # stores INDICES, kept in decreasing value order result = [] for i, n in enumerate(nums): while dq and nums[dq[-1]] < n: dq.pop() dq.append(i) if dq[0] <= i - k: dq.popleft() if i >= k - 1: result.append(nums[dq[0]]) return result
T23Hard

Answer

The first line prints a valid hash. The second raises TypeError: unhashable type: 'list' — a tuple is only hashable if ALL of its elements are hashable; containing a list breaks that.

T25Hard

T26Hard
Implement a LRU-cache-like structure's core idea using an OrderedDict.
+
Answer
from collections import OrderedDict class LRUCache: def __init__(self, capacity): self.cache = OrderedDict() self.capacity = capacity def get(self, key): if key not in self.cache: return -1 self.cache.move_to_end(key) # mark as recently used return self.cache[key] def put(self, key, value): if key in self.cache: self.cache.move_to_end(key) self.cache[key] = value if len(self.cache) > self.capacity: self.cache.popitem(last=False) # evict the LEAST recently used

Implement set-based deduplication that also tracks which items were duplicates.

[[1, 0, 0], [1, 0, 0], [1, 0, 0]] — [[0]*3]*3 creates THREE REFERENCES to the very same inner list, not three independent lists. Fix: [[0]*3 for _ in range(3)], which creates a genuinely new list on each iteration.
