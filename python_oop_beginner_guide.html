What is OOP, really?

OOP (Object-Oriented Programming) is just a way of organizing code around "things" (objects) instead of a long list of steps. Think of a blueprint for a house — the blueprint itself isn't a house you can live in, but you can use it to build as many actual houses as you want, each with its own address, paint color, and furniture.

In Python: the blueprint is called a class, and each actual house built from it is called an object (or an instance). This matters for data science because almost every library you use — scikit-learn models, pandas DataFrames, PyTorch layers — is built this way.

💡

Every time you write model = LinearRegression(), you're building one "house" (object) from the "blueprint" (class) called LinearRegression. That's it — that's OOP in one sentence.

Building Block 1

Classes & Objects

class Dog: # "class" keyword = start a blueprint named Dog def __init__(self, name, age): # special "constructor" method — runs when object is created self.name = name # self.name = "this specific dog's name" self.age = age def bark(self): # a regular method (a function that belongs to the class) return f"{self.name} says woof!" my_dog = Dog("Rex", 3) # building an OBJECT from the CLASS print(my_dog.bark()) # "Rex says woof!"

What does every piece mean?

  • class — the keyword that says "I'm defining a blueprint," followed by the name you choose (here, Dog)
  • __init__ — pronounced "dunder init" (double underscore + init + double underscore). It's a special method Python calls automatically the moment you create a new object. Think of it as the "setup" step.
  • self — refers to "this specific object." When you later call my_dog.bark(), Python quietly passes my_dog in as self. It's how each object keeps track of its own data separately.
  • my_dog = Dog("Rex", 3) — this line actually builds the object. Dog is the blueprint; my_dog is the real thing built from it.
💡

You never call __init__ yourself — writing Dog("Rex", 3) triggers it automatically. self is always the first parameter of any method inside a class, and Python fills it in for you — you never pass it manually.

Quick Check — Classes & Objects
Q. If you create two dogs, Dog("Rex", 3) and Dog("Fido", 5), do they share the same name value?
A. No — each object has its own self.name, stored separately. That's the whole point of self: it keeps each object's data independent.
Q. What would happen if you forgot to include self as the first parameter in bark(self)?
A. Calling my_dog.bark() would raise a TypeError — Python automatically passes the object as the first argument, so the method must have a parameter to receive it.
Interview Trap

Beginners often think self is a reserved keyword like class or def. It isn't — it's just a naming convention everyone follows. You could technically name it anything, but never do this in real code; interviewers may ask "what if you renamed self?" to test if you understand it's positional, not magic.

Building Block 2

Encapsulation — Protecting Data

Encapsulation means keeping an object's internal data safe from accidental, uncontrolled changes — bundling the data together with the rules for how it's allowed to change. Imagine a bank account: you shouldn't be able to just set someone's balance to a random number without going through a deposit/withdraw process that checks it's valid.

Naming conventions (Python's version of "private")

StyleMeaning
balancePublic — anyone can read/change it freely
_balance"Protected" by convention only — a signal to other devs: "please don't touch this directly," but Python doesn't stop you
__balance"Private" — Python secretly renames it internally, making accidental access harder (but not impossible)

@property and @x.setter — explained slowly

First: what is a decorator? A decorator is something written as @word placed directly above a function/method. It doesn't call the function — it wraps the function with extra behavior. Think of it like a gift-wrapper: the gift (your function) stays the same, but the wrapper changes how you interact with it.

class BankAccount: def __init__(self, balance): self._balance = balance # store the real value in a "protected" variable @property # decorator #1: turns the method below into a "getter" def balance(self): # now accessed WITHOUT parentheses: acc.balance return self._balance @balance.setter # decorator #2: turns this method into a "setter" def balance(self, value): # runs whenever someone does acc.balance = value if value < 0: raise ValueError("Balance cannot be negative") self._balance = value acc = BankAccount(100) acc.balance = 50 # looks like plain attribute access, but SECRETLY calls the setter method print(acc.balance) # looks like plain attribute access, but SECRETLY calls the getter method

What does each piece mean?

  • @property — placed above a method named e.g. balance. It lets you call acc.balance (no parentheses!) even though it's really running a method behind the scenes. This is the "getter."
  • @balance.setter — placed above a second method, also named balance. It's what runs when someone writes acc.balance = 50 (an assignment). This is the "setter," and it's where you'd add validation (like blocking negative numbers).
  • Why bother? Because it lets you add safety checks later without breaking code that already uses acc.balance — the outside world never needs to know it's secretly a method call.
Quick Check — Encapsulation
Q. Is __balance (double underscore) truly impossible to access from outside the class?
A. No — Python renames it to _BankAccount__balance internally (called "name mangling"), but you can still access it if you know that renamed version. It discourages accidental access, it doesn't enforce true privacy.
Q. What happens if you write acc.balance = -50 given the code above?
A. It raises a ValueError, because assigning to acc.balance secretly calls the @balance.setter method, which checks value < 0 before storing it.
Interview Trap

Both the getter and setter methods must have the exact same name (here, balance) — only their decorators (@property vs @balance.setter) differ. Beginners often accidentally name them differently and get a confusing error.

Building Block 3

Inheritance — Reusing a Blueprint

Inheritance lets one class "borrow" everything from another class, then add or change only what's different. Think of it like a general "Vehicle" blueprint, and a more specific "Car" blueprint that reuses everything from Vehicle but adds car-only details.

class Animal: # the PARENT (a.k.a base/super) class def __init__(self, name): self.name = name def speak(self): return "..." class Dog(Animal): # the CHILD (a.k.a subclass) — (Animal) means "inherits from Animal" def speak(self): # OVERRIDING — replacing the parent's version return f"{self.name} says Woof!" class Cat(Animal): def __init__(self, name, indoor): super().__init__(name) # calls Animal's __init__ to set self.name, so we don't repeat that code self.indoor = indoor def speak(self): return f"{self.name} says Meow!" d = Dog("Rex") print(d.speak()) # "Rex says Woof!"

What does each piece mean?

  • class Dog(Animal): — the (Animal) part is what actually creates the inheritance — Dog now automatically has everything Animal has
  • super() — means "go run the parent class's version of this method." super().__init__(name) literally calls Animal.__init__ so you don't have to retype that logic
  • overriding — when a child class defines a method with the same name as the parent, the child's version "wins" and replaces the parent's behavior for that class
💡

Why use super().__init__(name) instead of just writing self.name = name again in Cat? Because if Animal's setup logic ever changes, every child class using super() automatically gets the update — you're not duplicating code.

Quick Check — Inheritance
Q. If Cat didn't define its own speak() method at all, what would Cat("Tom", True).speak() return?
A. "..." — it would fall back to Animal's version, since Cat doesn't override it. This "fallback" behavior is the core benefit of inheritance.
Q. What would happen if Cat's __init__ forgot to call super().__init__(name)?
A. self.name would never get set by Animal's logic, so accessing cat.name later would raise an AttributeError (unless Cat sets it manually itself).
Interview Trap

super() is not "just a way to refer to the parent by name" — you could technically write Animal.__init__(self, name) instead, but that hardcodes the parent's name. If the class hierarchy changes later, super() automatically adapts; a hardcoded parent name won't.

Building Block 4

Polymorphism — Same Name, Different Behavior

"Poly" = many, "morph" = forms. Polymorphism just means: the same method name can behave differently depending on which object calls it. You saw this already above — .speak() did something different for Dog vs Cat.

for animal in [Dog("Rex"), Cat("Tom", True)]: print(animal.speak()) # same line of code, different output each time

Duck typing — Python's relaxed version

Python doesn't check "is this really an Animal?" before calling .speak() — it just checks "does this object have a .speak() method?" This is called duck typing: "if it walks like a duck and quacks like a duck, treat it like a duck" — Python doesn't care about the object's official type, only whether it has the method you're trying to call.

💡

This is why len(my_list), len(my_string), and len(my_dict) all work with the same len() function — each of those types secretly defines its own version of a method Python calls behind the scenes (__len__, covered in Magic Methods below).

Quick Check — Polymorphism
Q. Does the object in the for-loop above need to officially inherit from Animal for duck typing to work?
A. No — any object with a .speak() method works in that loop, even a totally unrelated class. Python only checks that the method exists at the moment it's called.
Q. Is polymorphism the same thing as method overriding?
A. Overriding is one common WAY to achieve polymorphism (via inheritance), but polymorphism is the broader concept — duck typing achieves it without any inheritance at all.
Interview Trap

Don't confuse polymorphism with function overloading (having multiple versions of a function with different argument counts/types, like in Java/C++). Python doesn't really support that — if you define the same method name twice in one class, the second definition simply replaces the first.

Building Block 5

Abstraction — Enforcing a Contract

Abstraction means hiding the messy implementation details and only exposing "here's what you MUST implement." Imagine you're designing a rule: "every model class must have a fit() and a predict() method" — but you don't care exactly how each one does it.

from abc import ABC, abstractmethod # importing tools for abstraction class Model(ABC): # ABC = "Abstract Base Class" — a special kind of class @abstractmethod # decorator: marks this method as "required, no default behavior" def fit(self, X, y): ... # "..." is just a placeholder meaning "no code here" @abstractmethod def predict(self, X): ... class LinearModel(Model): # must implement BOTH fit and predict, or Python will error def fit(self, X, y): return self def predict(self, X): return X # Model() --> TypeError! Can't create an object directly from an abstract class lm = LinearModel() # works fine — all required methods are filled in

What does each piece mean?

  • ABC — short for "Abstract Base Class," imported from Python's built-in abc module. Inheriting from it turns your class into a "rulebook" that can't be built directly.
  • @abstractmethod — a decorator that marks a method as "required." Any class inheriting from this ABC MUST provide its own version, or Python refuses to let you create an object from it.
Quick Check — Abstraction
Q. What happens if LinearModel only implements fit() but not predict()?
A. Trying to create LinearModel() raises a TypeError, because one required abstract method is still missing.
Q. What if a class defines a method with @abstractmethod but forgets to inherit from ABC?
A. Nothing is enforced — the decorator has no effect without ABC, and the class instantiates normally even with "abstract" methods left unimplemented.
Interview Trap

Just adding @abstractmethod above a method does nothing on its own — it only works because the class also inherits from ABC. This combination is easy to forget half of.

Building Block 6

Magic (Dunder) Methods

"Dunder" = Double UNDERscore. Methods like __init__, __str__, __len__ are called "magic methods" because Python calls them automatically behind the scenes in response to normal-looking code (like print(), len(), or +) — you rarely call them directly yourself.

MethodTriggered by
__init__Creating an object, e.g. Dog("Rex")
__str__print(obj) or str(obj) — human-readable text
__repr__Typing obj in a Python console — developer-facing text
__len__len(obj)
__eq__obj1 == obj2
__add__obj1 + obj2
class Vector: def __init__(self, x, y): self.x, self.y = x, y def __add__(self, other): # runs automatically when you write v1 + v2 return Vector(self.x + other.x, self.y + other.y) def __repr__(self): # runs automatically when Python needs to display this object return f"Vector({self.x}, {self.y})" v1, v2 = Vector(1, 2), Vector(3, 4) print(v1 + v2) # Vector(4, 6) — the "+" secretly calls __add__
Quick Check — Magic Methods
Q. If you remove __repr__ entirely, what would print(v1) show instead?
A. Something unhelpful like <__main__.Vector object at 0x7f...> — Python's default representation, just a memory address.
Q. Do you ever call __add__ directly, like v1.__add__(v2)?
A. You technically can, and it works — but it's not idiomatic. You're meant to just write v1 + v2 and let Python call it for you.
Interview Trap

Confusing __str__ and __repr__: if only __repr__ is defined, Python uses it as a fallback for both print() and the console. But if only __str__ is defined, typing the object directly in a console still shows the default ugly memory-address text — __str__ is NOT used as a fallback for __repr__.

Building Block 7

Instance vs @classmethod vs @staticmethod

Not every method needs access to self (the specific object). Python gives you three flavors:

class Employee: raise_pct = 1.05 # shared by all employees def __init__(self, salary): self.salary = salary def give_raise(self): # INSTANCE method — needs self, touches one object's data self.salary *= Employee.raise_pct @classmethod # decorator: marks this as a classmethod def from_string(cls, data_str): # "cls" = the class itself, not one object salary = float(data_str) return cls(salary) # same as writing Employee(salary) @staticmethod # decorator: marks this as a staticmethod def is_valid_salary(salary): # no self, no cls — just a helper function living inside the class return salary > 0
TypeFirst argPlain-English use case
instance methodself"Do something to THIS specific object"
@classmethodcls"Build a new object in an alternative way" (e.g. from a string, a file, a dict)
@staticmethodnone"A helper function that's related but doesn't need any object/class data"
Quick Check — Method Types
Q. Could is_valid_salary() just as easily be a regular function outside the class?
A. Yes, technically — @staticmethod is mainly about organization: keeping a clearly-related helper grouped with the class it serves, for readability.
Q. Why does from_string use cls(salary) instead of writing Employee(salary) directly?
A. So it still works correctly if a subclass calls it — cls automatically refers to whichever class actually called the method, not always Employee.
Interview Trap

Forgetting the decorator entirely and just adding cls or removing self won't work — the decorator (@classmethod/@staticmethod) is what actually changes how Python calls the method. Without it, Python still tries to pass the object in as the first argument.

Quick Reference

Glossary of Every Symbol Used

self
Refers to "this specific object." Always the first parameter in instance methods; Python fills it in automatically.
cls
Refers to "the class itself" (not one object). Used as the first parameter in @classmethods.
@decorator
Any line starting with @ directly above a method/function. It wraps that method with extra behavior without changing its core code.
__init__
The "constructor" — runs automatically when an object is created, used to set up initial attributes.
dunder
Short for "Double UNDERscore." Refers to methods like __init__, __str__, wrapped in double underscores on both sides.
super()
A way to call the parent class's version of a method, most often used inside __init__ to reuse the parent's setup logic.
ABC
"Abstract Base Class" — a special class (from Python's abc module) that can't be instantiated directly; it exists only to be inherited from.
@abstractmethod
Marks a method as "required" in an ABC — subclasses must implement it or they can't be created.
@property
Turns a method into something accessed like a plain attribute (no parentheses) — the "getter."
@x.setter
Paired with @property; defines what runs when you assign a new value to that "attribute" — the "setter."
instance
One specific object built from a class — e.g. my_dog is an instance of Dog.
override
When a child class redefines a method that already existed in its parent class, replacing that behavior.

50 Interview Questions

Conceptual Questions (Explain in Words)

C1Easy
What is a class, and what is an object? How are they related?
+
Answer

A class is a blueprint describing what attributes and behaviors something will have. An object is one actual "thing" built from that blueprint — you can build many objects from a single class, each with its own data.

C2Easy
What is self, and why does every instance method need it as the first parameter?
+
Answer

self refers to the specific object the method is being called on. Python automatically passes the calling object in as the first argument, so the method needs a parameter to receive it — otherwise it has no way to know which object's data to use.

C3Easy
What does the __init__ method do?
+
Answer

It's the constructor — a special method Python calls automatically right after an object is created, typically used to set up the object's initial attributes.

C4Easy
What are the four pillars of OOP?
+
Answer
  • Encapsulation — bundling data with rules for changing it
  • Inheritance — reusing/extending another class's behavior
  • Polymorphism — same method name behaves differently per object
  • Abstraction — hiding implementation, exposing only a required interface
C5Easy
What's the difference between a class attribute and an instance attribute?
+
Answer

A class attribute is defined directly under the class and shared by every object of that class. An instance attribute is set inside __init__ (usually via self.x = ...) and belongs uniquely to that one object.

C6Medium
Is anything truly "private" in Python? Explain how __name mangling works.
+
Answer

No — Python has no enforced privacy. A __name attribute gets renamed internally to _ClassName__name, which mostly prevents accidental collisions in subclasses rather than blocking outside access entirely.

C7Medium
Why would you use @property instead of just letting people access a plain attribute directly?
+
Answer

It lets you add validation or computed logic later without changing how the attribute is accessed from outside — code using obj.value keeps working exactly the same, even though it's secretly running a method now.

C8Medium
What is method overriding, and how is it different from Python not supporting true method overloading?
+
Answer

Overriding is a subclass replacing a parent's method with the same name. Overloading (same name, different argument signatures, picked by the caller's arguments) doesn't really exist in Python — defining the same method twice just replaces the earlier definition.

C9Medium
Explain duck typing in your own words.
+
Answer

Python doesn't check an object's declared type before calling a method on it — it just checks whether that method exists. "If it walks like a duck and quacks like a duck, treat it like a duck."

C10Medium
What's the point of an abstract base class if you can't even create an object from it?
+
Answer

It defines a required contract/interface — guaranteeing that every subclass implements certain methods, which lets other code rely on those methods existing without knowing the subclass's exact implementation.

C11Medium
What does super() do, in plain English?
+
Answer

It gives you access to the parent class's version of a method, most commonly used to reuse the parent's __init__ setup logic instead of duplicating it in the child class.

C12Medium
What's a "dunder" method, and why does Python use double underscores for these?
+
Answer

"Dunder" = double underscore. These methods (like __init__, __str__) are called automatically by Python in response to built-in syntax (print(), +, len()). The unusual naming makes them visually distinct from regular methods you'd call yourself.

C13Hard
What is composition, and why might you prefer it over inheritance?
+
Answer

Composition means a class holds other objects as attributes and delegates work to them, instead of inheriting their behavior. It's often preferred because it avoids deep, fragile class hierarchies — you can swap out one "part" without affecting the rest.

C14Hard
What is the diamond problem, and how does Python resolve it?
+
Answer

It happens when a class inherits from two parents that both inherit from the same grandparent, creating ambiguity about which path to follow for a shared method. Python resolves it with the MRO (Method Resolution Order) — a deterministic, predictable search order you can check via ClassName.__mro__.

C15Hard
What's the difference between an ABC and a Protocol (structural typing)?
+
Answer

An ABC requires explicit inheritance — the subclass must formally say "I am a subclass of this." A Protocol just requires having the right methods, with no inheritance needed — it's checked structurally (by static type checkers), not through the class hierarchy.

C16Hard
Why does defining __eq__ without also defining __hash__ cause problems?
+
Answer

Python expects that equal objects must hash identically. Overriding __eq__ alone makes the object unhashable by default (Python sets __hash__ to None), so it can no longer be used as a dictionary key or set member unless you also define __hash__.

C17Hard
What is a mixin class?
+
Answer

A small class designed to be combined with others via multiple inheritance to add one specific piece of reusable behavior — not meant to be used standalone. Scikit-learn's TransformerMixin is a real example, adding a free fit_transform.

C18Hard
Why must a custom scikit-learn-style fit() method return self?
+
Answer

Sklearn relies on chaining calls like model.fit(X, y).predict(X), and pipelines depend on fit handing back the fitted object itself. If it returns None instead, that chaining breaks.

C19Hard
What is a metaclass, in simple terms?
+
Answer

A metaclass is "the class of a class" — it controls how classes themselves get created, the same way a class controls how objects get created. Most Python code never needs to touch this directly.

C20Hard
Explain "favor composition over inheritance" as if to a non-technical stakeholder.
+
Answer

"Instead of building one giant class that tries to inherit everything from many parents, we build small independent pieces and plug them together — like Lego blocks instead of one complicated carved statue. Easier to swap and test individual pieces."
