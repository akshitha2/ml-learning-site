QUESTION 01 Correct

You have a logistic regression model with 95% accuracy on a fraud dataset where 2% of transactions are fraud. Is this a good model? Justify with what you'd actually look at instead.

Must include

State that a "predict majority class always" baseline already scores ~98% accuracy here — so 95% is actually worse than doing nothing. Then name PR-AUC (not ROC-AUC) as the better metric for imbalanced classes.

Elaborated answer

With 2% positive class, a trivial "always predict negative" model scores 98% accuracy. So 95% accuracy here is worse than a model that does nothing — accuracy is actively misleading, not just uninformative.

Instead, use metrics that decompose performance by class: precision, recall, F1, and specifically PR-AUC over the full threshold range. PR-AUC is preferred over ROC-AUC here because its axes (precision, recall) never involve the true negative count, so it isn't diluted by the size of the majority class — see Q2 for the mechanism.

QUESTION 02 Repeated miss

Why does ROC-AUC give a misleadingly optimistic picture on imbalanced data, but PR-AUC doesn't? Given 10,000 negatives, 200 positives, 50 false positives, and 100 of 200 frauds caught — compute FPR and precision, and say which is more alarming.

Must include

FPR = FP / (FP + TN) — not TNR. With TP=100, FN=100, FP=50, TN=10000: FPR = 50/10050 ≈ 0.005, Precision = TP/(TP+FP) = 100/150 ≈ 0.67. The tiny FPR is what makes ROC-AUC deceptively optimistic; the 0.67 precision is the number that actually exposes the model's weakness.

Elaborated answer

ROC's x-axis is FPR = FP/(FP+TN). When TN is huge (majority class), FPR stays small even if the model produces many false positives in absolute terms — the huge denominator dilutes the effect. That's why ROC-AUC looks great on imbalanced data even for a mediocre model: it's measuring something the negative class overwhelms.

Precision's denominator is TP+FP — it never touches TN, so it isn't inflated by class size. Precision = 0.67 correctly signals "1 in 3 fraud alerts is a false alarm," which has direct business cost (analyst time), while FPR = 0.005 hides that same weakness. This is exactly why PR-AUC surfaces problems ROC-AUC conceals under imbalance.

QUESTION 03 Partial

Explain the fundamental difference in how Random Forest and XGBoost combine trees, and why GBM is more prone to overfitting than Random Forest despite both being tree ensembles.

Must include

Bagging = parallel, independent trees averaged → reduces variance, bias roughly unchanged. Boosting = sequential, fits residuals → reduces bias, but at the cost of increasing variance/overfitting risk if unregularized.

Elaborated answer

Random Forest trains many trees independently on bootstrap samples and averages predictions — this is variance reduction: Var(average) = ρσ² + (1−ρ)σ²/n, where ρ is inter-tree correlation. Bias is essentially unchanged because each tree's expectation ≈ the expectation of a tree trained on the full data; averaging doesn't shift that expectation, only shrinks the spread around it.

XGBoost trains trees sequentially, each one fitting the residual error of the ensemble so far. This directly attacks bias — training error keeps dropping — but each new tree is free to chase noise/outliers in the residuals just as eagerly as real signal, which increases variance. That's why boosting needs explicit regularization (learning rate/shrinkage, max_depth, early stopping) to keep the bias-reduction machinery from overfitting.

QUESTION 04 Repeated miss, then corrected

Name two XGBoost hyperparameters that control overfitting and explain their mechanism (not just "it regularizes"). What is gamma's exact role, and how does reg_lambda differ from reg_alpha?

Must include

gamma = minimum loss reduction required before a split is allowed (regularizes tree structure) — it is not L1. reg_alpha (L1) shrinks leaf weights to exactly zero (sparse). reg_lambda (L2) shrinks leaf weights smoothly toward zero without eliminating them.

Elaborated answer

gamma is a complexity penalty on splitting: a candidate split is only made if its loss-reduction (gain) exceeds gamma; otherwise the split is pruned. It regularizes the tree's structure/depth, not the leaf output values.

reg_alpha (L1) penalizes the leaf weight magnitudes with an absolute-value penalty, which can drive small weights exactly to zero — sparse leaf outputs. reg_lambda (L2) penalizes squared leaf weights, producing proportional shrinkage that gets small but never hits exactly zero.

QUESTION 05 Partial

Why do we initialize NN weights randomly instead of all zero? Be specific about the failure mode.

Must include

All-zero weights means every neuron in a layer receives identical gradients at every step (same weights + same input → same output → same backprop signal), so they stay identical forever — it's an invariant from step 1, not something they "eventually converge to."

Elaborated answer

If all weights in a layer start at zero, every neuron in that layer computes the exact same function of the input. Because they're identical, they receive the exact same gradient during backprop, so their updates are identical too — this holds at every subsequent step, not just at initialization. The layer is permanently stuck behaving like a single neuron regardless of how many neurons it has, since nothing ever breaks the symmetry.

QUESTION 06 Repeated miss, answer given

Does the symmetry-breaking problem apply to biases too? Can biases be safely zero-initialized while weights cannot?

Must include

Yes, biases can safely be zero — because once weights are randomly initialized, neurons already receive different inputs to their activations and therefore different gradients, regardless of bias value. Symmetry is already broken by the weights; bias has nothing left to symmetrize.

Elaborated answer

Symmetry-breaking requires neurons to be identical in every parameter feeding their computation. With random weight init, neuron A and neuron B already have different incoming weight vectors, so they compute different pre-activations and receive different gradients — independent of whether their biases are both zero. This is why standard init schemes (Xavier, He) routinely set biases to zero while randomizing only the weights; it's standard practice, not a workaround.

QUESTION 07 Repeated miss, answer given

Explain geometrically why L1 regularization produces sparse (exact-zero) solutions but L2 doesn't.

Must include

L2's constraint region is a smooth circle/sphere — no corners, so the loss-contour tangent point generically lands where no coordinate is exactly zero. L1's constraint region is a diamond with corners sitting exactly on the axes — loss contours are much more likely to first touch at a corner, which is precisely a point where a coordinate equals zero.

Elaborated answer

Regularized regression can be framed as minimizing loss subject to a constraint on weight norm. L2's constraint ‖w‖₂ ≤ t is a sphere — smooth everywhere, so the point where the loss's elliptical contours first touch it can be anywhere on the surface, generically off-axis (no zero coordinates).

L1's constraint ‖w‖₁ ≤ t is a diamond with sharp vertices exactly on the coordinate axes. Loss contours are disproportionately likely to first touch at one of these corners, because corners protrude — and touching a corner means one or more coordinates are exactly zero. That's the geometric source of sparsity.

Algebraically: L2's gradient contribution is −λw (proportional to current weight) → multiplicative shrinkage that approaches but never reaches zero. L1's gradient contribution is −λ·sign(w) (constant magnitude regardless of w) → a fixed-size step that can overshoot past zero and gets clipped exactly to zero, and stays there.

QUESTION 08 Off-topic answer

Why does increasing k in k-fold CV reduce bias in the performance estimate but increase variance? Be specific about what changes in each fold.

Must include

This is about bias/variance of the performance estimate itself, not model overfitting. Small k → less training data per fold → pessimistic bias. Large k → training folds overlap heavily (highly correlated models) and test folds are tiny → noisy, high-variance estimate.

Elaborated answer

Bias: with small k (e.g. 5), each training fold uses 80% of the data — less than the full dataset the final model will use in production, so performance is underestimated (pessimistic bias). Larger k (e.g. 20) uses 95% of data per fold, closer to production conditions, shrinking that bias.

Variance: with large k, test folds are tiny and training folds overlap almost completely across folds — the k models become highly correlated, and each fold's score is noisy due to the small test set. This makes the estimate (the average across folds) unstable across different random splits — higher variance, despite lower bias.

QUESTION 09 Definitions inverted

Explain covariate shift vs concept drift, and give one production-viable detection method for each.

Must include

Covariate shift = P(X) changes, P(Y|X) stays the same. Concept drift = P(Y|X) itself changes. Detect covariate shift with PSI/KS-test on feature distributions; detect concept drift via a proxy (rolling model performance / accuracy decay), since P(Y|X) isn't directly observable without timely labels.

Elaborated answer

Covariate shift: the input distribution changes (e.g., desktop-heavy traffic shifts to mobile-heavy) but the relationship between features and outcome is unchanged. Detect by comparing feature distributions over time — Population Stability Index (PSI) or KS-test per feature between training-time and production data.

Concept drift: the same inputs now map to a different outcome (e.g., COVID changing what "normal" purchase behavior means). You can't monitor P(Y|X) directly without fresh labels, so use a proxy: rolling-window model accuracy/AUC as labels arrive, or monitor prediction confidence shifts and champion–challenger disagreement when labels are delayed.

QUESTION 10 Partial

A churn model uses "average support tickets in the last 30 days" as a feature. What data leakage risk does this introduce, and how do you fix it, even assuming the 30-day window is computed with correct calendar dates?

Must include

This is temporal proximity leakage: ticket spikes right before churn are a consequence of the customer already deciding to leave, not a leading indicator. Fix with an embargo/buffer gap between the feature window and the prediction point (e.g., use days 60–90 to predict churn at day 100+).

Elaborated answer

Even with a correctly computed, no-future-dates window, leakage can occur if the window sits too close to the churn event. Customers who are about to churn often spike support tickets right before leaving — that spike is caused by the same decision you're trying to predict, not a cause that precedes it. A model trained on this essentially memorizes "people who just churned had high tickets," which is true but useless for early prediction.

Fix: introduce a buffer/embargo gap — e.g., compute tickets over days 60–90 and predict churn at day 100+, so the feature reflects genuine leading signal rather than a trailing symptom of the label having already occurred.

QUESTION 11 No attempt

Why does SGD's noisy update sometimes act as an implicit regularizer and help generalization, rather than just being "worse" optimization?

Must include

Full-batch GD follows the exact gradient and can settle into sharp, narrow minima that overfit training noise. SGD's per-batch gradient noise acts as perturbation, biasing the search toward flatter, wider minima, which generalize better because the loss doesn't spike for small train/test distribution shifts.

Elaborated answer

Full-batch GD computes the exact training-loss gradient at every step, letting it converge precisely into sharp minima that fit training data (including its noise) very tightly. SGD's gradient is a noisy estimate — that noise perturbs the trajectory and makes it unlikely to settle in narrow minima (small perturbations would kick it back out), so it tends to land in flat, wide minima instead.

Flat minima generalize better empirically because the loss surface doesn't change sharply for nearby weight values — small shifts between train and test distributions don't cause large loss increases. So SGD's noise isn't merely a speed/approximation trade-off; it actively biases the optimizer toward solutions that generalize better.

QUESTION 12 No attempt

Explain precisely what the "kernel trick" does — not "it maps to higher dimensions," but why it avoids ever computing that mapping explicitly.

Must include

SVM's dual formulation only ever needs dot products ⟨xᵢ, xⱼ⟩ between data points. A kernel function K(xᵢ,xⱼ) computes a value equal to ⟨φ(xᵢ), φ(xⱼ)⟩ directly from the original-space vectors — the high-dimensional (possibly infinite-dimensional) mapping φ is never actually materialized.

Elaborated answer

SVM's dual optimization problem depends only on dot products between pairs of points, never on raw feature vectors in isolation. Normally, mapping to a higher-dimensional space via some φ(x) would require explicitly computing φ(x) for every point (potentially infinite-dimensional, e.g. under an RBF-style mapping) and then taking dot products in that space — computationally infeasible.

The kernel trick exploits the existence of a function K(x,z) that equals ⟨φ(x), φ(z)⟩ but can be computed directly from x and z in the original space. E.g., K(x,z) = exp(−γ‖x−z‖²) corresponds to an infinite-dimensional feature mapping, yet is computed with a simple closed-form expression — the mapping itself is never performed.

QUESTION 13 Wrong diagnosis, then vague

Train accuracy 98%, validation 97%, test (a separate, more recent batch) 65%. Is this overfitting? Diagnose, then say what you'd check first before touching the model.

Must include

Not overfitting — train and val are nearly identical, which is the signature of good generalization within the same distribution. The gap only appears against a differently-distributed test set, which points to distribution shift. First check: compare feature distributions (PSI/KS-test) and label base rates between train/val and test — a statistical check, before investigating "why."

Elaborated answer

Overfitting shows up as a gap between train and validation (same distribution) — e.g. train 98%, val 70%. Here train and val are nearly identical (98% vs 97%), meaning the model generalizes fine to unseen data from the same collection process. The failure only appears on the test batch, which is explicitly a separate, more recent collection — that's a distribution-shift signature (covariate shift, concept drift, or a labeling/process change), not overfitting.

Before touching the model: run a statistical comparison — PSI or KS-test on feature distributions between train/val and test, and check whether the label base rate itself changed. Only after confirming what shifted should you investigate why (seasonality, business changes, etc.) — jumping straight to "why" without a quantitative check first is a common mistake.

QUESTION 14 Mechanically wrong, then partial

Explain "vanishing gradients" mechanically in a deep sigmoid network, layer by layer. Then explain specifically why ReLU mitigates it, referencing its derivative.

Must include

Sigmoid's derivative σ(z)(1−σ(z)) maxes out at 0.25. Backprop multiplies these derivatives across L layers via the chain rule, so the gradient reaching early layers shrinks like 0.25^L — exponential decay with depth. ReLU's derivative is exactly 1 when active (z>0), so multiplying many 1's doesn't shrink the gradient — but neurons with z≤0 get derivative 0 and can die permanently ("dying ReLU").

Elaborated answer

Backprop computes the gradient at an early layer via the chain rule — a product of derivative terms across every layer between it and the output. Sigmoid's derivative σ(z)(1−σ(z)) has a maximum of 0.25 (at z=0) and shrinks further in saturated regions (z far from 0). For a network with L layers, the gradient reaching layer 1 involves multiplying ~L such terms — even in the best case that's 0.25^L, which for L=10 is around 10⁻⁶. This is a multiplicative chain-rule effect across layers, not a shrinking forward-pass activation value.

ReLU's derivative is piecewise constant: exactly 1 for z>0, exactly 0 for z≤0. When active, multiplying many 1's across layers doesn't shrink the gradient at all, avoiding the exponential decay. The trade-off: a neuron with z≤0 gets zero gradient and can never update again — "dying ReLU," a distinct failure mode from vanishing gradients.
