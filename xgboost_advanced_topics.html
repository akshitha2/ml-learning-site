Section 01

Feature Importance — 3 types & their biases

XGBoost exposes three distinct feature importance metrics. All three are flawed in different ways. Interviewers at Big Tech will ask you to explain the bias of each — and why SHAP replaced them all.

Type 1 — Weight (Frequency)

Counts how many times each feature is used as a split across all trees. The most commonly reported metric by default in older XGBoost versions.

weight(feature f) = total number of splits using feature f across all trees

// Example: feature A splits in tree 1 at node 2, tree 3 at node 1, tree 3 at node 4
weight(A) = 3
⚠
Critical bias: High-cardinality features (age, income, timestamp) have many possible split points. XGBoost tries more of them → more likely to find a useful split → split more often → inflated weight score. A feature used 50 times with tiny Gain looks more important than a feature used 3 times with massive Gain. Weight is the least reliable metric.

Type 2 — Gain (Average)

Average loss reduction (Gain) across all splits where the feature is used. Directly connected to the Gain formula you already know.

gain(feature f) = Σ Gain(split s using f) / count(splits using f)

// If feature A is used in 3 splits with Gain = 10, 8, 12:
gain(A) = (10 + 8 + 12) / 3 = 10.0

// If feature B is used in 1 split with Gain = 9.5:
gain(B) = 9.5 / 1 = 9.5

// Gain says A > B, but B had comparable impact in far fewer uses
⚠
Bias: Features used early (near the root) typically show higher Gain because root splits affect all examples. Features used deeper in the tree affect fewer examples — their Gain is smaller by construction, not because they're less informative. Gain is biased toward root-level features.

Type 3 — Cover

Average number of examples (weighted by Hessian) affected by splits on this feature.

cover(feature f) = Σ H_node / count(splits using f)

// H_node = sum of Hessians at the node where the split happens
// For MSE where h=1: cover = average number of examples at the split node

// Feature used at root (1000 examples) and depth 3 (50 examples):
cover = (1000 + 50) / 2 = 525
⚠
Bias: Also favours features used at the root (high node population → high cover). Doesn't distinguish between a split that actually matters and one that just happens to cut through a large portion of the data. Cover ignores the quality of the split entirely.

Section 02

SHAP Values — Theory & Numerical Example

SHAP (SHapley Additive exPlanations) is the modern standard for explaining tree model predictions. XGBoost has native O(TLD) SHAP support — faster than model-agnostic SHAP libraries. This is a mandatory topic at Big Tech interviews.

What SHAP measures

SHAP answers: "How much did each feature contribute to pushing this prediction away from the baseline (average prediction)?" Unlike feature importance (which is global and biased), SHAP gives a per-example, per-feature attribution.

prediction(x) = base_value + φ₁ + φ₂ + ... + φₙ

// base_value = E[ŷ] = mean prediction over training data
// φᵢ = SHAP value for feature i for this specific example
// φᵢ > 0 → feature i pushed prediction ABOVE baseline
// φᵢ < 0 → feature i pushed prediction BELOW baseline
// Σφᵢ = prediction - base_value (exact decomposition, no residual)

Shapley values — the game theory origin

SHAP values come from cooperative game theory. Imagine each feature is a "player" in a game, and the "payout" is the prediction. The Shapley value for a player is their fair share of the payout, defined as the average marginal contribution across all possible feature coalitions.

φᵢ = Σ [|S|!(n−|S|−1)!/n!] · [f(S∪{i}) − f(S)]
S⊆F\{i}

// For each subset S of features NOT including feature i:
// compute prediction with S ∪ {i} minus prediction with S
// weight by |S|!(n−|S|−1)!/n! (coalition size weight)
// sum over all subsets

// For 3 features: need to evaluate 2³ = 8 subsets per feature
// Exact SHAP is O(2ⁿ) — XGBoost uses a tree-specific O(TLD) algorithm

shap_values = model.predict(dmatrix, pred_contribs=True)
# Returns array of shape (n_samples, n_features + 1)
# Last column = base_value (same for all rows)
# shap_values[i, j] = φⱼ for example i, feature j

# Returns (n_samples, n_features, n_features) — φᵢⱼ for each feature pair
◆
The Shapley axioms — why SHAP is theoretically correct: Efficiency (values sum to prediction), Symmetry (symmetric features get equal values), Dummy (zero-contribution features get zero), Additivity (values from independent sub-models sum correctly). These four axioms uniquely determine the Shapley value. No other attribution method satisfies all four simultaneously.

xgb.cv() — Built-in Cross-Validation

XGBoost ships its own cross-validation function that differs from sklearn's CV in important ways — primarily its ability to run early stopping within each fold. This is how you properly tune n_estimators without data leakage.

Why xgb.cv() vs sklearn cross_val_score

Featuresklearn cross_val_scorexgb.cv()
Early stopping per fold✗ — needs custom wrapper✓ native support
Per-round CV metrics✗ — only final score✓ mean±std per boosting round
Finds optimal n_estimators✗✓ automatically
Stratified folds✓manual stratify param
Pipeline integration✓ easyrequires DMatrix

How xgb.cv() works internally

  1. Splits data into k folds. For each round (tree) from 1 to num_boost_round, trains on k−1 folds and evaluates on the held-out fold.
  2. Tracks mean and std of the eval metric across folds at every boosting round: test-rmse-mean, test-rmse-std.
  3. If early stopping is enabled, stops when the mean metric hasn't improved for early_stopping_rounds consecutive rounds.
  4. Returns a DataFrame of shape (num_rounds, 4): train-metric-mean, train-metric-std, test-metric-mean, test-metric-std.
  5. The optimal n_estimators is the row with the best test-metric-mean.
import xgboost as xgb

dtrain = xgb.DMatrix(X_train, label=y_train)

params = {
  'objective': 'binary:logistic',
  'max_depth': 4,
  'learning_rate': 0.05,
  'eval_metric': 'aucpr',
  'subsample': 0.8,
}

cv_results = xgb.cv(
  params=params,
  dtrain=dtrain,
  num_boost_round=5000, # set high, early stopping finds optimum
  nfold=5,
  early_stopping_rounds=50, # stops if no improvement for 50 rounds
  stratified=True, # for classification
  seed=42
)

# Find optimal n_estimators
best_rounds = cv_results['test-aucpr-mean'].idxmax() + 1
best_score = cv_results['test-aucpr-mean'].max()
std_at_best = cv_results['test-aucpr-std'].iloc[best_rounds - 1]

# Now train final model on ALL data with this n_estimators
final_model = xgb.train(params, dtrain, num_boost_round=best_rounds)
⚠
The "1-SE rule" for n_estimators: Instead of picking the exact round with best mean score, consider picking the earliest round within 1 standard error of the best. This gives a simpler model (fewer trees) with statistically equivalent performance. Formula: pick smallest n where score ≥ best_score − best_std.
✓
Production workflow: (1) Use xgb.cv() to find optimal n_estimators with the full hyperparameter set. (2) Retrain final model on all training data using that n_estimators. (3) Never use the validation folds from xgb.cv() to report final test metrics — use a completely held-out test set.

Monotone Constraints

Force XGBoost to learn monotone relationships between features and the target. Critical in regulated industries (banking, insurance, healthcare) where the model must obey business logic. Sleeper topic — comes up constantly in ML-at-scale interviews.

The problem

Without constraints, XGBoost can learn non-monotone relationships even when the domain demands monotonicity. Example: a credit scoring model should always predict lower default probability for higher credit scores. Without constraints, the model might find a local pattern where very-high-credit-score applicants default more (perhaps due to sampling bias) and encode this — producing regulatory violations.

What can go wrong without monotone constraints
Credit score rangeActual default rate in dataModel learnsBusiness requirement
300–400 (poor)35%35%✓ correct
400–550 (fair)18%18%✓ correct
550–700 (good)8%8%✓ correct
700–800 (great)3%3%✓ correct
800–850 (excellent)4% (sampling artifact)4% — UP from 700–800!✗ violation — must be ≤ 3%

How monotone constraints work in XGBoost

When a monotone constraint is set on feature f, XGBoost enforces it during tree building by restricting which splits are valid. For a monotone increasing constraint on feature f:

// For monotone_constraints = +1 on feature f:
// At any node splitting on feature f with threshold t:

left_child_prediction ≤ right_child_prediction
// i.e. lower values of f → lower prediction
// higher values of f → higher prediction

// Enforced by clamping leaf values after each tree build:
// Left leaf: w*_left = min(unconstrained_w*, parent_upper_bound)
// Right leaf: w*_right = max(unconstrained_w*, parent_lower_bound)
Numerical example — monotone constraint in action

Features: credit_score (constrained +1), loan_amount (constrained −1, higher loan = higher risk). Target: default probability.

Without constraint
credit=750 → default_prob = 0.03
credit=800 → default_prob = 0.05 ← violation!
credit=820 → default_prob = 0.02
credit=850 → default_prob = 0.01
With constraint (+1 on credit)
credit=750 → default_prob = 0.04
credit=800 → default_prob = 0.03
credit=820 → default_prob = 0.02
credit=850 → default_prob = 0.01
# Monotone constraints — tuple or dict
# +1 = increasing, -1 = decreasing, 0 = no constraint
params = {
  'monotone_constraints': (1, -1, 0, 0),
  # Feature order: credit_score=+1, loan_amount=-1, age=0, income=0
}

# Or as a dict (cleaner for many features)
params['monotone_constraints'] = {
  'credit_score': 1,
  'loan_amount': -1
}
⚠
Cost of monotone constraints: Constraining the model reduces its capacity to fit the data — expected accuracy loss, especially when the true relationship is genuinely non-monotone in some range. The tradeoff is accuracy vs compliance. In regulated industries, compliance wins. Measure the accuracy cost explicitly: train with and without constraints, compare AUC-PR.
◆
Interview question this unlocks: "How would you build a credit scoring model that must satisfy regulatory monotonicity requirements?" The answer is: monotone constraints + SHAP for per-decision explanation + AUC-PR as primary metric + document the accuracy cost of the constraints explicitly.
Section 05

Interaction Constraints

Restrict which features are allowed to interact in splits within the same tree path. A powerful tool for domain-driven regularisation and compliance in regulated ML systems.

What "interaction" means in a tree

In a decision tree, two features interact if they both appear on the same root-to-leaf path. If feature A splits the root and feature B splits a child node reached via feature A, then A and B interact — the model has learned that the effect of B depends on the value of A.

// Tree path that uses BOTH age and income:
root: if age < 30 → go left
  left: if income < 50k → predict 0.7 (high default)
  left: if income ≥ 50k → predict 0.3 (moderate default)

// This encodes: "for young people (age < 30), income matters"
// → age and income are INTERACTING in this path

Why you'd want to restrict interactions

Two scenarios where interaction constraints are useful:

  1. Regulatory compliance: A model must not use certain features together. Example: race and zip-code cannot both appear on the same decision path in a lending model (proxy discrimination concern).
  2. Domain-driven regularisation: You know from domain knowledge that two features should not interact. Forcing the model to respect this prevents overfitting to spurious interaction patterns in the training data.
# interaction_constraints: list of lists
# Each inner list = features ALLOWED to interact with each other
# Features from different lists CANNOT appear on the same path

params['interaction_constraints'] = [
  [0, 1, 2], # features 0,1,2 can interact with each other
  [3, 4], # features 3,4 can interact with each other
  [5] # feature 5 can only interact with itself (no interactions)
]
# Features 0 and 3 will NEVER appear on the same root-to-leaf path

# Example: credit scoring model
# Group 1: financial features [income, debt, assets]
# Group 2: behavioural features [payment_history, credit_utilisation]
# Group 3: demographic features [age] — isolated, no interactions
◆
Interaction constraints vs feature selection: Interaction constraints do NOT prevent a feature from being used — they only prevent it from appearing on the same path as features from other groups. Feature 5 above can still be used as the root split in every tree — it just can't interact with any other feature within a path. This is subtly different from simply dropping the feature.
⚠
Accuracy cost: Like monotone constraints, interaction constraints reduce model capacity. A model that cannot let age interact with income loses the ability to capture "young borrowers respond differently to income changes." Measure the cost: compare AUC-PR with and without constraints. If the cost is high, investigate whether the constraint is actually necessary or too restrictive.
Section 06

When XGBoost Fails — Failure Modes

XGBoost is not a universal solution. Knowing precisely when it fails — and why — is what separates strong candidates from those who just know how to run the model.

Failure modeWhy it happensWhat to use instead
Extrapolation Trees partition the feature space and predict the mean within each partition. There are no partitions beyond the training range. Predictions for values outside training range = mean of the most extreme leaf. Linear models, neural networks (anything that can extrapolate)
High-dim sparse data Text (bag-of-words), one-hot encoded categoricals with 1000s of levels. Most values are zero. XGBoost's exact splits become noise-chasing. Memory cost of column blocks explodes. Logistic regression with L1, linear SVM, neural networks for text/NLP
Very noisy labels Boosting sequentially amplifies residuals. If early trees learn noise patterns, subsequent trees must "undo" this — compounding noise. RF dilutes noise by averaging. XGBoost accumulates it. Random Forest, heavy regularisation (γ++, α++, low max_depth), label smoothing
Very small datasets (<500 rows) With enough regularisation parameters, XGBoost can memorise even a few hundred examples. The sequential boosting compounds overfitting even with low learning rate. Random Forest, logistic regression, SVM, Gaussian processes
Online / streaming learning XGBoost requires the full dataset to compute gradients and Hessians. Cannot update incrementally without retraining from scratch. Online gradient descent, vowpal wabbit, river (Python online ML library)
Temporal / sequential data No native time-series awareness. Naive cross-validation causes future data leakage. No concept of sequence, lag, or trend. LSTMs, Temporal Fusion Transformers, Prophet (with XGBoost for residuals)
Unstructured data (images, text, audio) Trees operate on tabular feature vectors. Cannot learn spatial hierarchies, attention, embeddings, or sequential context natively. CNNs, transformers, BERT/GPT for text, ResNets for images
Many correlated features Feature importance metrics become meaningless when features are correlated — importance gets split arbitrarily across correlated features. SHAP handles this better but still has issues. Feature selection first, PCA, or Lasso + XGBoost in two stages

Extrapolation — numerical illustration

Training data range vs test prediction
// Training data: house_size from 500 to 3000 sqft
// For size=2800: XGBoost predicts correctly using right-most leaf
// For size=5000: XGBoost has no leaf for this range
// → falls into the rightmost leaf (training max)
// → predicts same value as size=3000
// → prediction PLATEAUS, never grows with feature value

prediction(5000 sqft) = prediction(3000 sqft) ← extrapolation failure
// A linear model would extrapolate: prediction(5000) > prediction(3000)
⚠
Interview trap: "XGBoost doesn't extrapolate — this is a weakness." Partially correct — it depends on your use case. For many real-world applications (fraud, churn, pricing) you never need to extrapolate because inputs stay within training range. But for forecasting or regression on unbounded targets, this is a genuine failure mode that must be acknowledged.

XGBoost vs sklearn GBM — Precise Differences

sklearn's GradientBoostingClassifier and XGBoost are often treated as equivalent. They are not. Knowing the precise differences is a strong signal of depth in interviews.

Dimensionsklearn GBMXGBoost
Leaf value computationFirst-order only: w* = mean residual (MSE) or median (MAE)Second-order: w* = −G/(H+λ). Newton's method. Better leaf values for non-MSE losses.
Regularisation on leavesNone — no L1/L2 penalty on leaf valuesλ (L2) and α (L1) directly in leaf formula. Default λ=1 already regularising.
Split penaltyNone — any positive gain triggers a splitγ parameter: split only if Gain > γ. Principled pruning.
Missing value handlingRequires imputation before trainingNative learned default directions. No imputation needed.
ParallelismSequential within and across trees (GIL-limited in Python)Parallel split evaluation within each tree. Column blocks enable multi-core.
Tree building algorithmExact greedy onlyexact, approx, hist, gpu_hist. hist is ~10× faster on large datasets.
Subsamplingsubsample: row sampling (same as XGBoost)subsample + colsample_bytree + colsample_bylevel + colsample_bynode
Early stoppingn_iter_no_change with validation_fractionearly_stopping_rounds with explicit eval_set. More flexible.
Monotone constraintsNot supportedmonotone_constraints parameter. Native support.
Interaction constraintsNot supportedinteraction_constraints parameter. Native support.
SHAP valuesVia external shap library (slow)Native O(TLD) fast tree SHAP. pred_contribs=True.
Default parameter namesmax_features, min_samples_leafcolsample_bytree, min_child_weight (Hessian-based, not count-based)
Speed at scale (>1M rows)Slow — exact greedy, no parallelismFast — hist + column blocks + multi-core
◆
The one-line answer: sklearn GBM and XGBoost implement the same boosting loop but XGBoost adds: (1) Newton's method for leaf values, (2) explicit L1/L2/γ regularisation, (3) native missing values, (4) column blocks + parallelism, (5) monotone/interaction constraints, (6) native SHAP. For small datasets they're often equivalent in accuracy. For large datasets, XGBoost wins on speed by 10–100×.
⚠
min_child_weight trap: sklearn's min_samples_leaf is a count of examples. XGBoost's min_child_weight is the minimum sum of Hessians in a leaf. For MSE where h=1, they're equivalent. For log-loss where h=p(1−p), they're not — a leaf with 100 very confident predictions might have sum(h) = 2.0, which could be below a min_child_weight of 5. Never assume these parameters are interchangeable when porting models between sklearn and XGBoost.
