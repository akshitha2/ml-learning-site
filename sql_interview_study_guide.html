Mental Model

SQL Query Execution Order

Almost every SQL "gotcha" traces back to this one fact: the clauses don't run in the order you write them.

FROM → // resolve tables, apply joins WHERE → // filter rows (no aggregates allowed yet) GROUP BY → // form groups HAVING → // filter groups (aggregates allowed) SELECT → // compute expressions, window functions run here ORDER BY → // sort final output LIMIT → // trim rows
💡

This single ordering explains: why WHERE can't use aggregates (M3), why HAVING can (M2), and why you can't filter a window function in WHERE (R3) — window functions are computed at the SELECT stage, after WHERE has already run.

Core

Core Querying

Syntax

SELECT column, AGG_FUNC(column) FROM table WHERE row_condition GROUP BY column HAVING group_condition ORDER BY column LIMIT n;
Trap — WHERE vs HAVING

WHERE filters individual rows before grouping; HAVING filters aggregated groups after. Using WHERE COUNT(*) > 5 fails — aggregates don't exist yet at that stage.

Joins

Joins & Anti-Joins

INNER JOIN
  • Only matching rows on both sides
  • Same result via comma syntax + WHERE, as long as condition is identical
LEFT JOIN
  • All left rows + matches from right (NULL if none)
  • Classic base for anti-joins

Anti-Join Pattern (find unmatched rows)

SELECT a.name FROM customers a LEFT JOIN orders b ON a.customer_id = b.customer_id WHERE b.customer_id IS NULL; -- customers with zero orders
Trap — ON vs WHERE on OUTER joins

Putting a filter on the right table in WHERE instead of ON silently converts a LEFT JOIN into an INNER JOIN, because unmatched rows have NULLs, and NULL = 'x' is never true.

-- keeps all left rows, filters right side only when matched LEFT JOIN b ON a.id = b.id AND b.status = 'active' -- BUG: silently drops unmatched left rows LEFT JOIN b ON a.id = b.id WHERE b.status = 'active'
Window Functions

Cheat Sheet

Window functions compute a value across a set of related rows without collapsing them — every original row survives, unlike GROUP BY.

FUNC(...) OVER ( PARTITION BY col -- optional: resets the window per group ORDER BY col -- optional: defines row sequence ROWS/RANGE BETWEEN x AND y -- optional: defines the frame )
FunctionWhat it returns
ROW_NUMBER()Unique sequential number per row, no ties
RANK()Same rank for ties, skips subsequent numbers
DENSE_RANK()Same rank for ties, no gaps
NTILE(n)Assigns each row to 1 of n equal-ish buckets
LAG(col, n)Value from n rows before (NULL if none)
LEAD(col, n)Value from n rows after (NULL if none)
FIRST_VALUE(col)First value in the frame
LAST_VALUE(col)Last value in the frame — needs frame override, see below
PERCENT_RANK()Relative rank as a fraction 0–1
CUME_DIST()Cumulative distribution — % of rows ≤ current
PERCENTILE_CONT(p)Interpolated value at percentile p (needs WITHIN GROUP)
PERCENTILE_DISC(p)Actual row value at percentile p, no interpolation

Worked Example — Rank Within Region

SELECT employee, region, amount, RANK() OVER (PARTITION BY region ORDER BY amount DESC) AS rank FROM sales;
Trap — RANK vs DENSE_RANK on ties
FunctionRow1 (300)Row2 (300, tie)Row3 (100)
ROW_NUMBER()123
RANK()113 (skips 2)
DENSE_RANK()112 (no skip)

"2nd highest salary" queries must use DENSE_RANK — RANK would skip straight to a rank that may not exist if the top two tie.

NTILE — Uneven Buckets

7 rows into NTILE(3) does not error — it distributes the remainder to the earliest groups: sizes come out as 3, 2, 2 (not 2, 2, 3).

Trap — NTILE ≠ percentile value

NTILE(4) assigns a group label (1–4) to each row. It does NOT return the dollar value at the 25th/50th/75th percentile — that's what PERCENTILE_CONT does.

LAG / LEAD

SELECT date, revenue, revenue - LAG(revenue) OVER (ORDER BY date) AS day_over_day_change FROM daily_revenue;
💡

First row's LAG is NULL (no prior row). Wrap in COALESCE(LAG(revenue), 0) when the result feeds into further math.

FIRST_VALUE / LAST_VALUE

SELECT employee, amount, FIRST_VALUE(amount) OVER (PARTITION BY region ORDER BY amount DESC) AS top_amount FROM sales;
Trap — LAST_VALUE's default frame

The default window frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — so LAST_VALUE only sees up to the current row and just returns the current row's own value.

-- WRONG (returns current row, not region minimum) LAST_VALUE(amount) OVER (PARTITION BY region ORDER BY amount DESC) -- CORRECT (frame extended to the whole partition) LAST_VALUE(amount) OVER ( PARTITION BY region ORDER BY amount DESC ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING )

Multi-Partition Windows

RANK() OVER (PARTITION BY region, quarter ORDER BY amount DESC)

Each unique (region, quarter) pair becomes its own independent mini-window — ranking resets at every new combination. Used heavily in cohort analysis: PARTITION BY signup_cohort, days_since_signup.

Frames

ROWS vs RANGE, Running Totals & Moving Averages

Frame Syntax

ROWS/RANGE BETWEEN <start> AND <end> -- start/end options: UNBOUNDED PRECEDING -- from the very first row of the partition N PRECEDING -- N rows/values before current CURRENT ROW N FOLLOWING -- N rows/values after current UNBOUNDED FOLLOWING -- to the very last row of the partition

Running Total

SELECT date, revenue, SUM(revenue) OVER (ORDER BY date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_total FROM daily_revenue;

Moving Average (last 3 rows incl. current)

SELECT date, revenue, AVG(revenue) OVER (ORDER BY date ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) AS moving_avg_3d FROM daily_revenue;
Trap — off-by-one on window size

2 PRECEDING AND CURRENT ROW = 3 total rows (2 before + current). A "5-day moving average" needs 4 PRECEDING, not 5.

ROWS vs RANGE — the tie-breaking difference

Both give identical results until the ORDER BY column has duplicate values.

daterevenueSUM … ROWSSUM … RANGE
2024-01-01100100200
2024-01-02100200200
2024-01-03130330330
  • ROWS = physical row count — even tied rows get their own distinct running total.
  • RANGE = logical peer grouping — all rows sharing the same ORDER BY value get the same result (sum "as of that value").
Rule of Thumb
If you don't specify a frame explicitly, the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — which silently produces duplicate totals when ties exist. Always write ROWS BETWEEN ... explicitly for a true row-by-row running total.
Percentiles

PERCENTILE_CONT / DISC, PERCENT_RANK, CUME_DIST

PERCENTILE_CONT — interpolated value

SELECT PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY spend) AS p75_spend FROM customers;

Returns the actual dollar value at the 75th percentile, interpolating between two data points if the percentile falls between them.

PERCENTILE_DISC — discrete value

SELECT PERCENTILE_DISC(0.75) WITHIN GROUP (ORDER BY spend) AS p75_spend FROM customers;

Same idea, but always returns an actual row's value — no interpolation, so the result always exists in the dataset.

PERCENT_RANK vs CUME_DIST

SELECT name, score, PERCENT_RANK() OVER (ORDER BY score) AS pct_rank, CUME_DIST() OVER (ORDER BY score) AS cume_dist FROM students;
  • PERCENT_RANK = (rank − 1) / (total rows − 1). Always 0 for the first row, 1 for the last.
  • CUME_DIST = (rows ≤ current) / (total rows). Never 0, reflects real cumulative proportion including ties.
Trap — NTILE vs PERCENTILE_CONT confusion

Interviewers often ask "what's the 90th percentile spend?" — the correct tool is PERCENTILE_CONT, not NTILE(10). NTILE only tells you which decile bucket a row falls in, not the cutoff dollar value.

Row-Level Output — PARTITION BY Only, No ORDER BY / Frame

PERCENTILE_CONT/DISC can return a value on every row — same row-level behavior as any other window aggregate — by adding OVER (PARTITION BY ...). No GROUP BY needed.

SELECT customer_id, region, spend, PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY spend) OVER (PARTITION BY region) AS p75_region FROM customers;
customer_idregionspendp75_region
1East100250
2East300250
3East250250
4West400380

Every East row shows the same 250 — that's the region's 75th-percentile spend, repeated across all rows in the partition, exactly like SUM(x) OVER (PARTITION BY region) would repeat a regional total.

Trap — the OVER clause only accepts PARTITION BY here

Unlike RANK, LAG, or SUM, you cannot add ORDER BY or a ROWS/RANGE BETWEEN frame inside this function's OVER(...). Trying to do so errors out (SQL Server: "The function 'PERCENTILE_CONT' may not have ORDER BY in OVER clause").

-- WRONG — errors PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY spend) OVER (PARTITION BY region ORDER BY spend ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) -- CORRECT — ordering already lives in WITHIN GROUP PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY spend) OVER (PARTITION BY region)

Why: the sort order is already fixed by WITHIN GROUP (ORDER BY ...) — it tells the function how to order values before computing the percentile. A percentile is a property of the whole partition, not a sliding frame, so a second ORDER BY or a ROWS/RANGE frame inside OVER would conflict with that and isn't allowed.

Dialect Note
Postgres does not support PERCENTILE_CONT/DISC as a window function at all — only as a plain aggregate with GROUP BY (no OVER clause). SQL Server, Snowflake, Redshift, and Yellowbrick all support the PARTITION-BY-only window form shown above.
CTEs

CTEs & Recursive Queries

Basic CTE

WITH avg_sal AS ( SELECT AVG(salary) AS avg_salary FROM employees ) SELECT e.* FROM employees e JOIN avg_sal ON e.salary >= avg_sal.avg_salary;

Prefer CTEs over nested subqueries for readability, reusability within the same query, and because recursion requires them.

Recursive CTE Pattern (memorize this shape)

WITH RECURSIVE subordinates AS ( -- 1. Base case: anchor row(s) SELECT id, name, manager_id, 1 AS level FROM employees WHERE manager_id = 1 UNION ALL -- 2. Recursive case: join CTE to itself SELECT e.id, e.name, e.manager_id, s.level + 1 FROM employees e JOIN subordinates s ON e.manager_id = s.id ) SELECT * FROM subordinates;
  • 1Base case runs once — the anchor rows.
  • 2Recursive case joins back to the CTE itself, finding the next level down.
  • 3Repeats until a pass finds zero new rows, then stops automatically.
Trap — UNION vs UNION ALL in recursion

Use UNION ALL, not UNION. UNION deduplicates every iteration (slower), and on cyclic data can behave unpredictably. Also don't forget the RECURSIVE keyword (required in Postgres/MySQL 8+).

DS Patterns

Funnels, Cohorts & A/B Tests

Funnel — Simple Stage Counts

SELECT COUNT(DISTINCT CASE WHEN event_name = 'signup' THEN user_id END) AS signup_users, COUNT(DISTINCT CASE WHEN event_name = 'view_item' THEN user_id END) AS view_item_users, COUNT(DISTINCT CASE WHEN event_name = 'purchase' THEN user_id END) AS purchase_users FROM events;

Funnel — Strict, Order-Enforced

WITH user_funnel AS ( SELECT user_id, MIN(CASE WHEN event_name='signup' THEN event_time END) AS signup_time, MIN(CASE WHEN event_name='view_item' THEN event_time END) AS view_time, MIN(CASE WHEN event_name='purchase' THEN event_time END) AS purchase_time FROM events GROUP BY user_id ) SELECT COUNT(signup_time) AS signup_users, COUNT(CASE WHEN view_time > signup_time THEN 1 END) AS view_item_users, COUNT(CASE WHEN purchase_time > view_time THEN 1 END) AS purchase_users FROM user_funnel;
Trap — naive funnels double-count out-of-order events

A plain GROUP BY event_name, COUNT(*) counts anyone who ever fired "purchase," even a user who purchased before viewing (bad data / weird tracking). The strict version checks each stage happened after the previous one.

Cohort Retention %

WITH cohort_activity AS ( SELECT signup_date AS cohort, DATE_DIFF('week', signup_date, activity_date) AS week_num, COUNT(DISTINCT user_id) AS active_users FROM activity GROUP BY 1, 2 ) SELECT cohort, week_num, active_users, active_users * 1.0 / FIRST_VALUE(active_users) OVER (PARTITION BY cohort ORDER BY week_num) AS retention_pct FROM cohort_activity;
💡

Week 0's count becomes the baseline denominator via FIRST_VALUE — every later week is expressed as a % of that baseline, per cohort.

A/B Test Conversion Rate

SELECT variant, COUNT(DISTINCT user_id) AS users, COUNT(DISTINCT CASE WHEN converted=1 THEN user_id END) AS converted_users, COUNT(DISTINCT CASE WHEN converted=1 THEN user_id END) * 1.0 / COUNT(DISTINCT user_id) AS conversion_rate FROM experiment GROUP BY 1;
Beyond the SQL
SQL gets you the rates — it doesn't tell you if the difference between variants is statistically significant. That needs a two-proportion z-test or chi-square test on top, especially with small sample sizes.
Consolidated

Gotchas Summary

01Integer division silently truncates

COUNT(x) / COUNT(y) with integer columns truncates to 0. Always write * 1.0 before dividing.

02LAST_VALUE's default frame

Returns the current row, not the partition's last row, unless the frame is overridden to UNBOUNDED FOLLOWING.

03RANGE vs ROWS on tied ORDER BY values

RANGE groups ties into one running total; ROWS gives each row its own — pick ROWS for a true row-by-row running total.

04ON vs WHERE on OUTER joins

A right-table filter in WHERE turns a LEFT JOIN into an INNER JOIN by discarding NULL (unmatched) rows.

05Aggregates can't go in WHERE

WHERE MAX(x) = ... fails — aggregates aren't computed yet at the WHERE stage. Use HAVING or a subquery.

06Window functions can't be filtered in WHERE

WHERE RANK() OVER (...) = 1 fails for the same reason — wrap it in a subquery/CTE and filter outside.

07NTILE is a group label, not a percentile value

Use PERCENTILE_CONT/DISC when you need the actual cutoff value, not just a bucket number.

08RANK vs DENSE_RANK on "Nth highest" questions

RANK skips numbers after ties (can skip past the value you want); DENSE_RANK doesn't. Use DENSE_RANK for "Nth highest distinct value."

09UNION vs UNION ALL in recursive CTEs

Use UNION ALL — UNION's deduplication is slower and can behave unpredictably on cyclic data.

10Naive funnels over-count out-of-order events

Enforce stage ordering with MIN(event_time) per stage and explicit "later than previous stage" checks.

Practice

50 Interview Questions

Click a card to expand. 20 core/fundamentals questions, plus 30 weighted toward window functions and DS-style queries — filter to focus your review.

Core & Fundamentals

C1
What's the difference between WHERE and HAVING?
+
Answer

WHERE filters individual rows before grouping and can't reference aggregates. HAVING filters groups after aggregation and can reference aggregate functions like SUM/COUNT.

C2
Difference between DELETE, TRUNCATE, and DROP?
+
Answer
  • DELETE — removes rows (optionally filtered by WHERE), logged row-by-row, can be rolled back, triggers fire.
  • TRUNCATE — removes all rows, minimal logging, faster, resets identity columns, can't use WHERE.
  • DROP — removes the entire table structure plus data.
C3
What's the difference between UNION and UNION ALL?
+
Answer

UNION removes duplicate rows across the combined result (requires a sort/hash step); UNION ALL keeps every row including duplicates and is faster. Default to UNION ALL unless you specifically need deduplication.

C4
Difference between INNER, LEFT, RIGHT, and FULL OUTER JOIN?
+
Answer
  • INNER — only rows matching in both tables.
  • LEFT — all left rows, matched right columns or NULL.
  • RIGHT — all right rows, matched left columns or NULL.
  • FULL OUTER — all rows from both sides, NULL where no match on either side.
C5
EXISTS vs IN — when would you prefer one over the other?
+
Answer

EXISTS stops scanning as soon as one matching row is found and handles NULLs safely — generally preferred for correlated subqueries against large tables. IN materializes the full subquery result first and can silently return no rows if the subquery list contains a NULL alongside a NOT IN.

Trap

NOT IN with a subquery that can return NULL silently returns zero rows for the entire query — a very common bug.

C6
What does COALESCE(a, b, c) do?
+
Answer

Returns the first non-NULL value among its arguments, evaluated left to right, per row. Commonly used to supply a fallback value when a column might be NULL, e.g. COALESCE(current_price, last_known_price, 0).

C7
What does NULLIF(a, b) do, and when is it useful?
+
Answer

Returns NULL if a = b, otherwise returns a. Commonly used to avoid divide-by-zero errors: x / NULLIF(y, 0) returns NULL instead of erroring when y is 0.

C8
Explain the ACID properties of a transaction.
+
Answer
  • Atomicity — all statements in the transaction succeed, or none do.
  • Consistency — the database moves from one valid state to another.
  • Isolation — concurrent transactions don't interfere with each other's intermediate state.
  • Durability — once committed, changes survive a crash.
C9
What's the difference between a primary key and a unique key?
+
Answer

A table can have only one primary key, and it can't contain NULLs. A table can have multiple unique keys, and (in most databases) a unique key column can allow one NULL, since NULL isn't considered equal to another NULL.

C10
What does an index do, and what's the trade-off of adding one?
+
Answer

An index is a separate sorted structure (often a B-tree) that lets the engine find rows without scanning the whole table — speeds up SELECT/WHERE/JOIN lookups. Trade-off: every INSERT/UPDATE/DELETE also has to update the index, so writes get slower and storage grows.

C11
Write a query using CASE WHEN to bucket ages into groups.
+
Answer
SELECT name, CASE WHEN age < 18 THEN 'minor' WHEN age BETWEEN 18 AND 64 THEN 'adult' ELSE 'senior' END AS age_group FROM people;
C12
How do you paginate results — return rows 21 to 30?
+
Answer
SELECT * FROM orders ORDER BY order_id LIMIT 10 OFFSET 20;
Trap

OFFSET without a deterministic ORDER BY can return inconsistent pages, since row order isn't guaranteed otherwise.

C13
What's the difference between a subquery and a JOIN — when would results differ?
+
Answer

A JOIN combines columns from both tables side by side and can multiply rows on a one-to-many match. A subquery in WHERE/SELECT just returns a value or list to filter/compute against, without adding columns or duplicating rows. If you only need to filter (not display) columns from another table, a subquery avoids accidental row duplication that a JOIN could cause.

C14
What is a self-join, and give an example use case.
+
Answer
SELECT e.name AS employee, m.name AS manager FROM employees e LEFT JOIN employees m ON e.manager_id = m.id;

A table joined to itself — typically used for hierarchical data (employee/manager) or comparing rows within the same table (e.g., finding pairs of events).

C15
What does GROUP BY do when you select a column that isn't in the GROUP BY or an aggregate?
+
Answer

Standard SQL (and Postgres) rejects it with an error. MySQL historically allowed it and silently returned an arbitrary row's value for that column — a common source of subtly wrong results if ONLY_FULL_GROUP_BY isn't enabled.

C16
Write a query to count NULL vs non-NULL values in a column.
+
Answer
SELECT COUNT(*) - COUNT(email) AS null_count, COUNT(email) AS non_null_count FROM users;

COUNT(*) counts all rows; COUNT(column) only counts non-NULL values in that column — the difference between them is the NULL count.

C17
What's the difference between CROSS JOIN and INNER JOIN?
+
Answer

CROSS JOIN returns the Cartesian product — every row of table A paired with every row of table B, no ON condition. INNER JOIN requires a matching condition and only returns rows that satisfy it. A CROSS JOIN with a WHERE filter added after is functionally an INNER JOIN, just less readable.

C18
How do you find rows in table A that don't exist in table B, using EXISTS instead of a JOIN?
+
Answer
SELECT a.* FROM table_a a WHERE NOT EXISTS ( SELECT 1 FROM table_b b WHERE b.a_id = a.id );

Functionally equivalent to the LEFT JOIN + IS NULL anti-join pattern; NOT EXISTS is often preferred since it can't be fooled by NULLs the way NOT IN can.

C19
What's the difference between a VIEW and a table?
+
Answer

A view is a saved, named SELECT query — it has no data of its own (unless it's a materialized view) and re-runs the underlying query each time it's referenced. Useful for encapsulating complex logic or restricting which columns/rows a user can see, without duplicating data.

C20
Write a query to find the total and average order amount per customer, only for customers with more than 3 orders.
+
Answer
SELECT customer_id, SUM(amount) AS total, AVG(amount) AS avg_amount FROM orders GROUP BY customer_id HAVING COUNT(*) > 3;

Window Functions & Hard / DS Questions

W1
Find the 2nd highest salary from an employees table, handling ties correctly.
+
Answer
SELECT DISTINCT salary FROM ( SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk FROM employees ) t WHERE rnk = 2;
Trap

Using RANK() instead of DENSE_RANK() can skip past the actual 2nd distinct value if the top salary is tied.

W2
Why can't you filter a window function's result directly in WHERE?
+
Answer

Window functions execute at the SELECT stage of logical query processing — after WHERE has already run. The value doesn't exist yet when WHERE is evaluated.

SELECT * FROM ( SELECT *, RANK() OVER (ORDER BY amount DESC) AS rnk FROM sales ) t WHERE rnk = 1;
W3
Compute a 7-day moving average of daily revenue.
+
Answer
SELECT date, revenue, AVG(revenue) OVER (ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS ma_7d FROM daily_revenue;
Trap

"7-day" means 6 PRECEDING + current row = 7 total, not 7 PRECEDING.

W4
What's wrong with using LAST_VALUE() with no explicit frame to find each region's maximum?
+
Answer

Default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — LAST_VALUE only sees up to the current row, so it just returns that row's own value.

LAST_VALUE(amount) OVER ( PARTITION BY region ORDER BY amount DESC ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING )
W5
Find each customer's first and most recent order date, and days between them, in one query.
+
Answer
SELECT DISTINCT customer_id, FIRST_VALUE(order_date) OVER (PARTITION BY customer_id ORDER BY order_date) AS first_order, LAST_VALUE(order_date) OVER ( PARTITION BY customer_id ORDER BY order_date ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING) AS last_order FROM orders;
Trap

Forgetting DISTINCT leaves duplicate rows since the window function repeats the same value on every row per customer.

W6
You need the dollar value marking the top 10% of customer spend. Which function, and why not NTILE?
+
Answer
SELECT PERCENTILE_CONT(0.9) WITHIN GROUP (ORDER BY spend) AS p90_spend FROM customers;

NTILE(10) would only tell you which decile bucket a customer falls into — it doesn't return the actual cutoff dollar value.

W7
Compute month-over-month % growth in revenue.
+
Answer
SELECT month, revenue, (revenue - LAG(revenue) OVER (ORDER BY month)) * 100.0 / NULLIF(LAG(revenue) OVER (ORDER BY month), 0) AS pct_growth FROM monthly_revenue;
Trap

Divide-by-zero if a prior month had 0 revenue — wrap the denominator in NULLIF to avoid an error, and remember integer division truncation if not using * 1.0 / 100.0.

W8
Rank products within each category by sales, and also show the category's total sales alongside each row.
+
Answer
SELECT product, category, sales, RANK() OVER (PARTITION BY category ORDER BY sales DESC) AS rank_in_cat, SUM(sales) OVER (PARTITION BY category) AS category_total FROM products;

Two window functions with the same PARTITION BY can run side-by-side in one SELECT — no need for separate queries or joins.

W9
Detect the first purchase event for every user — the "acquisition" event.
+
Answer
SELECT * FROM ( SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY event_time) AS rn FROM events WHERE event_name = 'purchase' ) t WHERE rn = 1;
Trap

Using RANK() instead of ROW_NUMBER() here can return multiple rows if two purchase events tie on exact timestamp — ROW_NUMBER guarantees exactly one row per user.

W10
Write a recursive CTE to list an employee and all their subordinates (direct + indirect).
+
Answer
WITH RECURSIVE subs AS ( SELECT id, manager_id, 1 AS lvl FROM employees WHERE manager_id = 1 UNION ALL SELECT e.id, e.manager_id, s.lvl + 1 FROM employees e JOIN subs s ON e.manager_id = s.id ) SELECT * FROM subs;
Trap

UNION ALL, not UNION — dedup on every recursive pass is wasteful and risky on cyclic data.

W11
Compute week-1 retention % for each signup cohort.
+
Answer
WITH cohort AS ( SELECT signup_week, week_num, COUNT(DISTINCT user_id) AS users FROM logins GROUP BY 1,2 ) SELECT signup_week, week_num, users * 1.0 / FIRST_VALUE(users) OVER (PARTITION BY signup_week ORDER BY week_num) AS retention_pct FROM cohort;
Trap

Forgetting * 1.0 truncates the ratio to 0 or 1 under integer division.

W12
Build a 3-step funnel (signup → view_item → purchase) that enforces correct event ordering per user.
+
Answer
WITH f AS ( SELECT user_id, MIN(CASE WHEN event_name='signup' THEN event_time END) AS t1, MIN(CASE WHEN event_name='view_item' THEN event_time END) AS t2, MIN(CASE WHEN event_name='purchase' THEN event_time END) AS t3 FROM events GROUP BY user_id ) SELECT COUNT(t1) AS signup, COUNT(CASE WHEN t2 > t1 THEN 1 END) AS viewed, COUNT(CASE WHEN t3 > t2 THEN 1 END) AS purchased FROM f;
Trap

A plain COUNT(*) GROUP BY event_name double-counts users who purchased without a valid prior view — inflating conversion.

W13
Given an experiment table (user_id, variant, converted), compute conversion rate per variant safely.
+
Answer
SELECT variant, COUNT(DISTINCT CASE WHEN converted=1 THEN user_id END) * 1.0 / COUNT(DISTINCT user_id) AS conv_rate FROM experiment GROUP BY 1;

Follow-up: SQL gives you the rate, not statistical significance — that needs a two-proportion z-test on top, especially with small n.

W14
Find users who churned — active last month but not this month.
+
Answer
SELECT DISTINCT last.user_id FROM monthly_activity last LEFT JOIN monthly_activity curr ON last.user_id = curr.user_id AND curr.month = last.month + 1 WHERE last.month = 'Jan' AND curr.user_id IS NULL;

Classic anti-join pattern (LEFT JOIN + IS NULL) applied to a time-shifted self-join — very common in churn queries.

W15
Assign a session ID to events, starting a new session whenever the gap between events exceeds 30 minutes.
+
Answer
WITH gaps AS ( SELECT user_id, event_time, CASE WHEN event_time - LAG(event_time) OVER (PARTITION BY user_id ORDER BY event_time) > INTERVAL '30 minutes' THEN 1 ELSE 0 END AS new_session FROM events ) SELECT *, SUM(new_session) OVER (PARTITION BY user_id ORDER BY event_time) AS session_id FROM gaps;

A running SUM of a 0/1 flag is a classic pattern for assigning group IDs based on a gap condition — worth memorizing the shape.

W16
What's the difference between PERCENT_RANK and CUME_DIST?
+
Answer

PERCENT_RANK = (rank − 1) / (n − 1) — always 0 for the first row. CUME_DIST = (rows ≤ current) / n — reflects true cumulative share and is never 0.

Trap

Confusing these when asked to compute "what percentile is this score in" — CUME_DIST is usually the intended answer for that phrasing.

W17
Find the top 3 highest-paid employees per department.
+
Answer
SELECT * FROM ( SELECT *, DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rnk FROM employees ) t WHERE rnk <= 3;
Trap

ROW_NUMBER would arbitrarily cut ties and might exclude someone tied for 3rd place; DENSE_RANK includes all ties within top 3.

W18
Compute the cumulative % of total revenue contributed as you move from the highest to lowest spending customer (a Pareto/80-20 check).
+
Answer
SELECT customer_id, revenue, SUM(revenue) OVER (ORDER BY revenue DESC ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) * 1.0 / SUM(revenue) OVER () AS cum_pct FROM customers;

SUM(revenue) OVER () with empty parentheses — no PARTITION, no ORDER — computes one grand total across all rows, used here as the denominator.

W19
What does SUM(amount) OVER () with completely empty parentheses do?
+
Answer

No PARTITION BY means the whole table is one window; no ORDER BY means no frame restriction — so it returns the grand total of the column, repeated on every row.

W20
Identify products whose sales this month dropped compared to last month.
+
Answer
SELECT product, month, sales, LAG(sales) OVER (PARTITION BY product ORDER BY month) AS prev_sales FROM monthly_sales QUALIFY sales < LAG(sales) OVER (PARTITION BY product ORDER BY month); -- if QUALIFY unsupported: wrap in a subquery and filter in the outer WHERE instead

QUALIFY (Snowflake/BigQuery) filters directly on a window function without a wrapping subquery — worth mentioning as a dialect-specific shortcut if asked.

W21
Explain why WHERE MAX(amount) = ... in a query fails, and how to fix it.
+
Answer

WHERE executes before aggregation in logical query order — MAX() doesn't exist yet at that stage. Fix with a subquery or HAVING (if grouping).

SELECT * FROM orders WHERE amount = (SELECT MAX(amount) FROM orders);
W22
Find duplicate rows in a table and keep only one copy per duplicate group.
+
Answer
DELETE FROM t WHERE id NOT IN ( SELECT MIN(id) FROM t GROUP BY col1, col2, col3 ); -- or, non-destructive, to just view survivors: SELECT * FROM ( SELECT *, ROW_NUMBER() OVER (PARTITION BY col1, col2, col3 ORDER BY id) AS rn FROM t ) x WHERE rn = 1;

ROW_NUMBER + PARTITION BY the duplicate-defining columns is the standard dedup pattern — very common in data cleaning interview questions.

W23
Pivot a long table (user_id, metric, value) into a wide table with one column per metric.
+
Answer
SELECT user_id, MAX(CASE WHEN metric='clicks' THEN value END) AS clicks, MAX(CASE WHEN metric='views' THEN value END) AS views FROM long_table GROUP BY user_id;

MAX + CASE WHEN is the portable pivot pattern (works everywhere); native PIVOT syntax exists but varies by dialect.

W24
Given daily active users, compute a 7-day rolling unique user count (not just a rolling sum).
+
Answer

Window functions can't do a rolling DISTINCT COUNT directly (COUNT DISTINCT isn't supported as a window aggregate in most engines). Typical fix: self-join each date to the last 7 days and COUNT DISTINCT user_id in a GROUP BY, or use approximate distinct-count functions (e.g., APPROX_COUNT_DISTINCT) if scale demands it.

Trap

Assuming COUNT(DISTINCT x) OVER (...) works like SUM/AVG — many engines reject it or restrict its frame options.

W25
Compute the median of a column using window/aggregate functions.
+
Answer
SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY amount) AS median FROM orders;

Median is just the 50th percentile — PERCENTILE_CONT(0.5) handles even-count interpolation automatically, unlike manually averaging the two middle rows.

W26
For each order, show the number of orders placed by that customer so far (including the current one).
+
Answer
SELECT order_id, customer_id, order_date, ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date) AS order_sequence FROM orders;

This "order_sequence" pattern is frequently the first step toward identifying "1st order," "2nd order," or computing repeat-purchase rate.

W27
Compute the % contribution of each product category to total revenue, per month.
+
Answer
SELECT month, category, revenue, revenue * 1.0 / SUM(revenue) OVER (PARTITION BY month) AS pct_of_month FROM category_revenue;

SUM(...) OVER (PARTITION BY month) gives the monthly total on every row of that month, letting you compute a share without a separate join.

W28
Two-proportion comparison aside — what SQL-only check would you run before trusting an A/B test's conversion numbers?
+
Answer
  • Sample ratio check — are variant group sizes roughly what was intended (e.g., 50/50)? A skewed split can signal a bugged randomizer.
  • Check for duplicate user_ids assigned to both variants (contamination).
  • Check for users who appear multiple times inflating conversion counts if not using COUNT(DISTINCT ...).
W29
Find the longest streak of consecutive login days for each user.
+
Answer
WITH grp AS ( SELECT user_id, login_date, login_date - ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY login_date) * INTERVAL '1 day' AS grp_key FROM logins ) SELECT user_id, COUNT(*) AS streak_len FROM grp GROUP BY user_id, grp_key ORDER BY streak_len DESC;

Classic "date minus row_number" trick: consecutive dates minus their sequential row number always land on the identical value, forming a natural grouping key for streaks.

W30
You're asked to explain window functions to a product manager in two sentences. What do you say?
+
Answer

"A window function lets you compare each row to its neighbors — like its rank, the value before it, or a running total — without losing any of the original rows, unlike a GROUP BY which collapses them." Avoid jargon like PARTITION/frame; lean on "compares each row to its group" as the core idea.
