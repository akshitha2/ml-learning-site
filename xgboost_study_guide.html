02 Taylor expansion + Hessian — the math core math

XGBoost approximates the loss function locally using a second-order Taylor expansion — the same principle as Newton's method in optimisation.

// Second-order Taylor expansion of loss around current prediction F
L(F + w) ≈ L(F) + g·w + ½·h·w²

// where:
g = ∂L/∂F // gradient = first derivative of loss
h = ∂²L/∂F² // Hessian = second derivative of loss
w // leaf value we want to find

Worked example using MSE loss: L(F) = ½·(y − F)²

Numerical example — y = 3, F = 1
gradient g
−2
g = ∂L/∂F = F − y = 1 − 3
Hessian h
1
h = ∂²L/∂F² = 1 (constant for MSE)
Taylor approx: L(1 + w) ≈ 2 + (−2)·w + ½·(1)·w² = 2 − 2w + ½w²
◆ Gradient descent vs Newton's method: GBM uses only g (slope). XGBoost uses g + h (slope + curvature). High h = fast-changing slope = take smaller steps. Low h = stable slope = take larger steps. Newton's method converges faster and self-adjusts step size.

Why h matters — classification case:

// For log-loss (binary classification):
g = sigmoid(F) − y
h = sigmoid(F) · (1 − sigmoid(F))

// h varies per example:
sigmoid near 0 or 1 → h small → confident prediction → large step
sigmoid near 0.5 → h large → uncertain prediction → small step
Two examples, same gradient g = 0.3
Example A — low curvature
−3.0
w* = −0.3 / 0.1 = −3.0 (large step)
Example B — high curvature
−0.33
w* = −0.3 / 0.9 = −0.33 (small step)
⚠ Interview trap: "XGBoost uses the Hessian instead of the gradient." Wrong — it uses both. The Hessian modulates the step size; the gradient provides the direction. For MSE, h = 1 always, so −g/h = residual. Hessian only does meaningful work for other losses.
03 Optimal leaf value derivation derivation

XGBoost finds the analytically optimal leaf value by minimising the Taylor-approximated loss. This is not a heuristic — it is an exact closed-form solution.

// Simplified loss for one leaf (ignoring L(F) constant):
Ĺ(w) = g·w + ½·h·w²

// Minimise: differentiate wrt w, set to zero
dĹ/dw = g + h·w = 0

// Solve:
w* = −g / h

// With L2 regularisation (λ on leaf values):
w* = −G / (H + λ)

// where G = Σgᵢ and H = Σhᵢ summed over all examples in the leaf
✓ Key insight: For MSE, h = 1 for every example, so w* = −G/H = −Σresiduals / n = mean residual. XGBoost recovers the same answer as vanilla GBM for MSE. The Hessian only changes the answer for other loss functions.

Substituting w* back gives the minimum achievable loss for a leaf — used in the Gain formula:

L*(leaf) = −½ · G² / (H + λ)
04 Gain formula + γ regularisation splits

XGBoost evaluates every candidate split using a Gain formula derived from the optimal leaf losses. A split only happens if Gain > 0.

Gain = ½ · [ G_L²/(H_L + λ) + G_R²/(H_R + λ) − (G_L+G_R)²/(H_L+H_R+λ) ] − γ

// Split happens only if Gain > 0
// G_L, G_R = sum of gradients in left/right child
// H_L, H_R = sum of Hessians in left/right child
// λ = L2 penalty (regularises the leaf score)
// γ = minimum loss reduction required to justify the split
γ = 0 (no penalty)

Any split that reduces loss at all is accepted. Vanilla GBM behaviour. Prone to fitting noise.

γ high (strong penalty)

Only splits with large loss reduction survive. Shallower trees. Stronger regularisation. Better on noisy data.

◆ γ vs α/λ: γ acts at the split decision — it prevents a leaf from being created. α and λ act after the leaf exists — they shrink the leaf value. For noisy data, prefer γ first (prevention) then α/λ (correction).

06 Missing value handling sparsity-aware

XGBoost's sparsity-aware split finding learns the optimal direction to send missing values — without any imputation.

Training — try both directions: For each split candidate, send all missing-value rows left → compute Gain. Then send all right → compute Gain. Pick the direction with higher Gain.
Storage: The winning direction is stored in the node as the default_direction. One bit per node per feature. Different nodes can have different default directions for the same feature.
Inference: If a feature value is missing at a node → follow the stored default direction. No computation. No lookup of training rows. Just follow the arrow.
Vanilla GBM (imputation)

Requires mean/median imputation before training. Destroys the signal that missingness itself is informative. All missing values in a feature get the same imputed value → same treatment everywhere in the tree.

XGBoost (learned directions)

Missingness is a signal. "Income = missing" may mean high earner (declined to answer). Different nodes learn different directions. No imputation, no information loss.

⚠ Precision trap: "Some missing values go left, some go right" — wrong at node level. At each node, ALL missing values go the same direction (the stored default). Variation happens across different nodes, not within one node.
07 Regressor vs Classifier implementation

The architecture is identical. Differences are the loss function, gradient/Hessian forms, and output transformation.

dimensionXGBRegressorXGBClassifier
F₀ (init)mean(y)log(p / (1−p)) — log-odds of mean class rate
gradient gF − y (residual)sigmoid(F) − y
Hessian h1 (constant)sigmoid(F) · (1 − sigmoid(F)) — varies per example
leaf value w*−G/H = mean residual−G/(H+λ) — Hessian does real work here
final outputF_M (raw value)sigmoid(F_M) → probability
predict()raw valueclass label (threshold 0.5 by default)
predict_proba()n/asigmoid(F_M) — probability between 0 and 1
◆ Key insight: The Hessian is always 1 for MSE regression. XGBoost's mathematical innovation (second-order gradients) only changes results for non-MSE losses like log-loss. For regression, XGBoost with MSE is exactly equivalent to vanilla GBM in terms of leaf values — just faster.
✓ Imbalanced classification: Set scale_pos_weight = count(negative) / count(positive). This upweights the gradient and Hessian contributions of minority class examples. Then use predict_proba() and set your own threshold — default 0.5 is wrong for imbalanced data. Evaluate with AUC-PR, not AUC-ROC.
08 Approximate splits + column blocks speed
Exact greedy split finding

For each feature, sort all values O(n log n), evaluate every possible split point. Optimal but expensive. Fine for small datasets.

Approximate (histogram)

Bucket feature values into quantile bins (e.g. 256 bins). Evaluate only bin boundaries as split candidates. Tiny accuracy loss, massive speed gain.

tree_method = 'exact' // exact greedy — small datasets
tree_method = 'approx' // quantile sketch — medium datasets
tree_method = 'hist' // histogram — large datasets (fastest, similar to LightGBM)

Column blocks + cache-aware access:

Feature columns are pre-sorted once at training start and stored in contiguous memory blocks. This eliminates re-sorting at every node (saves O(n log n) per node) and ensures sequential memory access — data stays in CPU L1/L2 cache (~100× faster than RAM reads). Within-tree parallelisation is possible because different features can be evaluated simultaneously from their independent column blocks.
◆ One-line summary: Pre-sort once (saves time), store contiguously (stays in CPU cache), parallelise across features (uses all cores). This is why XGBoost is faster than sklearn GBM on large datasets — same math, better engineering.
09 Level-wise vs leaf-wise — bridge to LightGBM tree building
XGBoost — level-wise

Breadth-first. Evaluates all nodes at depth d before going to depth d+1. Produces balanced, symmetric trees.

Depth 1: split root
Depth 2: split both children
Depth 3: split all 4 grandchildren
Wastes compute on low-gain splits
LightGBM — leaf-wise

Best-first. Always splits the leaf with highest Gain anywhere in the tree. Produces deep, asymmetric trees.

Round 1: split best leaf globally
Round 2: split next best leaf
...
